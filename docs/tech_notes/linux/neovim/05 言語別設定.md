## C# — roslyn.nvim

### 背景

C# の LSP サーバーには歴史的に OmniSharp が使われてきましたが、現在は **Roslyn** ベースのサーバーへの移行が進んでいます。VS Code の C# 拡張も内部的には Roslyn LSP サーバーを使っています。Neovim でも Roslyn 路線に乗るのが自然です。

`roslyn.nvim` は OmniSharp の置き換えを明示的に掲げており、以下の機能に対応しています。

- 複数ソリューションの同時管理
- Source Generated Files のサポート
- Fix All（一括修正）
- 複雑な Code Action

### 注意点

- **Razor は非対応**です。Blazor の `.razor` ファイルは roslyn.nvim では扱えません
- Roslyn LSP サーバーは Mason の**追加レジストリ**（`Crashdummyy/mason-registry`）に含まれています。[[04 LSP・補完・整形]] の mason.nvim 設定で追加済みです

### 設定ファイル

`~/.config/nvim/lua/plugins/csharp.lua`:

```lua
return {
  {
    "seblyng/roslyn.nvim",
    ft = "cs",
    opts = {
      filewatching = "auto",
      broad_search = true,
    },
    config = function(_, opts)
      require("roslyn").setup(opts)
    end,
  },
}
```

### 設定の意味

| オプション | 値 | 効果 |
| --- | --- | --- |
| `ft` | `"cs"` | `.cs` ファイルを開いたときだけ読み込む（遅延読み込み） |
| `filewatching` | `"auto"` | ファイル変更の監視方法を自動選択する |
| `broad_search` | `true` | ソリューション外のファイルも検索対象に含める |

### Roslyn LSP サーバーのインストール

Mason の追加レジストリ経由でインストールします。

```vim
:MasonInstall roslyn
```

### 動作確認

1. `.sln` ファイルがあるディレクトリで `.cs` ファイルを開く
2. LSP が起動し、補完・定義ジャンプ・Code Action が動作することを確認する
3. 保存時に csharpier で整形されることを確認する（conform.nvim 経由）

### dotnet ツールの整理

C# 開発で使う dotnet tool の一覧です。

```bash
dotnet tool install -g csharpier    # フォーマッター（01 で実行済み）
```

## Rust — rustaceanvim

### Rust の LSP 背景

Rust の Neovim サポートはかつて `rust-tools.nvim` が担っていましたが、現在はその流れを引き継いだ **rustaceanvim** が主流です。rust-analyzer との連携を強化し、Rust 特有の操作（expand macro, open Cargo.toml, join lines, view HIR/MIR 等）をまとめて提供します。

### lang-extra.lua の設定

`~/.config/nvim/lua/plugins/lang-extra.lua` に Rust と Markdown の設定をまとめます。

```lua
return {
  -- Rust
  {
    "mrcjkb/rustaceanvim",
    version = "^6",
    ft = { "rust" },
  },

  -- Markdown（バッファ内レンダリング）
  {
    "MeanderingProgrammer/render-markdown.nvim",
    ft = { "markdown" },
    dependencies = {
      "nvim-treesitter/nvim-treesitter",
      "nvim-tree/nvim-web-devicons",
    },
    opts = {},
  },

  -- Markdown（ブラウザプレビュー）
  {
    "iamcco/markdown-preview.nvim",
    ft = { "markdown" },
    build = function()
      vim.fn["mkdp#util#install"]()
    end,
  },
}
```

### rustaceanvim の設計判断

| 項目 | 説明 |
| --- | --- |
| `version = "^6"` | メジャーバージョン 6 系の最新を追う。破壊的変更を自動では取り込まない |
| `ft = { "rust" }` | `.rs` ファイルを開いたときだけ読み込む |
| `opts` 未指定 | rustaceanvim はデフォルト設定で十分に実用的。rust-analyzer の設定はプロジェクトの `rust-analyzer.json` や `.cargo/config.toml` で管理する方が良い |

### rustaceanvim が提供する追加コマンド

通常の LSP 操作（補完、定義ジャンプ、Code Action）は Neovim の LSP 基盤で動きます。rustaceanvim は以下の Rust 固有操作を追加します。

| コマンド | 操作 |
| --- | --- |
| `:RustLsp expandMacro` | マクロを展開して表示する |
| `:RustLsp openCargo` | Cargo.toml を開く |
| `:RustLsp joinLines` | Rust の意味を考慮して行を結合する |
| `:RustLsp moveItem up/down` | 構造体フィールド等を上下に移動する |
| `:RustLsp explainError` | エラーの詳細説明を表示する |

### rust-analyzer のインストールについて

[[01 Ubuntu Neovim 環境 OS 基盤セットアップ]] で `rustup component add rust-analyzer` を実行済みです。rustaceanvim は PATH 上の `rust-analyzer` を自動検出するため、追加設定は不要です。

> **lsp.lua で `vim.lsp.enable("rust_analyzer")` も書いているのは冗長か？**: rustaceanvim は独自に rust-analyzer を管理するため、lsp.lua 側の `vim.lsp.enable("rust_analyzer")` は**削除しても問題ありません**。ただし、rustaceanvim がまだ読み込まれていない状況（ft による遅延読み込み前に別の理由で `.rs` が参照される場合など）のフォールバックとして残しておいても害はありません。

## Python — basedpyright

### Python の LSP 背景

Python の LSP サーバーとしては Pyright が広く使われていますが、**basedpyright** はその fork で、以下の改善点を持っています。

- Pylance（VS Code 専用の Python 拡張）の一部機能の取り込み
- より厳密な型チェック
- inlay hints の強化
- better error message

### 設定

basedpyright は [[04 LSP・補完・整形]] の lsp.lua で `vim.lsp.enable("basedpyright")` として既に有効化しています。追加のプラグインファイルは不要です。

型チェックの厳密さを変えたい場合は lsp.lua でカスタマイズできます。

```lua
vim.lsp.config("basedpyright", {
  settings = {
    basedpyright = {
      analysis = {
        typeCheckingMode = "standard",  -- "off" / "basic" / "standard" / "strict" / "all"
      },
    },
  },
})
```

| モード | 説明 |
| --- | --- |
| `off` | 型チェックを無効化する |
| `basic` | 最小限のチェック |
| `standard` | 推奨。実用的な範囲でチェックする |
| `strict` | 厳密。型注釈の不足も警告する |
| `all` | 最も厳密。Pylance の全チェックに相当 |

### Python フォーマッターと仮想環境

フォーマッターは conform.nvim 経由で `ruff_format` を使います（[[04 LSP・補完・整形]] で設定済み）。

プロジェクトごとの仮想環境は uv で管理します。

```bash
uv init myproject
cd myproject
uv add requests  # 例
```

basedpyright は `.venv/` を自動検出します。プロジェクトルートに `.venv/` があれば、追加設定なしで仮想環境内のパッケージを認識します。

## Markdown — render-markdown.nvim + markdown-preview.nvim

### 用途の使い分け

Markdown は2つのプラグインを目的別に使い分けます。

| プラグイン | 用途 | 表示場所 |
| --- | --- | --- |
| **render-markdown.nvim** | バッファ内で Markdown をリッチ表示する | Neovim 内 |
| **markdown-preview.nvim** | ブラウザで最終形を確認する | ブラウザ |

### render-markdown.nvim

Neovim のバッファ内で、見出し・リスト・テーブル・コードブロック等を装飾表示します。編集とプレビューを一画面でこなせるため、書きながら確認するのに向いています。

- `ft = { "markdown" }`: Markdown ファイルのみで有効化
- `opts = {}`: デフォルト設定で十分

### markdown-preview.nvim

ブラウザでリアルタイムプレビューを表示します。以下の記法にも対応しています。

- Mermaid（ダイアグラム）
- KaTeX（数式）
- PlantUML
- 同期スクロール

keymaps.lua で `<leader>mp` に割り当てています。

- `build = function() vim.fn["mkdp#util#install"]() end`: 初回インストール時にプレビューサーバーの依存をビルドする。Node.js が必要（01 でインストール済み）

## トラブルシューティング

### LSP サーバーが起動しない

`:checkhealth lsp` で状態を確認してください。よくある原因:

- 外部ツールが PATH に通っていない → `which basedpyright` 等で確認
- Mason でインストールしたが registries が未設定 → mason.nvim の `registries` 設定を確認

### treesitter パーサーのビルドに失敗する

`build-essential` がインストールされていることを確認してください。

```bash
sudo apt install -y build-essential
```

### Roslyn が起動しない

追加レジストリ（`Crashdummyy/mason-registry`）の設定が mason.nvim にあることを確認してください。`:MasonInstall roslyn` が成功しているかどうかも確認してください。

### blink.cmp の補完が遅い、または出ない

`:checkhealth blink` で確認してください。Rust ベースのバイナリが正しくビルドされていない場合、`:Lazy build blink.cmp` で再ビルドできます。
