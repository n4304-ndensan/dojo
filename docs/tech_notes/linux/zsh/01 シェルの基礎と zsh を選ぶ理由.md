# シェルの基礎と zsh を選ぶ理由

## この文書の責務

シェルとは何か、主要なシェルの違い、そして zsh を選ぶ理由を解説します。「zsh が速い」「zsh が良い」だけでなく、なぜそうなのかの技術的根拠を示します。

## シェルとは何か

シェルはカーネルとユーザーの間に立つインターフェースです。ユーザーが入力したコマンドを解釈してカーネルに渡し、結果を返します。

```text
ユーザー → シェル → カーネル → ハードウェア
         ↑ ここが zsh / bash / fish
```

シェルが担う仕事は大きく3つあります。

1. **対話的操作**: コマンドの入力、補完、履歴の管理
2. **スクリプト実行**: 自動化のためのシェルスクリプトの解釈・実行
3. **環境管理**: 環境変数、PATH、エイリアス、関数の管理

対話的操作の快適さとスクリプト実行の互換性は、しばしばトレードオフの関係にあります。この2つをどう折り合いをつけるかが、シェル選択の核心です。

## 主要なシェルの比較

### sh（Bourne Shell）

すべての Unix 系 OS に存在する最小のシェルです。POSIX 標準の基盤であり、`/bin/sh` として利用できます。対話的機能はほぼなく、スクリプトの互換性だけを目的としています。

シェルスクリプトの `#!/bin/sh` はこのシェルを指しています。可搬性が最も高いですが、対話的に使うものではありません。

### bash（Bourne-Again Shell）

sh の上位互換として開発され、Linux ディストリビューションの標準シェルとして広く使われています。

強み:

- ほぼすべての Linux に標準搭載されている
- sh との後方互換性が高い
- 膨大なドキュメントと知見がある
- シェルスクリプトの事実上の標準

弱み:

- 補完システムが後付けで、設定が煩雑
- プロンプトのカスタマイズが PS1 のエスケープシーケンスに依存して読みにくい
- プラグインエコシステムが公式には存在しない
- 配列やハッシュの構文が古く、扱いにくい

### zsh（Z Shell）

bash との高い互換性を保ちつつ、対話的機能を大幅に強化したシェルです。macOS では Catalina（2019年）からデフォルトシェルになっています。

強み:

- bash のスクリプトがほぼそのまま動く互換性
- 強力な補完システム（コマンドのオプション、引数、ファイルパスをコンテキストに応じて補完）
- グロブ拡張（`**/*.rs` のような再帰パターン）
- プラグインエコシステムが成熟（zinit, sheldon, antidote 等）
- 連想配列・算術式・パラメータ展開が bash より洗練されている

弱み:

- Ubuntu にはデフォルトで入っていない（`apt install zsh` が必要）
- bash との完全互換ではない（`#!/bin/bash` のスクリプトは bash で実行すべき）
- 設定が多機能ゆえに複雑になりやすい（oh-my-zsh で重くなる典型例）

### fish（Friendly Interactive Shell）

「すぐ使える」を重視した設計のシェルです。インストール直後から構文ハイライト、自動サジェスト、タブ補完が動きます。

強み:

- 設定なしでリッチな対話的機能が使える
- 構文が直感的でスクリプトが書きやすい
- Web UI で設定を変更できる
- エラーメッセージが親切

弱み:

- **POSIX 非互換**: bash / zsh とスクリプト構文が完全に異なる
- `export VAR=value` ではなく `set -x VAR value` を使う
- 既存のシェルスクリプト資産が流用できない
- CI/CD やサーバー管理では bash を使うことになるため、知識が分散する

### nushell

構造化データ（テーブル）をネイティブに扱うモダンシェルです。パイプラインがテキストではなくデータを渡します。

強み:

- `ls | where size > 1mb` のようなデータクエリが書ける
- JSON / YAML / CSV をネイティブに扱える
- エラーメッセージが極めて丁寧

弱み:

- POSIX 非互換で、既存スクリプトは使えない
- まだエコシステムが小さい
- 学習コストが高い

## なぜ zsh を選ぶのか

### 理由 1: bash との互換性を保ったまま対話的機能が強くなる

fish や nushell は魅力的ですが、POSIX 非互換という根本的な問題があります。サーバー、CI/CD、Dockerfile、チームの共有スクリプトはほぼすべて bash（または sh）前提です。fish を選ぶと、対話的シェルとスクリプト実行で完全に別の言語を使い分けることになります。

zsh は bash と構文がほぼ同じです。`.bashrc` で使っていた `export`, `alias`, 制御構文、パイプライン、リダイレクトはそのまま `.zshrc` に持ち込めます。bash のスクリプト知識が無駄になりません。

### 理由 2: 補完システムが設計段階で強い

bash の補完は `complete` コマンドで後付けする仕組みです。zsh は `compsys`（completion system）という統合された補完フレームワークを持っています。

具体例:

- `git checkout` の後にブランチ名を補完する
- `ssh` の後にホスト名（`~/.ssh/config` から）を補完する
- `kill` の後にプロセス名と PID を表示する
- `cd` の後にディレクトリだけを表示する（ファイルを除外）

これらは zsh では標準機能ですが、bash では個別に設定ファイルを用意する必要があります。

### 理由 3: プラグインエコシステムが成熟している

bash にはプラグインマネージャーの文化がありません。zsh には zinit（旧 zplugin）、sheldon、antidote、zplug など複数のマネージャーがあり、以下のようなプラグインを宣言的に管理できます。

- **zsh-autosuggestions**: 入力中に過去のコマンドをグレーで表示し、→ キーで採用
- **zsh-syntax-highlighting**: 入力中にコマンドの正否をリアルタイムでハイライト
- **zsh-completions**: 数百のコマンドに対する追加補完定義
- **zsh-history-substring-search**: 部分一致で履歴を検索

これらが fish では組み込みですが、bash では実現困難です。zsh はプラグインで同等の体験を得つつ、POSIX 互換を保っています。

### 理由 4: macOS でもデフォルト

macOS は 2019 年から zsh がデフォルトシェルです。Ubuntu と macOS の両方で同じ `.zshrc` を使い回せるため、dotfiles の再現性が高くなります。

### 理由 5: 速度は設定次第

「zsh は bash より遅い」という指摘があります。これは半分正しく、半分誤りです。素の zsh の起動速度は bash とほぼ同じです。遅くなるのは oh-my-zsh のような「全部入り」フレームワークを入れた場合です。

zinit の遅延ロード（`wait` / `lucid`）を使えば、プラグインを入れても起動速度を 50ms 以下に保てます。詳細は [[05 tmux 連携と起動速度最適化]] で扱います。

## oh-my-zsh を使わない理由

oh-my-zsh は zsh プラグインフレームワークとして最も有名（GitHub 180k+ stars）ですが、この構成では使いません。理由は3つあります。

1. **起動が遅い**: oh-my-zsh は起動時にすべてのプラグインとテーマを一括ロードします。プラグインを増やすほど起動速度が線形に悪化します。zinit の遅延ロードではこの問題が発生しません。

2. **粒度が粗い**: oh-my-zsh のプラグインは「有効/無効」の2択です。zinit では「いつロードするか」「どの条件でロードするか」を個別に指定できます。

3. **暗黙の設定が多い**: oh-my-zsh は有効化しただけで大量のエイリアスと設定が暗黙に入ります。何が有効で何が無効かを把握するのが困難です。明示的に `.zshrc` に書く方が管理しやすく、再現性も高くなります。

## この構成の設計原則

| 原則 | 説明 |
| --- | --- |
| 速度優先 | 起動 50ms 以下を目標にする。遅延ロードをデフォルトにする |
| 明示的 | 暗黙の設定を避ける。`.zshrc` に書いてあるものだけが有効 |
| 互換性維持 | bash のスクリプト知識・資産をそのまま活用できる |
| 再現性 | dotfiles として Git 管理し、新しいマシンで即再現できる |
| 分離 | シェル(zsh) / 検索(fzf) / 移動(zoxide) / 表示(starship) / 編集(neovim) を独立に管理する |
