# XEでSQL実行内容を確認するための整理

> 位置づけ: この文書は、Extended Events で SQL 実行内容を追うときの調査メモです。RCSI 評価の主経路ではなく、「どのイベントを選べば何が見えるか」を整理するための補助資料として使います。
>
> 読み分け: SQL Server フォルダ全体の案内は [[SQL Server ドキュメントガイド]] を参照してください。

## 1. 文脈と目的

このドキュメントは、Extended Events で取得できる `sql_batch_completed`、`rpc_completed`、`sql_statement_completed`、`sp_statement_completed` の違いを、実務で判断できる粒度まで整理することを目的にしています。

今回この整理が必要になった背景には、バッチ処理の更新確認を行う中で、想定していた `INSERT` や `UPDATE` のログが見当たらず、代わりに `SELECT` 文や `SET FMTONLY ON` を含むログが取得されていた、という状況があります。

そのため本ドキュメントでは、個々のイベント定義を説明するだけではなく、SQL Server に対する処理がどの単位で実行され、そのどの単位を XE が拾っているのか、という観点から全体像を整理します。これにより、今後同様の調査を行う際にも、目的に応じて適切なイベントを選択できる状態を目指します。

## 2. まず押さえるべき前提

XE のイベント名は似ていますが、見ている対象の粒度が異なります。ここを曖昧にしたまま調査すると、「ログは取れているのに知りたいことが分からない」という状態になりやすくなります。

今回の論点を整理すると、確認したいものは大きく次の四種類に分かれます。

- バッチとしてまとめて送られた SQL 全体
- ストアドや `sp_executesql` の呼び出し全体
- ad-hoc SQL の文単位
- ストアド内部で実行された文単位

特に、アプリケーションからの実行では `sp_executesql` やストアド経由で SQL が流れることが多いため、`sql_batch_completed` だけ見ても、実際に更新された SQL 文まで追えないことがあります。

## 3. 四つのイベントの違い

これら四つは「SQL 文の種類」そのものの違いではなく、XE がどの実行単位を見ているかの違いです。先に全体を一枚で見ると、使い分けがぶれにくくなります。

| イベント | 見ている単位 | 向いている確認 | 粒度 |
| --- | --- | --- | --- |
| `sql_batch_completed` | ひとかたまりで送られたバッチ全体 | 処理の外枠や流量の確認 | 粗い |
| `rpc_completed` | ストアドや `sp_executesql` の呼び出し全体 | アプリ実行の実態確認 | やや粗い |
| `sql_statement_completed` | ad-hoc バッチ内の個々の SQL 文 | 文単位の更新確認 | 細かい |
| `sp_statement_completed` | ストアド内部の個々の SQL 文 | ストアド内部の更新確認 | 細かい |

### 3.1 `sql_batch_completed` とは何か

`sql_batch_completed` は、SQL Server に対してひとかたまりで送られたバッチ全体が完了したことを表すイベントです。

たとえば、次のように二文をまとめて送った場合、XE からは一つのバッチとして見えます。

```sql
SELECT * FROM MST_A;
UPDATE MST_B SET NAME = 'A' WHERE ID = 1;
```

このとき `sql_batch_completed` は、「このバッチが完了した」という事実を捉えます。バッチの全体像をつかむには有効ですが、内部のどの一文が更新だったのかを細かく見たい場合には情報が粗くなります。

### 3.2 `rpc_completed` とは何か

`rpc_completed` は、リモートプロシージャ呼び出し全体が完了したことを表すイベントです。実務では、ストアドプロシージャの呼び出しや、`.NET` などのアプリケーションが `sp_executesql` を使ってパラメータ付き SQL を実行するケースでよく関係します。

たとえばアプリ側では、見た目上はただの `SELECT` や `UPDATE` に見えても、実際には次のような形で SQL Server に渡されていることがあります。

```sql
EXEC sp_executesql
    N'UPDATE T2 SET COL = @p1 WHERE ID = @p2',
    N'@p1 int, @p2 int',
    @p1 = 1, @p2 = 10;
```

この場合、外側の実行単位としては「`sp_executesql` を呼んだ」という形になるため、`rpc_completed` はその呼び出し全体を捉えます。

### 3.3 `sql_statement_completed` とは何か

`sql_statement_completed` は、ad-hoc のバッチ内で実行された一文ごとの SQL を捉えるイベントです。

一つのバッチ内に `SELECT`、`UPDATE`、`INSERT` が含まれていれば、それぞれの文単位で取得できます。つまり、`sql_batch_completed` が「全体を見る」イベントであるのに対し、`sql_statement_completed` は「その中身を一文ずつ見る」ためのイベントです。

### 3.4 `sp_statement_completed` とは何か

`sp_statement_completed` は、ストアドプロシージャの内部で実行された一文ごとの SQL を捉えるイベントです。

ストアドプロシージャの外側だけ見ていると、「ストアドが呼ばれた」ことまでは分かっても、中で何をしたかまでは見えません。ストアドの中で `SELECT`、`UPDATE`、`INSERT` など複数の SQL が順に実行される場合、その内部の個別文まで見たいときに使うのが `sp_statement_completed` です。

## 4. 今回の `SET FMTONLY ON` を含むログは何か

今回確認していたログには、次のような内容が含まれていました。

```sql
SET FMTONLY OFF;
SET NO_BROWSETABLE ON;
SET FMTONLY ON;
SELECT * FROM XAGAKUNENJOHOMST ...
```

このログで重要なのは、最後に実行されているのが `SELECT` である点です。そのため、このログ自体を見て「更新が行われた」と判断することはできません。

さらに `SET FMTONLY ON` が含まれていることから、これは実データ更新のための SQL というより、結果セットや列構成の確認、あるいは内部的なメタ情報取得のために発行されている SQL と考える方が自然です。

つまり、このログは次のように扱うのが適切です。

- 更新ログそのものではありません。
- `INSERT`、`UPDATE`、`DELETE` の代替としては扱いません。
- 実更新を確認したい場合は、別途更新 SQL の出力を確認する必要があります。

## 5. 実務でどう使い分けるべきか

この文書の責務は、最終的に「何を見ればよいか」を判断できるようにすることです。その観点で言うと、まず目的を明確にする必要があります。

処理の外枠を把握したいだけであれば、`sql_batch_completed` と `rpc_completed` を使うのが基本です。これにより、どの時間帯にどの処理が流れたか、どのアプリから呼ばれたか、といった全体像を把握しやすくなります。

実際に更新 SQL が流れたかを確認したい場合は、`sql_statement_completed` と `sp_statement_completed` を追加する必要があります。これにより、単に `SELECT` しか流れていないのか、実際に `UPDATE` や `INSERT` が発行されているのかを文単位で確認できます。

特にアプリケーションからの実行では `sp_executesql` やストアド経由になることが多いため、`sql_batch_completed` だけでは十分でないことがあります。この場合は、少なくとも `rpc_completed` を併用し、必要に応じて `sp_statement_completed` まで確認するのが安全です。

## 6. 推奨する調査手順

後から見返しても判断しやすいよう、調査は次の二段階で考えると整理しやすくなります。

1. まず `sql_batch_completed` と `rpc_completed` で、処理が呼ばれているか、どのタイミングで流れたか、対象アプリは何かを確認します。
2. 外枠が確認できたら、`sql_statement_completed` と `sp_statement_completed` を追加し、本当に `UPDATE`、`INSERT`、`DELETE` が発行されたかを確認します。

この順番にしておくと、最初から粒度を上げすぎて大量ログに埋もれることを避けやすくなります。

## 7. よくある誤解

今回のような調査では、次の誤解が起きやすいため注意が必要です。

1. `SELECT` が出ているから、更新前後確認の一部として更新ログとみなしてよいとは限りません。
2. `rpc_completed` が取れているから、中で実行された SQL もすべて把握できているとは限りません。
3. `sql_batch_completed` が見えているから、更新 SQL も必ず拾えているとは限りません。
4. `sp_executesql` の呼び出しが見えていても、その内部文を確認しなければ更新有無は断定できません。

## 8. まとめ

今回のポイントは、XE の各イベントが似た名前でも、見ている単位が異なることです。これを理解すると、更新調査の進め方が整理しやすくなります。

要点をまとめると、`sql_batch_completed` は SQL のまとまり全体を見て、`rpc_completed` はストアドや `sp_executesql` の呼び出し全体を見ます。`sql_statement_completed` は ad-hoc SQL の文単位を見て、`sp_statement_completed` はストアド内部の文単位を見ます。したがって、「一定期間に何が実行されたか」をざっくり知りたいなら batch / rpc completed を使い、「実際にどの更新 SQL が流れたか」まで知りたいなら statement completed 系を追加する、という使い分けが基本になります。

## 9. 実運用上の注意

statement completed 系は便利ですが、件数が増えやすいため、長時間かつ無制限に取得するとログ量が大きくなります。そのため実運用では、必要に応じて次の条件で絞り込む方が安全です。

- アプリケーション名
- DB 名
- ホスト名
- ユーザー名
- セッション ID

この絞り込みを適切に行うことで、必要な情報を取りつつ調査コストを抑えやすくなります。
