# XEでSQL実行内容を確認するための整理

## このドキュメントの目的

このドキュメントは、Extended Events（XE）で取得できる `sql_batch_completed`、`rpc_completed`、`sql_statement_completed`、`sp_statement_completed` の違いを、実務で判断できる粒度まで整理することを目的としています。

今回この整理が必要になった背景には、バッチ処理の更新確認を行う中で、想定していた `INSERT` や `UPDATE` のログが見当たらず、代わりに `SELECT` 文や `SET FMTONLY ON` を含むログが取得されていた、という状況があります。

そのため本ドキュメントでは、個々のイベント定義を説明するだけではなく、**SQL Server に対する処理がどの単位で実行され、そのどの単位を XE が拾っているのか** という観点から全体像を整理します。これにより、今後同様の調査を行う際にも、目的に応じて適切なイベントを選択できる状態を目指します。

---

## まず押さえるべき前提

XE のイベント名は似ていますが、見ている対象の粒度が異なります。ここを曖昧にしたまま調査すると、「ログは取れているのに知りたいことが分からない」という状態になりやすいです。

今回の論点を整理すると、確認したいものは大きく次の4種類に分かれます。

- バッチとしてまとめて送られた SQL 全体

- ストアドや `sp_executesql` の呼び出し全体

- ad-hoc SQL の文単位

- ストアド内部で実行された文単位


この4つは似ているようで用途が異なります。特に、アプリケーションからの実行では `sp_executesql` やストアド経由で SQL が流れることが多いため、`sql_batch_completed` だけ見ても、実際に更新された SQL 文まで追えないことがあります。

---

## 1. `sql_batch_completed` とは何か

`sql_batch_completed` は、SQL Server に対して **ひとかたまりで送られたバッチ全体** が完了したことを表すイベントです。

たとえば、以下のように 2 文をまとめて送った場合、XE からは 1 つのバッチとして見えます。

```sql
SELECT * FROM MST_A;
UPDATE MST_B SET NAME = 'A' WHERE ID = 1;
```

このとき `sql_batch_completed` は、「このバッチが完了した」という事実を捉えます。バッチの全体像をつかむには有効ですが、内部のどの 1 文が更新だったのかを細かく見たい場合には情報が粗いです。

### 主な性質

- 粒度は粗い
    
- 全体像の把握に向く
    
- 実行された文の明細までは追いにくい
    

### 実務での用途

- 一定期間にどのような SQL の塊が流れたかを確認したい場合
    
- 処理全体の実行時間や件数をざっくり把握したい場合
    
- SSMS や ad-hoc 実行のバッチを大きな単位で見たい場合
    

---

## 2. `rpc_completed` とは何か

`rpc_completed` は、**リモートプロシージャ呼び出し（RPC）全体** が完了したことを表すイベントです。実務では、ストアドプロシージャの呼び出しや、`.NET` などのアプリケーションが `sp_executesql` を使ってパラメータ付き SQL を実行するケースでよく関係します。

たとえばアプリ側では、見た目上はただの `SELECT` や `UPDATE` に見えても、実際には次のような形で SQL Server に渡されていることがあります。

```sql
EXEC sp_executesql
  N'UPDATE T2 SET COL = @p1 WHERE ID = @p2',
  N'@p1 int, @p2 int',
  @p1 = 1, @p2 = 10;
```

この場合、外側の実行単位としては「`sp_executesql` を呼んだ」という形になるため、`rpc_completed` はその呼び出し全体を捉えます。

### 主な性質

- 呼び出し単位で見える
    
- パラメータ付き SQL やストアド実行と相性がよい
    
- アプリケーションの実行実態に近い
    

### 実務での用途

- バッチアプリや .NET アプリが発行した SQL の外枠を追いたい場合
    
- ストアドや `sp_executesql` の呼び出し有無を確認したい場合
    
- 処理の開始・完了を呼び出し単位で把握したい場合
    

---

## 3. `sql_statement_completed` とは何か

`sql_statement_completed` は、**ad-hoc のバッチ内で実行された 1 文ごとの SQL** を捉えるイベントです。

たとえば 1 つのバッチ内に `SELECT`、`UPDATE`、`INSERT` が含まれていれば、それぞれの文単位で取得できます。つまり、`sql_batch_completed` が「全体を見る」イベントであるのに対し、`sql_statement_completed` は「その中身を 1 文ずつ見る」ためのイベントです。

### 主な性質

- 粒度が細かい
    
- 実際にどの文が流れたかを確認しやすい
    
- その分、イベント件数が増えやすい
    

### 実務での用途

- 実際に `UPDATE` や `INSERT` が発行されたかを確認したい場合
    
- 1つの処理の中で何本の SQL が流れたかを確認したい場合
    
- 遅い SQL を文単位で特定したい場合
    

---

## 4. `sp_statement_completed` とは何か

`sp_statement_completed` は、**ストアドプロシージャの内部で実行された 1 文ごとの SQL** を捉えるイベントです。

ストアドプロシージャの外側だけ見ていると、「ストアドが呼ばれた」ことまでは分かっても、中で何をしたかまでは見えません。実際には、ストアドの中で `SELECT`、`UPDATE`、`INSERT` など複数の SQL が順に実行されることがあります。

このとき、その内部の個別文まで見たい場合に使うのが `sp_statement_completed` です。

### 主な性質

- ストアド内部の詳細確認に向く
    
- 更新 SQL の有無を内部まで追いやすい
    
- 件数が増えやすいため、無制限に取得するとログ量が大きくなる
    

### 実務での用途

- ストアドの中でどの更新 SQL が実行されたかを確認したい場合
    
- バッチ処理の本体がストアドで構成されている場合
    
- `rpc_completed` で外枠は見えたが、内部の実更新が不明な場合
    

---

## 4つの違いを一言で整理すると

これら4つは「SQL文の種類」そのものの違いではなく、**XE がどの実行単位を見ているかの違い** です。ここを誤解しないことが重要です。

整理すると次のようになります。

- `sql_batch_completed`
    
    - SQL のまとまり全体を見る
        
- `rpc_completed`
    
    - ストアドや `sp_executesql` の呼び出し全体を見る
        
- `sql_statement_completed`
    
    - ad-hoc バッチ内の個々の SQL 文を見る
        
- `sp_statement_completed`
    
    - ストアド内部の個々の SQL 文を見る
        

---

## 今回の `SET FMTONLY ON` を含むログは何か

今回確認していたログには、次のような内容が含まれていました。

```sql
SET FMTONLY OFF;
SET NO_BROWSETABLE ON;
SET FMTONLY ON;
SELECT * FROM XAGAKUNENJOHOMST ...
```

このログで重要なのは、最後に実行されているのが `SELECT` である点です。そのため、このログ自体を見て「更新が行われた」と判断することはできません。

さらに `SET FMTONLY ON` が含まれていることから、これは実データ更新のための SQL というより、**結果セットや列構成の確認、あるいは内部的なメタ情報取得のために発行されている SQL** と考えるのが自然です。

つまり、このログは以下のように扱うのが適切です。

- 更新ログそのものではない
    
- `INSERT` / `UPDATE` / `DELETE` の代替としては扱わない
    
- 実更新を確認したい場合は、別途更新 SQL の出力を確認する必要がある

## 実務でどう使い分けるべきか

このドキュメントの責務は、最終的に「何を見ればよいか」を判断できるようにすることです。その観点で言うと、まず目的を明確にする必要があります。

### 1. 処理全体の流れを把握したい場合

処理の外枠を把握したいだけであれば、次を使うのが基本です。

- `sql_batch_completed`
    
- `rpc_completed`
    

これにより、どの時間帯にどの処理が流れたか、どのアプリから呼ばれたか、といった全体像を把握しやすくなります。

### 2. 実際に更新 SQL が流れたか確認したい場合

更新有無を確認したい場合は、次を追加する必要があります。

- `sql_statement_completed`
    
- `sp_statement_completed`
    

これにより、単に `SELECT` しか流れていないのか、実際に `UPDATE` や `INSERT` が発行されているのかを文単位で確認できます。

### 3. アプリ経由で `sp_executesql` が使われている場合

アプリケーションからの実行では `sp_executesql` やストアド経由になることが多いため、`sql_batch_completed` だけでは十分でないことがあります。この場合は、少なくとも `rpc_completed` を併用し、必要に応じて `sp_statement_completed` まで確認するのが安全です。

---

## 調査方針の推奨パターン

後から見返しても判断しやすいよう、調査方針は次の2段階で考えるのが実務的です。

### 第1段階: 外枠を把握する

まずは以下で全体の流れを掴みます。

- `sql_batch_completed`
    
- `rpc_completed`
    

この段階では、「処理が呼ばれているか」「どのタイミングで流れたか」「対象アプリは何か」を見ることを主目的とします。

### 第2段階: 実際の更新文を確認する

外枠が確認できたら、必要に応じて以下を追加します。

- `sql_statement_completed`
    
- `sp_statement_completed`
    

この段階では、「本当に `UPDATE` / `INSERT` / `DELETE` が発行されたか」を確認します。

---

## 典型的な勘違い

今回のような調査では、次の誤解が起きやすいため注意が必要です。

- `SELECT` が出ているから更新前後確認の一部として更新ログとみなしてよい、とは限らない
    
- `rpc_starting` が取れているから中で実行された SQL もすべて把握できている、とは限らない
    
- `sql_batch_completed` が見えているから更新 SQL も必ず拾えている、とは限らない
    
- `sp_executesql` の呼び出しが見えていても、その内部文を確認しなければ更新有無は断定できない
    

---

## まとめ

今回のポイントは、XE の各イベントが似た名前でも、見ている単位が異なることです。これを理解すると、更新調査の進め方が整理しやすくなります。

要点をまとめると次の通りです。

- `sql_batch_completed` は SQL のまとまり全体を見る
    
- `rpc_completed` はストアドや `sp_executesql` の呼び出し全体を見る
    
- `sql_statement_completed` は ad-hoc SQL の文単位を見る
    
- `sp_statement_completed` はストアド内部の文単位を見る
    
- `SET FMTONLY ON` を含む `SELECT` ログは、更新ログそのものとは扱わない
    
- 実際の更新有無を確認したい場合は、statement completed 系まで見ないと不足することがある
    

したがって、**「一定期間に何が実行されたか」をざっくり知りたいなら batch / rpc completed、**  
**「実際にどの更新 SQL が流れたか」まで知りたいなら statement completed 系を追加する、**  
という使い分けが基本になります。

---

## 補足: 実運用上の注意

statement completed 系は便利ですが、件数が増えやすいため、長時間・無制限に取得するとログ量が大きくなります。そのため実際の運用では、必要に応じて以下の条件で絞り込むことが推奨されます。

- アプリケーション名
    
- DB名
    
- ホスト名
    
- ユーザー名
    
- セッションID
    

この絞り込みを適切に行うことで、必要な情報を取りつつ調査コストを抑えやすくなります。
