# AKS とサービスメッシュによるマイクロサービスアーキテクチャ設計

## 1. 背景（シナリオ）

ある企業では eコマースアプリケーションをマイクロサービスアーキテクチャで構築する計画があります。このシステムでは、注文管理、商品カタログ、支払い処理、在庫管理などの複数のサービスがそれぞれ独立して動作する構造になっています。

このアーキテクチャでは、各マイクロサービスを個別に開発・デプロイできることが重要です。また、サービスの更新時にシステム全体を停止させることなく、継続的にサービスを提供する必要があります。さらに、このシステムはオンプレミス環境と Azure の両方で実行されるハイブリッド構成を想定しています。

加えて、マイクロサービス同士の通信はアプリケーションのパフォーマンスに大きく影響するため、低遅延で高可用性の通信を提供する仕組みも必要になります。

## 2. 要件整理

このシナリオの要件を整理すると、マイクロサービスアーキテクチャに求められる重要なポイントが明確になります。

まず、各マイクロサービスは独立して更新できる必要があります。これは CI/CD を活用した継続的デリバリーを実現するための重要な要件です。さらに、更新中もサービス停止が発生しないよう、ローリングアップデートなどの仕組みが必要になります。

また、このシステムはオンプレミスと Azure の両方で実行されるため、環境間で一貫した実行基盤を持つことが望ましいです。加えて、マイクロサービス同士の通信はアプリケーション全体のレスポンスに直結するため、低遅延かつ高可用性である必要があります。

これらの要件をまとめると、次のような特徴が必要になります。

・マイクロサービスの独立したデプロイ  
・ダウンタイムなしの更新（ローリングアップデート）  
・オンプレミスとクラウドの両方で動作する環境  
・サービス間通信の管理と可観測性  
・高可用性とスケーラビリティ  

## 3. 技術の基本概念

マイクロサービスアーキテクチャでは、コンテナとコンテナオーケストレーション技術が重要な役割を果たします。その代表的なプラットフォームが Kubernetes です。

Kubernetes はコンテナ化されたアプリケーションを管理するためのオーケストレーションプラットフォームであり、以下のような機能を提供します。

まず、Kubernetes はローリングアップデート機能を持っています。これはアプリケーションを段階的に更新することで、サービス停止を発生させずに新しいバージョンへ移行できる仕組みです。

次に、Kubernetes はサービスディスカバリやロードバランシングを提供します。これにより、マイクロサービス同士が効率よく通信できるようになります。

さらに、Azure では Kubernetes をマネージドサービスとして提供する Azure Kubernetes Service（AKS）があります。AKS を利用することで、Kubernetes クラスターの管理を Azure が担当し、開発者はアプリケーションの開発と運用に集中できます。

また、マイクロサービス間通信の管理にはサービスメッシュという技術が利用されます。サービスメッシュはサービス間通信を制御するためのインフラレイヤーであり、以下のような機能を提供します。

・トラフィック管理  
・リトライ制御  
・サーキットブレーカー  
・サービス間のセキュリティ  
・可観測性（トレーシング・メトリクス）

代表的なサービスメッシュとして Istio や Linkerd などがあります。

## 4. アーキテクチャまたは設計のポイント

このシナリオでは、コンテナベースのマイクロサービスを効率的に運用できる基盤が必要になります。そのため、Kubernetes を利用したアーキテクチャが適しています。

AKS を利用することで、マイクロサービスをコンテナとしてデプロイし、それぞれ独立してスケーリングや更新を行うことが可能になります。ローリングアップデート機能により、サービスを停止することなくアプリケーションを更新できます。

さらに、サービスメッシュを組み合わせることで、マイクロサービス間の通信をより高度に制御できます。これにより、トラフィック管理、リトライ、障害隔離などの機能が利用でき、システム全体の信頼性が向上します。

また、Kubernetes はオンプレミスでも実行可能であるため、Azure とオンプレミスの両方で同じプラットフォームを利用するハイブリッド構成を実現できます。

## 5. 設計判断（なぜこの構成になるか）

この問題の要件を満たすためには、コンテナオーケストレーションとサービス通信管理の両方を提供できるプラットフォームが必要です。

Azure Kubernetes Service（AKS）は Kubernetes をベースとしたマネージドサービスであり、マイクロサービスのデプロイ・スケーリング・更新を自動化できます。これにより、サービス停止なしで更新を行うことが可能になります。

さらにサービスメッシュを導入することで、マイクロサービス間通信の管理をアプリケーションコードから分離できます。これにより、低遅延で信頼性の高い通信を実現しつつ、トラフィック制御や障害対応などの高度な機能を利用できます。

また Kubernetes はオンプレミスでも実行できるため、クラウドとオンプレミスの両方で同じアーキテクチャを採用できる点も大きな利点です。

## 6. 他の選択肢が誤りな理由

この問題では他にもいくつかの Azure サービスが提示されていますが、それぞれ要件の一部を満たしていません。

まず Azure Service Fabric はマイクロサービスアプリケーションの構築に利用できるプラットフォームですが、現在のクラウドネイティブ環境では Kubernetes が標準的な基盤となっています。また、ハイブリッド環境での標準化という観点では Kubernetes の方が柔軟性が高いです。

次に Azure Logic Apps はイベント駆動型のワークフロー自動化サービスであり、マイクロサービスのコンテナオーケストレーションやサービス間通信の管理には適していません。

さらに Azure API Management は API の公開や管理を行う API ゲートウェイサービスですが、マイクロサービスの実行基盤ではありません。そのため、サービス間通信やコンテナ管理を直接提供するものではありません。

## 7. 最終回答

A. Azure Kubernetes Service（AKS）とサービスメッシュ

## 8. まとめ

マイクロサービスアーキテクチャを設計する際には、サービスの独立性・スケーラビリティ・可用性を確保することが重要です。そのためには、コンテナとコンテナオーケストレーション技術を利用することが一般的です。

Azure Kubernetes Service（AKS）は Kubernetes をマネージドで提供し、マイクロサービスのデプロイやスケーリングを効率的に管理できます。また、サービスメッシュを組み合わせることで、マイクロサービス間通信の信頼性や可観測性を高めることができます。

さらに Kubernetes はオンプレミス環境でも実行できるため、クラウドとオンプレミスを組み合わせたハイブリッドアーキテクチャにも適しています。このため、AKS とサービスメッシュの組み合わせは、今回の要件を満たす最も適したソリューションとなります。
