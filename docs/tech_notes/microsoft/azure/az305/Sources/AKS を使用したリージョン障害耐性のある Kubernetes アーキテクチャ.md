# AKS を使用したリージョン障害耐性のある Kubernetes アーキテクチャ

## 1. 背景（シナリオ）

ある企業では、コンテナ化されたアプリケーションを Azure 上で運用するために **Azure Kubernetes Service（AKS）** を導入しようとしています。AKS は Kubernetes のマネージドサービスであり、インフラストラクチャ管理を Azure が担当するため、企業はアプリケーションの開発と運用に集中できます。

しかし、この企業が構築しようとしているシステムはビジネスクリティカルなサービスであり、非常に高い可用性が求められています。そのため、単なるノードレベルの冗長化だけではなく、**Azure データセンター全体の障害にも耐えられる構成**が必要です。

また、運用面での要件もあります。企業は Kubernetes の制御プレーンを自分で管理することを避けたいと考えており、管理作業を可能な限り少なくしたいと考えています。つまり、フルマネージドの AKS のメリットを最大限に活用する必要があります。

このような条件を満たす Kubernetes アーキテクチャを設計する必要があります。

---

## 2. 要件整理

この問題では、AKS クラスター設計に関する複数の要件があります。これらを整理することで最適な構成が見えてきます。

まず、可用性に関する要件があります。企業は単一のデータセンター障害だけでなく、Azure のリージョン全体の障害にも耐えられるシステムを構築する必要があります。

- **Azure データセンター全体の障害に耐える必要がある**

次に、Kubernetes 管理に関する要件があります。企業は Kubernetes の制御プレーン管理をできるだけ Azure に任せたいと考えています。

- **制御プレーン管理の運用負荷を最小化**

さらに、アプリケーションのトラフィックを適切にルーティングする仕組みが必要になります。

- **障害発生時の自動フェイルオーバー**

これらの条件を満たすには、単一クラスターの設計だけではなく、複数リージョンを利用した構成を検討する必要があります。

---

## 3. 技術の基本概念

### Azure Kubernetes Service（AKS）

Azure Kubernetes Service は、Azure が Kubernetes の制御プレーンを管理するマネージドサービスです。ユーザーは主にワーカーノード（データプレーン）の管理に集中することができます。

AKS の重要な特徴を理解するためには、Kubernetes クラスターの構成要素を整理する必要があります。

Kubernetes クラスターは主に次の2つの要素で構成されます。

- **Control Plane（制御プレーン）**
- **Worker Nodes（ワーカーノード）**

制御プレーンは Kubernetes API、スケジューラー、コントローラーなどを含み、クラスター全体を管理する役割を持っています。AKS ではこの制御プレーンを Azure が管理するため、ユーザーはその管理作業を行う必要がありません。

---

### Azure Traffic Manager

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure DNS]]
Azure Traffic Manager は DNS ベースのトラフィックルーティングサービスです。世界中のユーザーからのリクエストを複数のエンドポイントに分散することができます。

Traffic Manager は次のような機能を提供します。

- グローバルトラフィック分散
- ヘルスチェック
- 自動フェイルオーバー

これにより、あるリージョンで障害が発生した場合でも、別のリージョンのサービスにトラフィックを自動的に切り替えることができます。

---

## 4. アーキテクチャまたは設計のポイント

今回のシナリオでは、Azure の単一リージョンだけに依存する構成ではデータセンター障害に耐えることができません。そのため、**複数リージョンに AKS クラスターを配置する構成**が必要になります。

まず、2つの異なる Azure リージョンに AKS クラスターをデプロイします。それぞれのリージョンには独立した Kubernetes クラスターが存在します。

次に、Azure Traffic Manager を使用してユーザーのトラフィックをこれらのクラスターに分散します。Traffic Manager は各クラスターの状態を監視し、障害が発生したリージョンへのトラフィックを自動的に別のリージョンへ切り替えます。

この構成により、ある Azure データセンターまたはリージョン全体が停止した場合でも、もう一方のリージョンでサービスを継続できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題の重要なポイントは **リージョン障害への耐性**です。

可用性ゾーンを使用すると、同じリージョン内の複数データセンターにワークロードを分散できます。しかし、リージョン全体が停止した場合にはゾーン冗長構成でもサービスは停止してしまいます。

そのため、リージョン全体の障害に対応するには **マルチリージョン構成**が必要になります。

AKS クラスターは単一リージョンに属するため、1つのクラスターを複数リージョンにまたがって配置することはできません。そのため、リージョンごとに独立したクラスターを構築する必要があります。

Traffic Manager を使用すると、これらのクラスター間で自動フェイルオーバーを実現できます。このアーキテクチャにより、高可用性と災害復旧の両方を実現できます。

---

## 6. 他の選択肢が誤りな理由

### A 可用性ゾーンに分散した単一クラスター

可用性ゾーンは同一リージョン内のデータセンター障害には対応できます。しかし、リージョン全体の障害には対応できません。

### B 単一ゾーンの AKS クラスター

単一ゾーン構成では可用性が低く、データセンター障害が発生するとクラスター全体が停止してしまいます。

### D 複数リージョンにまたがる単一クラスター

AKS は単一リージョンのサービスであり、1つのクラスターを複数リージョンにまたがって構成することはできません。

---

## 7. 最終回答

C. 2つの異なるリージョンに 2つの AKS クラスターを展開し、Azure Traffic Manager を使用する
