# AKS ステートフルアプリケーションの永続ストレージ（Azure Files RWX）

## 1. 背景（シナリオ）

Azure Kubernetes Service（AKS）では、多くのアプリケーションがコンテナとしてデプロイされます。Kubernetes のコンテナは通常ステートレスに設計されますが、実際のアプリケーションではデータを保持する必要がある **ステートフルアプリケーション**も存在します。

例えば次のようなワークロードです。

- データベース  
- ログ処理システム  
- ファイル共有サービス  
- 機械学習のデータ処理  

このようなアプリケーションでは **ポッドが再起動してもデータが保持される永続ストレージ**が必要になります。

今回のシナリオでは、AKS 上のアプリケーションが次の条件を満たすストレージを必要としています。

まず、ストレージは Kubernetes によって **動的にプロビジョニング**される必要があります。つまり、手動でストレージを作成するのではなく、アプリケーションの要求に応じて自動的に作成される仕組みが必要です。

さらに、Kubernetes ではポッドがノード障害やスケーリングによって **別のノードに再スケジュール**されることがあります。そのため、ストレージはノードに依存しない形でアクセス可能でなければなりません。

このような条件を満たす AKS 用ストレージソリューションを選択する必要があります。

---

## 2. Kubernetes 永続ストレージの基本

Kubernetes では永続ストレージを次の 2 つのリソースで管理します。

まず **PersistentVolume（PV）**です。これはクラスター内で利用できるストレージリソースを表します。

次に **PersistentVolumeClaim（PVC）**です。これはアプリケーションが要求するストレージの仕様を定義します。

アプリケーションは PVC を作成し、Kubernetes がそれを満たす PV を割り当てます。

また、多くのクラウド環境では **StorageClass** を利用してストレージの動的プロビジョニングを実現します。

つまり、PVC が作成されると Kubernetes が自動的にストレージを作成します。

---

## 3. アクセスモードの概念

Kubernetes のストレージには **アクセスモード**という概念があります。これはボリュームをどのようにマウントできるかを定義します。

主なアクセスモードには次の種類があります。

まず **ReadWriteOnce（RWO）**です。このモードでは、ボリュームは一度に一つのノードからのみ読み書きできます。

次に **ReadWriteMany（RWX）**です。このモードでは、複数のノードから同時に読み書きできます。

さらに **ReadOnlyMany（ROX）**があります。これは複数ノードから読み取りのみ可能なモードです。

今回の問題では **ポッドが別のノードに再スケジュールされてもアクセス可能である必要があります。**

この条件は RWX ストレージが最も適しています。

---

## 4. Azure Files の特徴

Azure Files は Azure が提供するマネージドファイル共有サービスです。

このサービスは SMB プロトコルを使用しており、複数のクライアントから同時にアクセスすることができます。

AKS では Azure Files を Kubernetes の CSI ドライバーを使用してマウントできます。

Azure Files の主な特徴は次の通りです。

まず、複数ノードから同時アクセスが可能です。これは Kubernetes の **ReadWriteMany（RWX）** に対応しています。

次に、Kubernetes StorageClass を利用して **動的プロビジョニング**が可能です。

さらに、Azure が管理するサービスであるため、ユーザーはインフラ管理を行う必要がありません。

このような特徴により、Azure Files は AKS のステートフルアプリケーションに適したストレージソリューションとなります。

---

## 5. なぜ Azure Files（RWX）が適切なのか

今回の問題では、次の要件があります。

まず、ストレージは Kubernetes によって **動的にプロビジョニング**される必要があります。

次に、ポッドが別のノードに再スケジュールされても **データにアクセスできる必要があります。**

Azure Files は RWX（ReadWriteMany）アクセスモードを提供しているため、複数ノードから同時にアクセスできます。

つまり、ポッドが別ノードに移動してもストレージは継続して利用できます。

そのため、このシナリオでは Azure Files が最も適したストレージソリューションになります。

---

## 6. 他の選択肢が適さない理由

### Azure Disk（ReadWriteOnce）

Azure Disk は高性能なブロックストレージですが、基本的に **ReadWriteOnce（RWO）** モードです。

これは一度に一つのノードからしかマウントできません。

そのため、ポッドが別のノードに移動するとディスクの再接続が必要になり、可用性要件を満たさない可能性があります。

---

### ポッドの静的 IP

ポッドの IP アドレスはネットワーク設定であり、ストレージとは関係ありません。

データの永続化や共有には使用できません。

---

### コンテナローカルエフェメラルストレージ

エフェメラルストレージはポッドが削除または再スケジュールされるとデータが消えます。

そのため、ステートフルアプリケーションの永続ストレージとしては使用できません。

---

## 7. 最終回答

B. ReadWriteMany アクセスモード付き Azure Files
