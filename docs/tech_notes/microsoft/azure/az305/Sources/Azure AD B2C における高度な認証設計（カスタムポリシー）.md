# Azure AD B2C における高度な認証設計（カスタムポリシー）

## 1. 背景（シナリオ）

ある組織では、外部パートナーがアクセスする複数のアプリケーションを運用しています。これらのアプリケーションは高いセキュリティ要件を満たす必要があり、外部ユーザーの認証を安全かつ柔軟に管理する仕組みが求められています。

この環境では **Azure AD B2C** を利用して外部ユーザー認証を提供する予定です。Azure AD B2C は、顧客やパートナーなどの外部ユーザー向けのID管理サービスであり、ソーシャルログインやカスタム認証フローを実装できます。

今回の設計では、以下のような高度な要件があります。

まず、ユーザーのリスクレベルに応じてアクセスを制御する必要があります。また、ユーザーが Google や Facebook などのソーシャルアカウントを利用してログインできることも求められています。さらに、企業ブランドに合わせたログイン画面のカスタマイズを実装し、ユーザー体験を向上させる必要があります。

加えて、疑わしいアクティビティを検出し、セキュリティインシデントに対応できる仕組みも必要になります。

## 2. 要件整理

この問題では、Azure AD B2C 環境において複数のセキュリティおよびユーザー体験要件を同時に満たす必要があります。

重要な要件を整理すると以下の通りです。

- 外部パートナーの認証
- ソーシャルアカウントログイン
- カスタムブランディング
- ユーザーリスクに応じたアクセス制御
- 不審なアクティビティの監視
- 柔軟な認証フローの設計

これらの要件を実現するには、Azure AD B2C の標準ユーザーフローだけではなく、より高度な認証構成が必要になります。

## 3. 技術の基本概念

Azure AD B2C では、認証フローを構築する方法として主に2種類の仕組みがあります。

まず **ユーザーフロー（Built-in user flows）** です。これは簡単に設定できる認証フローで、一般的なログインやサインアップシナリオに対応しています。

しかし、ユーザーフローはカスタマイズの自由度が限定されています。複雑なセキュリティロジックや外部サービスとの連携には向いていません。

そこで使用されるのが **カスタムポリシー（Custom Policies）** です。

カスタムポリシーは **Identity Experience Framework（IEF）** を利用して構築される高度な認証フローです。この仕組みでは、XMLベースのポリシーを定義することで認証プロセスを細かく制御できます。

カスタムポリシーでは次のような機能を実装できます。

- 複数のIDプロバイダー統合
- カスタム認証ロジック
- 外部API連携
- カスタムUI
- リスクベース認証

この柔軟性により、複雑なセキュリティ要件にも対応できます。

## 4. アーキテクチャまたは設計のポイント

Azure AD B2C の高度な認証設計では、柔軟な認証フローとセキュリティ機能を統合することが重要です。

カスタムポリシーを使用することで、次のような設計が可能になります。

まず、ソーシャルアカウント認証の統合です。Azure AD B2C は複数のIDプロバイダーと連携できるため、Google、Facebook、Microsoftアカウントなどのログインをサポートできます。

次に、ユーザー体験のカスタマイズです。カスタムポリシーを使用すると、ログインページや登録ページのUIを企業ブランドに合わせて自由にカスタマイズできます。

さらに、セキュリティ機能の統合も可能になります。例えば、Azure AD のリスク検出機能や監視機能と連携し、疑わしいログインや異常なアクティビティを検出できます。

このように、カスタムポリシーは複数のセキュリティ機能と認証フローを統合するための中心的な仕組みになります。

## 5. 設計判断（なぜこの構成になるか）

この問題では、複雑な認証要件とセキュリティ要件を同時に満たす必要があります。

具体的には以下の機能が求められています。

- ソーシャルログイン
- カスタムブランディング
- リスクベース認証
- セキュリティ監視
- 柔軟な認証フロー

これらの要件を満たすためには、Azure AD B2C の **カスタムポリシー** を使用する必要があります。

カスタムポリシーを利用すると、Identity Experience Framework を通じて認証プロセスを細かく制御できます。

この仕組みにより、認証フローに外部サービスを組み込み、ユーザーリスク評価や追加認証などの高度なセキュリティ対策を実装できます。

また、カスタムUIを実装できるため、外部パートナー向けに統一されたブランド体験を提供できます。

## 6. 他の選択肢が誤りな理由

### B. Azure AD 条件付きアクセス

条件付きアクセスは非常に強力なアクセス制御機能ですが、主に **Azure Entra ID（企業ユーザー向け）** で使用されます。

Azure AD B2C の高度な認証フローを構築するための中心機能ではありません。

### C. Azure AD Identity Protection

Identity Protection はユーザーリスクやサインインリスクを検出するサービスです。

しかし、この機能だけでは認証フローのカスタマイズやソーシャルログインなどの要件を満たすことはできません。

### D. Identity Experience Framework

Identity Experience Framework はカスタムポリシーを実装するための基盤です。

しかし、実際に構成する機能として問われているのは **カスタムポリシーそのもの**です。

## 7. 最終回答

このシナリオで導入すべき機能は次の通りです。

**Azure AD B2C カスタムポリシー**

したがって **正解は A** です。

## 8. まとめ

Azure AD B2C は外部ユーザー向けのID管理サービスであり、顧客やパートナーの認証を管理するために利用されます。

基本的な認証シナリオではユーザーフローを使用できますが、複雑な認証要件がある場合は **カスタムポリシー** が必要になります。

カスタムポリシーを使用すると、次のような高度な機能を実装できます。

- ソーシャルログイン
- カスタムブランディング
- 外部API連携
- リスクベース認証
- セキュリティ監視

この柔軟性により、厳格なセキュリティ要件を満たすAzure AD B2C認証環境を構築できます。
