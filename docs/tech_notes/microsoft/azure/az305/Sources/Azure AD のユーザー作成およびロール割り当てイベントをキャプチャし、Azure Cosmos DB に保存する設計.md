### Azure AD のユーザー作成およびロール割り当てイベントをキャプチャし、Azure Cosmos DB に保存する設計

---

### 1. 背景（シナリオ）

企業がクラウド環境でユーザー管理を行う場合、多くの組織は **Azure Active Directory（現在の Microsoft Entra ID）** を利用して認証・認可を管理する。Azure ADでは、ユーザーの作成、削除、グループ変更、ロールの割り当てなど、管理操作が行われるたびに **監査ログ（Audit Logs）** が生成される。これらのログはセキュリティ監査やコンプライアンス、運用トラブルの調査において非常に重要である。

今回のシナリオでは、企業は Azure AD 内で発生する「ユーザー作成」と「ロール割り当て」という特定の管理操作を監視し、それらのイベントをリアルタイムで取得して別のデータストアに保存する必要がある。保存先として指定されているのは **Azure Cosmos DB** である。Cosmos DB は分散型 NoSQL データベースであり、低レイテンシでのクエリ処理やグローバル分散が可能なため、監査データの長期保存や分析用途に適している。

しかし Azure AD の監査イベントは、そのまま Cosmos DB に直接保存されるわけではない。Azure AD が生成する監査ログをどこかのサービスで受け取り、それを処理し、必要なデータのみを Cosmos DB に書き込むような **イベント処理パイプライン** を構築する必要がある。この問題は、そのようなパイプラインを Azure サービスでどのように設計するのが適切かを問う内容である。

---

### 2. 要件整理

問題文をよく読むと、設計上の要件は比較的明確である。まず、Azure AD の管理操作、具体的にはユーザー作成やロール割り当てといったイベントを確実に取得する必要がある。これは Azure AD の監査ログに記録されるため、Azure AD のログ出力機能を利用することになる。次に、そのイベントをリアルタイムまたはほぼリアルタイムで処理し、必要な情報だけを Cosmos DB に保存する必要がある。つまり、ログを単に保存するのではなく、途中で **イベント処理やフィルタリング** を行う仕組みが必要になる。

さらに、Azure AD のイベントは組織の規模によっては大量に発生する可能性があるため、ログ取り込みの部分には **高スループットかつスケーラブルなストリーミング基盤** が求められる。また、イベントを処理する部分については、インフラ管理の負担を減らすために **サーバーレスで自動スケール可能な処理環境** が望ましい。これらの要件を満たす Azure サービスの組み合わせを選択する必要がある。

---

### 3. 関連 Azure サービスの説明

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Functions]]
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Event Hubs]]
この問題を理解するうえで重要になるのは、Azure のログ収集とイベント処理の代表的なサービスである **Azure Event Hubs** と **Azure Functions** である。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Monitor]]
Azure Event Hubs は、大量のイベントデータをリアルタイムで取り込むためのストリーミングプラットフォームであり、IoTデータやログデータの取り込みに広く使用されている。Azure Monitor や Azure AD の診断設定を利用すると、監査ログやアクティビティログを Event Hubs に転送することができる。Event Hubs は内部的にパーティション分割されたイベントストリームを持っており、大量のイベントを並列処理できるため、大規模なログ収集システムの入口として非常に適している。

一方で Azure Functions は、イベント駆動型のサーバーレスコンピューティングサービスである。特定のイベントが発生したときに自動的にコードを実行する仕組みであり、Event Hubs から送られてきたメッセージをトリガーとして処理することができる。例えば、Event Hub に送られてきたログの中から「ユーザー作成」や「ロール割り当て」といった特定のイベントだけを抽出し、整形したデータを Cosmos DB に保存する処理を Functions で実装することが可能である。

この二つのサービスを組み合わせることで、Azure AD の監査ログを取り込み、処理し、最終的に Cosmos DB に保存するという一連のイベントパイプラインを構築できる。

---

### 4. 技術的な仕組み

この設計を具体的な処理の流れとして整理すると、次のようになる。まず Azure AD ではユーザー作成やロール割り当てといった管理操作が発生すると、それらは監査ログとして記録される。Azure の診断設定を利用すると、これらの監査ログを外部サービスへストリーミングすることができる。ここでログの送信先として **Azure Event Hubs** を指定する。

Event Hubs は、送られてきた監査イベントをストリームとして蓄積する。次に、Azure Functions を Event Hubs のイベントトリガーとして設定する。すると Event Hub に新しいイベントが到着するたびに Function が実行され、そのイベントデータを処理することができる。Function の中では、イベントの内容を確認し、ユーザー作成やロール割り当てといった特定のイベントだけを抽出し、必要なデータを Cosmos DB に書き込む処理を行う。

この構成を図で表すと、次のようなイベントパイプラインになる。

```
Azure AD
   │
   │ Audit Logs
   ▼
Diagnostic Settings
   │
   ▼
Azure Event Hubs
   │
   │ Event Trigger
   ▼
Azure Functions
   │
   ▼
Azure Cosmos DB
```

この構成は Azure のログ処理パイプラインとして非常に一般的であり、大量のイベントを処理する場合でもスケーラブルに動作する。

---

### 5. 設計判断（なぜこの構成か）

この問題のポイントは、Azure AD の監査ログを **どのサービスで受け取り、どのサービスで処理するのか**という設計判断にある。Azure Event Hubs はログやイベントデータのストリーミング取り込みに最適化されたサービスであり、Azure Monitor や Azure AD のログを受信することができる。これにより、Azure AD から発生するイベントを高スループットで受信できる。

その後の処理には Azure Functions を使用することで、イベントごとにコードを実行し、必要なデータだけを抽出して Cosmos DB に保存することができる。Functions はサーバーレスで自動スケールするため、イベント量が増えても自動的に処理能力を拡張できるという利点がある。さらに Cosmos DB は JSON ベースのデータ保存に適しており、イベントログのような半構造データを保存する用途に向いている。

このように、Event Hubs をイベント取り込み基盤として使用し、Functions をイベント処理エンジンとして使用する構成は、Azure のログ処理アーキテクチャとして非常に自然であり、スケーラブルかつリアルタイム性の高い設計となる。

### 6. 他の選択肢が誤りな理由

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Event Grid]]
選択肢 A は Azure Event Grid と Azure Functions の組み合わせである。Event Grid は Azure リソースイベントを配信するサービスであり、ストレージイベントやリソース管理イベントなどの通知には適している。しかし Azure AD の監査ログは Event Grid にネイティブで配信されるイベントではない。そのため、この構成を実現するには追加の変換処理などが必要になり、設計としては適切とは言えない。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Logic Apps]]
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Log Analytics]]
選択肢 C は Azure Monitor Logs と Logic Apps の組み合わせである。Azure Monitor の Log Analytics ワークスペースにログを収集すること自体は可能だが、そこから Logic Apps を使って Cosmos DB にデータを書き込む場合、ログをポーリングするような処理になりやすく、リアルタイム性やスケーラビリティの面で Event Hubs + Functions の構成より劣る。また、大量のイベント処理には Logic Apps はコスト面でも不利になることが多い。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Stream Analytics]]
選択肢 D は Notification Hubs と Stream Analytics の組み合わせであるが、Notification Hubs はモバイルアプリへのプッシュ通知を送るためのサービスであり、ログ収集やイベントストリーミングとは関係がない。そのため、この問題の要件とは全く一致しない。

---

### 7. 最終回答

この要件を満たす最適な組み合わせは

**Azure Event Hubs と Azure Functions**

である。

したがって正解は **B** となる。

---

### 8. まとめ

Azure AD の監査ログを外部システムに保存する場合、一般的な設計パターンは「ログストリーミングサービス + サーバーレス処理 + データストア」という構成になる。Azure ではこのパターンを実現するために Event Hubs と Functions を組み合わせることが多く、ログ取り込みパイプラインとして広く利用されている。

Azure試験では、このようなイベント処理パターンが頻繁に出題される。特に「ログの取り込み」「イベント処理」「データ保存」という三つの役割をどのサービスが担当するかを理解しておくことが重要である。今回の問題では、Event Hubs がイベント取り込み、Functions がイベント処理、Cosmos DB がデータ保存という役割を担う構成が最も適切である。
