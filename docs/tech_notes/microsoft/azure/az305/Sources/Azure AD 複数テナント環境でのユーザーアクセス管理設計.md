# Azure AD 複数テナント環境でのユーザーアクセス管理設計

## 1. 背景（シナリオ）
多国籍企業は複雑な Azure AD 環境を持ち、複数の子会社がそれぞれ独自の Azure AD テナントを所有しています。目的は、各子会社のユーザーが新しいアカウント作成や複雑なオンボーディング作業を行わずに、中央テナントの共有リソースへアクセスできるようにすることです。また、ユーザーアクセスは各子会社のセキュリティポリシーに準拠する必要があります。

## 2. 要件整理
- 複数の Azure AD テナント間でユーザーアクセスを管理
- 子会社のセキュリティポリシーに準拠させる
- 新規アカウント作成や複雑なオンボーディング不要
- 共有リソースへの安全なアクセスを提供

## 3. 技術の基本概念
- **Azure AD B2B コラボレーション**
  - 他の組織のユーザーをゲストとして中央テナントに招待
  - 既存の資格情報でログイン可能
  - セキュリティポリシーを維持しつつリソースを共有

- **クロステナントアクセス設定**
  - 複数テナント間でのアクセス方法と制御を定義
  - 子会社ごとにアクセス権をカスタマイズ可能
  - 各テナントのポリシーに基づいてアクセス制御を適用

## 4. アーキテクチャまたは設計のポイント
- 中央テナントで Azure AD B2B を設定し、子会社ユーザーをゲストとして招待
- クロステナントアクセス設定で、子会社テナントのポリシーに準拠したアクセス制御を定義
- ゲストユーザーは既存の資格情報でアクセス可能
- 最小権限の原則に基づき、必要なリソースへのアクセスのみを付与

## 5. 設計判断（なぜこの構成になるか）
- Azure AD B2B により、中央テナントでの新規アカウント作成を不要に
- クロステナントアクセス設定で子会社ポリシーを尊重した安全なアクセスを保証
- ゲストユーザー管理とアクセス制御を集中化し、オンボーディング負荷を最小化

## 6. 他の選択肢が誤りな理由
- **Azure AD DS + フェデレーテッドID**: オンプレAD連携向きでクロステナントアクセス管理には非効率
- **条件付きアクセス付きアプリケーションプロキシ**: アプリ単位のセキュリティ制御は可能だが、複数テナント管理やゲストユーザーアクセスには不十分
- **Azure AD 外部ID + アイデンティティガバナンス**: 外部ユーザー向けのガバナンス中心で、B2B のシームレスアクセス体験には及ばない

## 7. 最終回答
**A. Azure AD B2B とクロステナントアクセス設定との連携**

## 8. まとめ
複数テナント間でのユーザーアクセスを効率的に管理するには、Azure AD B2B とクロステナントアクセス設定を組み合わせることが最適です。ゲストユーザーは既存の資格情報でアクセス可能で、子会社のセキュリティポリシーを尊重しつつ、共有リソースへのアクセスを安全に提供できます。
