# Azure App Service とオンプレミスファイルサーバーのハイブリッドファイル共有設計

## 1. 背景（シナリオ）

ある企業では、.NET Web アプリケーションである **App2** を Azure App Service 上で運用しています。このアプリケーションは、ユーザーがアップロードしたファイルや生成されたデータファイルを保存する機能を持っています。

しかし、企業の運用環境ではファイルの保存場所に関する追加要件があります。アプリケーションが保存したファイルは、オンプレミス環境に存在するファイルサーバーにも複製される必要があります。さらに、オンプレミスのユーザーはそのファイルに対して **SMB（Server Message Block）プロトコルを使用してアクセスできる必要**があります。

この要件は、クラウドアプリケーションとオンプレミスのファイル共有を統合する **ハイブリッドストレージアーキテクチャ**を必要とします。また、企業は運用管理の負担を最小限に抑えたいと考えているため、継続的な同期処理は可能な限り自動化されたサービスによって実現する必要があります。

---

## 2. 要件整理

この問題を理解するためには、アーキテクチャ設計に影響する要件を整理することが重要です。問題文から読み取れる条件を整理すると、主にファイルアクセス方式、データ同期、運用管理の3つの観点が重要になります。

まず、ファイルアクセス方式に関する要件があります。オンプレミスのユーザーは既存のファイルサーバーと同じ方法でファイルにアクセスする必要があります。

- オンプレミスユーザーは **SMB プロトコルでアクセスする必要がある**

次に、データ配置に関する要件があります。アプリケーションは Azure App Service 上で動作しているため、クラウド側にファイルの保存場所が必要になります。

- App Service からアクセスできる **クラウドストレージ**

さらに、オンプレミス環境との同期が必要です。

- クラウドとオンプレミスのファイルサーバーを **自動的に同期**

最後に、運用管理の要件があります。

- 継続的な管理作業を **最小限に抑える**

これらの条件を満たす Azure サービスの組み合わせが必要になります。

---

## 3. 技術の基本概念

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Files]]
### Azure Files

Azure Files は Azure が提供するマネージドファイル共有サービスであり、クラウド上で SMB プロトコルを使用したファイル共有を提供します。Azure Files を使用すると、従来のオンプレミスファイルサーバーと同じようにネットワークドライブとしてファイル共有を利用することができます。

Azure Files の重要な特徴を理解するためには、ファイル共有サービスとしての機能を整理する必要があります。

- SMB プロトコルをサポート
- Windows や Linux からマウント可能
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Virtual Machines]]
- Azure App Service や VM から利用可能
- マネージドストレージサービス

これにより、アプリケーションはクラウド上のファイル共有に直接ファイルを書き込むことができます。

---

### Azure File Sync

Azure File Sync は、オンプレミスのファイルサーバーと Azure Files を同期するためのサービスです。このサービスを使用すると、オンプレミスのファイルサーバーを Azure ファイル共有のキャッシュとして動作させることができます。

Azure File Sync を理解するためには、ハイブリッドファイル共有の仕組みを整理する必要があります。

- Azure Files をクラウド側のファイルストレージとして使用
- オンプレミスの Windows ファイルサーバーと同期
- 変更されたファイルを自動的にレプリケーション
- ローカルキャッシュによる高速アクセス

この仕組みにより、クラウドとオンプレミスの両方で同じファイルを共有することができます。

---

## 4. アーキテクチャまたは設計のポイント

このシナリオでは、Azure App Service がファイルをクラウドストレージに保存し、そのファイルをオンプレミス環境に同期する必要があります。最も自然な構成は、Azure Files をクラウド側のファイル共有として使用し、Azure File Sync を使用してオンプレミスのファイルサーバーと同期する構成です。

まず、App Service は Azure Files にファイルを書き込みます。Azure Files は SMB ベースのクラウドファイル共有であるため、アプリケーションから簡単に利用することができます。

次に、Azure File Sync を使用してオンプレミスのファイルサーバーと Azure Files を同期します。この同期はサービスによって自動的に管理されるため、運用担当者が手動でデータコピーを行う必要はありません。

オンプレミスユーザーは、従来通りローカルのファイルサーバーに SMB 接続することでファイルにアクセスできます。Azure File Sync によって、クラウドとオンプレミスのデータは常に同期された状態になります。

このアーキテクチャは、クラウドアプリケーションとオンプレミスユーザーの両方の要件を満たすハイブリッドファイル共有モデルになります。

---

## 5. 設計判断（なぜこの構成になるか）

この問題の重要なポイントは **SMB アクセスと自動同期の両立**です。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Blob Storage]]
Azure Blob Storage はオブジェクトストレージであり、HTTP/HTTPS を使用してアクセスされます。そのため SMB プロトコルを使用したファイル共有には適していません。

一方で Azure Files は SMB プロトコルをサポートしており、Windows ファイルサーバーと同じようにファイル共有を提供できます。

さらに Azure File Sync を使用すると、オンプレミスのファイルサーバーとクラウドのファイル共有を自動的に同期できます。この機能により、オンプレミスユーザーは従来のファイルサーバーを使用し続けながら、クラウドストレージとデータを共有することができます。

この組み合わせは Azure のハイブリッドファイル共有ソリューションとして一般的に推奨されている構成です。

## 6. 他の選択肢が誤りな理由

### Azure Blob Storage

Blob Storage はオブジェクトストレージであり、SMB プロトコルをサポートしていません。そのためオンプレミスユーザーが SMB 経由でファイルにアクセスする要件を満たすことができません。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Data Box]]
### Azure Data Box Gateway

Data Box Gateway は大規模データ転送やアーカイブ用途に設計されたサービスです。継続的なファイル同期や SMB ベースのファイル共有には適していません。

### Azure Import/Export Service

Import/Export Service は大量データの初期移行に使用されるサービスであり、継続的な同期やファイル共有の用途には適していません。
