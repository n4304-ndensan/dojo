# Azure App Service のゼロダウンタイムデプロイ（Deployment Slots）

## 1. 背景（シナリオ）

クラウドアプリケーションでは、サービスを停止せずに新しいバージョンをリリースすることが重要です。ユーザーが利用している Web アプリケーションを更新する際に、アプリケーションを停止してしまうとサービスの可用性が低下し、ユーザー体験にも影響を与えてしまいます。

そのため、多くのクラウドアーキテクチャでは **ゼロダウンタイムデプロイ（Zero-Downtime Deployment）**の仕組みを採用しています。

今回のシナリオでは、Azure App Service 上の Web アプリケーションに対してアップデートをデプロイする必要があります。

さらに、このデプロイプロセスには **テスト用のステージング環境**を含める必要があります。つまり、新しいバージョンを本番環境にリリースする前に、本番に近い環境でテストを実施する必要があります。

そして、最も重要な要件として、ステージング環境から本番環境への切り替えは **ダウンタイムなしで行われる必要があります。**

このような要件を満たす Azure App Service の機能を選択する必要があります。

---

## 2. 要件整理

この問題では、デプロイ戦略に関する要件を整理することが重要です。

まず、新しいアプリケーションバージョンをテストするための **ステージング環境**が必要です。

次に、テストが完了した後に、本番環境へ安全にリリースする必要があります。

さらに、本番環境への切り替えは **ダウンタイムなしで行う必要があります。**

また、もし問題が発生した場合には、迅速にロールバックできる仕組みがあることも望ましいです。

このような要件を整理すると次のようになります。

- ステージング環境でのテスト  
- 本番環境への安全な切り替え  
- ゼロダウンタイムリリース  
- 簡単なロールバック  

Azure App Service では、このようなデプロイ要件を満たすために **Deployment Slots（デプロイスロット）**という機能が提供されています。

---

## 3. Deployment Slots の概要

Deployment Slots は Azure App Service の機能であり、一つの Web アプリケーションに複数のデプロイ環境を作成することができます。

通常、App Service には **production スロット**が存在します。

これに加えて、次のような追加スロットを作成できます。

- staging  
- testing  
- development  

各スロットは独立した環境として動作し、同じ App Service Plan 上で実行されます。

つまり、ステージング環境に新しいアプリケーションをデプロイし、本番環境には影響を与えずにテストを実施できます。

---

## 4. スロットスワップ（Slot Swap）

Deployment Slots の最も重要な機能が **スロットスワップ（slot swap）**です。

スロットスワップとは、ステージング環境と本番環境のトラフィックを入れ替える操作です。

この操作は次のように行われます。

まず、新しいアプリケーションバージョンを **staging スロット**にデプロイします。

次に、ステージング環境でテストや検証を実施します。

問題がなければ **swap 操作**を実行します。

この swap により、ステージング環境が production になり、production 環境が staging になります。

この切り替えはほぼ瞬時に行われるため、ユーザーに対してダウンタイムを発生させません。

---

## 5. ゼロダウンタイムデプロイの仕組み

Deployment Slots を利用すると、次のようなゼロダウンタイムデプロイを実現できます。

まず、新しいバージョンを staging スロットにデプロイします。この時点ではユーザーは production スロットのアプリを利用しています。

次に、ステージング環境でテストやパフォーマンス確認を実施します。

問題がない場合、swap 操作を行います。

swap 操作はトラフィックルーティングを切り替えるだけなので、ユーザー側のダウンタイムは発生しません。

さらに、swap 後に問題が発生した場合は **再度 swap するだけでロールバック**できます。

---

## 6. Deployment Slots のメリット

Deployment Slots を利用すると、次のようなメリットがあります。

まず、本番環境と同じインフラでテストを行うことができます。これにより、本番環境でのみ発生する問題を事前に検出できます。

次に、ゼロダウンタイムでデプロイできます。

さらに、ロールバックが非常に簡単です。

また、Azure App Service にネイティブに統合されているため、追加のインフラや複雑な構成は不要です。

---

## 7. 他の選択肢が適さない理由

### Azure DevOps Blue-Green Deployment

Blue-Green デプロイは有効な戦略ですが、Azure App Service では Deployment Slots を使用する方がシンプルでネイティブな方法です。

---

### App Service のクローンとホスト名交換

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure DNS]]
この方法は管理が複雑であり、設定の不整合や DNS の問題が発生する可能性があります。

---

### FTP デプロイと再起動

FTP デプロイでは App Service の再起動が必要になり、短時間のダウンタイムが発生します。

---

## 8. 最終回答

A. Deployment Slots（展開スロット）
