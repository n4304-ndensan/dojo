# Azure Batch における優先度別ワークロードの VM 設計

## 1. 背景（シナリオ）

ある組織では、Azure Batch を使用して Windows 仮想マシン上で複数の計算タスクを処理するシステムを設計しています。このシステムには、2種類のワークロードが存在します。

1つ目は、優先度が高く計算負荷の高いタスクです。これらのタスクはビジネスにとって重要であり、確実にコンピューティングリソースへアクセスできる必要があります。また、処理時間を短縮するために迅速なスケーリングも必要です。

2つ目は、優先度が低く中断を許容できるワークロードです。このタイプのジョブはリソースが空いているときに処理されればよく、多少の中断が発生しても問題ありません。

さらに、このソリューションではコスト最適化も重要な要件となっています。そのため、必要に応じて Azure Reserved Instances などを活用してコスト削減を実現する必要があります。

## 2. 要件整理

このシナリオの要件を整理すると、次のようなポイントが重要になります。

- Azure Batch を利用した計算処理  
- 高優先度の計算集約型タスク  
- 低優先度で中断可能なワークロード  
- コンピューティングリソースの効率的利用  
- 高優先度タスクのリソース保証  
- コスト最適化（Reserved Instances の利用）

これらの要件から、ワークロードの特性に応じて異なる仮想マシンタイプとプール構成を使用する設計が必要になります。

## 3. 技術の基本概念

Azure Batch は、大規模な並列処理や計算処理を実行するためのサービスです。Batch はジョブを複数の仮想マシンに分散して実行することができます。

Azure Batch では、仮想マシンプールを構成してジョブを処理します。このとき、仮想マシンの種類をワークロードの特性に応じて選択することが重要です。

主に次の2種類の仮想マシンタイプが利用されます。

まず、Burstable VM（Bシリーズ）です。このタイプは通常は低コストで動作し、必要に応じてCPUパフォーマンスを一時的に向上させることができます。比較的負荷が低く、断続的にCPUを利用するワークロードに適しています。

次に、Dedicated VM（専用仮想マシン）です。このタイプは常に専用の計算リソースを確保できるため、安定したパフォーマンスが必要なミッションクリティカルな処理に適しています。

## 4. アーキテクチャまたは設計のポイント

Azure Batch 環境を設計する際には、ワークロードの特性に応じて異なる VM タイプを使用することが重要です。

まず、優先度の低いワークロードについて考えます。このタイプのジョブは中断を許容できるため、コスト効率の高い仮想マシンを利用することが適しています。

そのため、Burstable VM を利用した Batch サービスを使用することで、リソース使用率を最適化しながらコストを抑えることができます。

一方で、優先度の高い計算集約型タスクについては、常に安定したリソースが必要になります。このようなタスクではリソース競合を避ける必要があるため、Dedicated VM を使用することが適しています。

さらに、Dedicated VM はユーザーサブスクリプション内で管理できるため、Azure Reserved Instances を利用してコストを最適化することも可能になります。

## 5. 設計判断（なぜこの構成になるか）

この問題では、ワークロードを2つのタイプに分けて設計することが重要です。

低優先度のワークロードでは、コスト効率と柔軟なスケーリングが重要です。そのため、Burstable VM を利用した Azure Batch プールが適しています。

一方で、高優先度タスクでは、安定したパフォーマンスとリソース保証が重要です。そのため、Dedicated VM を使用したユーザーサブスクリプション構成が最適になります。

このように、ワークロードの特性に応じて VM タイプを使い分けることで、リソース利用を最適化しながらコスト効率の高いアーキテクチャを実現できます。

## 6. 他の選択肢が誤りな理由

他の選択肢の構成では、優先度の異なるワークロードを効率的に管理することができません。

低優先度 VM は中断される可能性があるため、Burstable ワークロードの安定した処理には適していません。また、すべてのワークロードを同一タイプの VM で処理すると、コスト効率やリソース利用の観点で最適な設計とはなりません。

そのため、Burstable VM と Dedicated VM を組み合わせるアプローチが最もバランスの取れたソリューションになります。

## 7. 最終回答

- オーバーサブスクライブ可能なワークロード  
A. バースタブル仮想マシンによる Batch サービス

- 優先度の高いタスク  
D. 専用仮想マシンによるユーザーサブスクリプション

## 8. まとめ

Azure Batch を使用した計算処理では、ワークロードの優先度に応じて仮想マシンタイプを使い分けることが重要です。

低優先度のワークロードにはコスト効率の高い Burstable VM を使用することでリソース利用を最適化できます。一方で、優先度の高いタスクには Dedicated VM を使用することで安定した計算リソースを確保できます。

このように両者を組み合わせることで、パフォーマンスとコスト効率を両立した Azure Batch ソリューションを構築できます。
