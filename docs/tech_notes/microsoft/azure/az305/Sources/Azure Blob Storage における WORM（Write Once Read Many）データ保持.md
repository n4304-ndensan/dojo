# Azure Blob Storage における WORM（Write Once Read Many）データ保持

## 1. 背景（シナリオ）

ある企業では、法規制や監査要件を満たすためにデータアーカイブソリューションを設計しています。保存対象となるデータには財務記録やログデータなどが含まれており、これらのデータは一定期間の保存が義務付けられています。

規制要件では、データが保存された後に変更または削除されないことを保証する必要があります。具体的には、データは **WORM（Write Once Read Many）状態**で保存され、5年間保持されなければなりません。

WORMとは、一度書き込まれたデータが変更されることなく、読み取りのみ可能な状態で保存される仕組みを指します。この仕組みは、監査ログ、金融取引履歴、医療データなどの分野でよく使用されます。

Azure 上でこのようなアーカイブソリューションを構築する場合、データの削除や変更を防ぐストレージ機能を選択する必要があります。

---

## 2. 要件整理

この問題では、データアーカイブに関するいくつかの重要な要件があります。まずはシステム設計に影響する条件を整理する必要があります。

最初の要件は保存期間です。企業は規制要件を満たすためにデータを **5年間保存**する必要があります。

次に重要なのはデータの不変性です。保存されたデータは書き込み後に変更されてはいけません。これは監査証跡の完全性を確保するためです。

これらの条件を整理すると、以下のような要件になります。

- データは一度書き込まれた後に変更できない
- データを削除することもできない
- 保存期間は5年間
- 規制コンプライアンスを満たす必要がある

これらの条件を満たすためには、単なるバックアップ機能ではなく **不変ストレージ（Immutable Storage）**のような仕組みが必要になります。

---

## 3. 技術の基本概念

### WORM（Write Once Read Many）

WORMは「Write Once Read Many」の略であり、データを一度書き込むと変更や削除ができないストレージモデルです。このモデルはコンプライアンス要件や監査要件に対応するために使用されます。

通常のストレージでは、データは上書きや削除が可能です。しかしWORMストレージでは、保存期間中はデータの変更や削除が禁止されます。これにより、データの改ざんを防ぐことができます。

WORMは特に以下のようなデータで使用されます。

- 財務データ
- 監査ログ
- 医療記録
- 法的証拠データ

これらのデータは長期間の保存が求められるため、不変性を保証するストレージが必要になります。

---

### Azure Blob Storage の Immutable Storage

Azure Blob Storage には **Immutable Storage（不変ストレージ）**と呼ばれる機能があります。この機能を使用すると、BLOB データに対して WORM ポリシーを適用することができます。

Immutable Storage では、保存ポリシーを設定することでデータの変更や削除を防ぐことができます。このポリシーは時間ベースで設定されることが多く、例えば「5年間変更不可」といった設定が可能です。

この機能を利用することで、クラウド環境でもコンプライアンス要件を満たしたデータ保存が可能になります。

---

## 4. アーキテクチャまたは設計のポイント

Azure Blob Storage の Immutable Storage は、主にコンテナレベルで設定されます。コンテナに対して保持ポリシーを設定することで、そのコンテナ内のすべての BLOB に対して不変性が適用されます。

この仕組みでは、データが書き込まれた後に保持期間が開始されます。保持期間が終了するまでは、データの削除や上書きが許可されません。

また、Immutable Storage では以下のようなポリシーを使用できます。

まず時間ベースの保持ポリシーがあります。このポリシーでは、一定期間データの変更や削除を禁止します。

次にリーガルホールドがあります。これは法的要件によりデータ保持が必要な場合に使用されます。リーガルホールドが設定されている場合、保持期間に関係なくデータを削除することはできません。

このような仕組みにより、Azure Blob Storage は金融業界や医療業界などの規制要件に対応することができます。

---

## 5. 設計判断（なぜこの構成になるか）

今回の問題では **WORM状態で5年間保存する必要がある**という点が最も重要です。

Azure Blob Storage の Immutable Storage は、まさにこの要件を満たすために設計されています。保持ポリシーを設定すると、その期間中はデータの削除や変更が不可能になります。

この機能により、データの改ざんや意図的な削除を防ぐことができます。また、規制コンプライアンスを満たすデータ保持が可能になります。

他のストレージ機能はデータ保護には役立ちますが、不変性を保証するものではありません。そのため WORM 要件を満たすためには Immutable Storage を使用する必要があります。

---

## 6. 他の選択肢が誤りな理由

### Blob Soft Delete

Soft Delete は削除されたデータを一定期間復元できるようにする機能です。これは誤削除からの復旧には役立ちますが、データの変更や削除を完全に禁止するものではありません。そのため WORM 要件を満たすことはできません。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Storage Account]]
### Azure Storage Encryption

ストレージ暗号化はデータの機密性を保護する機能です。しかし暗号化はデータの削除や変更を防ぐものではありません。したがって不変性を保証することはできません。

### Blob Versioning

Blob Versioning はデータが更新されるたびに以前のバージョンを保存する機能です。しかし現在のデータを削除したり変更したりすることは可能です。そのためWORM要件には適していません。
