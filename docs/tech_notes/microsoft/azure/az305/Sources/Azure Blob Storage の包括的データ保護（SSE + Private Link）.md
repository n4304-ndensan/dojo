# Azure Blob Storage の包括的データ保護（SSE + Private Link）

## 1. 背景（シナリオ）

ある組織では、Azure Blob Storage に保存されている機密データを管理しています。このデータには顧客情報や機密ビジネスデータが含まれており、厳格なセキュリティとコンプライアンス要件を満たす必要があります。

近年、データ保護に関する規制はますます厳格になっており、組織は保存データの暗号化や安全なデータ通信など、包括的なセキュリティ対策を導入する必要があります。

このため、データが保存されているとき（保存時）と、ネットワークを通じて送信されるとき（転送中）の両方で安全性を確保する必要があります。また、不正アクセスを防ぐためのアクセス制御も重要な要素になります。

## 2. 要件整理

この問題では「包括的なデータ保護戦略」を構築することが求められています。つまり、単一のセキュリティ機能ではなく、複数のセキュリティレイヤーを組み合わせる必要があります。

問題文から読み取れる要件を整理すると次のようになります。

まず、保存されているデータを暗号化する必要があります。次に、データの送信経路を安全に保護する必要があります。

さらに、不正アクセスを防ぐためにネットワークレベルでのアクセス制御も必要になります。

これらの要件を整理すると次のようになります。

保存時のデータ暗号化  
転送経路の保護  
ネットワークレベルのアクセス制御  
機密データのコンプライアンス対応  

このような要件を満たすためには、Azure の暗号化機能とネットワークセキュリティ機能を組み合わせる必要があります。

## 3. 技術の基本概念

Azure Blob Storage では、保存されるデータを自動的に暗号化する **Storage Service Encryption（SSE）** が提供されています。

SSE は、データがストレージに書き込まれる際に自動的に暗号化され、読み取る際に自動的に復号されます。この暗号化はアプリケーション側で追加のコードを書く必要がありません。

さらに、組織のセキュリティ要件によっては **Customer Managed Keys（CMK）** を使用することができます。CMK を使用すると、暗号化キーを Azure Key Vault で管理することができ、企業がキーのライフサイクルを完全に管理できます。

また、データアクセスのネットワークセキュリティを強化するために **Azure Private Link** を利用できます。

Private Link を利用すると、Azure Storage へのアクセスをプライベートエンドポイント経由に限定でき、インターネット経由のアクセスを防ぐことができます。

## 4. アーキテクチャまたは設計のポイント

Azure Blob Storage の機密データを保護するためには、暗号化とネットワークセキュリティの両方を組み合わせる設計が重要です。

まず、Storage Service Encryption を有効化し、保存データを暗号化します。さらに、Customer Managed Key を使用することで、暗号化キーを Azure Key Vault で管理できます。

次に、Azure Private Link を利用してストレージアカウントへのアクセスをプライベートネットワークに限定します。

この構成では、ストレージアカウントはパブリックインターネットから直接アクセスできなくなり、仮想ネットワーク内のリソースのみがアクセスできるようになります。

このように複数のセキュリティレイヤーを組み合わせることで、包括的なデータ保護を実現できます。

## 5. 設計判断（なぜこの構成になるか）

Customer Managed Key を使用した SSE によって、保存されているデータは常に暗号化されます。さらに、暗号化キーを Key Vault で管理することで、キーの回転やアクセス制御を安全に実施できます。

一方、Private Link を使用することで、ストレージへのアクセスを完全にプライベートネットワークに制限できます。これにより、パブリックインターネット経由の攻撃リスクを大幅に削減できます。

この2つを組み合わせることで、データ保護の重要な3つの要素を満たすことができます。

保存時の暗号化  
安全なネットワークアクセス  
厳格なアクセス制御  

## 6. 他の選択肢が誤りな理由

### C Azure Data Lake Storage Gen2

Data Lake Storage Gen2 は主にビッグデータ分析やデータレイク用途に使用されるサービスです。セキュリティ機能はありますが、Blob Storage の包括的なデータ保護戦略として直接的な対策ではありません。

### D 共有アクセス署名（SAS）

SAS は一時的なアクセス権限を提供する仕組みであり、アクセス制御の補助的な手段です。しかし、保存時暗号化やネットワーク保護といった包括的なセキュリティ対策にはなりません。

## 7. 最終回答

A. **顧客管理キーを使用した Storage Service Encryption（SSE）**  
B. **Azure Private Link によるプライベートエンドポイントアクセス**

## 8. まとめ

Azure Blob Storage の機密データを保護するためには、暗号化とネットワークセキュリティの両方を実装する必要があります。

Storage Service Encryption に Customer Managed Key を組み合わせることで保存データの暗号化を強化できます。また、Azure Private Link を利用することでストレージへのアクセスをプライベートネットワークに限定できます。

この2つを組み合わせることで、Azure 上での包括的なデータ保護戦略を実現できます。

試験では **「保存時暗号化」「Customer Managed Key」「Private Link」「プライベートエンドポイント」** といったキーワードが出てきた場合、この組み合わせが正解になるケースが多いです。
