# Azure Cosmos DB における RBAC を使用したきめ細かなアクセス制御

## 背景

Azure Cosmos DB を利用したデータベースソリューションを設計する際、多くの組織では複数のユーザーが同じサブスクリプション内のデータベースにアクセスします。  
そのため、ユーザーの役割に応じてアクセス権限を適切に制御する仕組みが必要になります。

例えば、あるユーザーはデータを読み取るだけの権限が必要であり、別のユーザーはデータの追加や更新が可能である必要があります。また、管理者は削除操作などのより強い権限を持つ必要があります。

このような要件を満たすためには、ユーザーごとに細かくアクセス権限を設定できる仕組みが必要になります。

---

## Azure Cosmos DB のアクセス制御の基本

Azure Cosmos DB では、データベースへのアクセス制御を実現するために複数の方法が用意されています。その中でも最も推奨される方法は **Azure Active Directory と RBAC を組み合わせたアクセス制御**です。

Azure AD を利用することで、ユーザーの認証を Azure のアイデンティティ管理基盤で一元管理できます。  
さらに RBAC（Role-Based Access Control）を利用することで、ユーザーまたはグループに対して役割ベースの権限を割り当てることが可能になります。

この仕組みによって、ユーザーは自分に割り当てられた役割に応じた操作のみを実行できるようになります。

---

## RBAC を使用する理由

このシナリオでは、複数のユーザーが異なる権限を持つ必要があります。  
RBAC を使用することで、ユーザーの役割ごとに適切な権限を割り当てることができます。

RBAC を利用することで次のようなメリットがあります。

まず、ユーザーごとの権限を細かく設定できるため、必要最小限の権限のみを付与できます。これにより、セキュリティリスクを最小限に抑えることができます。

次に、Azure AD と統合されることで、多要素認証や条件付きアクセスなどのセキュリティ機能を活用できます。これにより、企業のセキュリティポリシーやコンプライアンス要件にも対応しやすくなります。

さらに、ユーザーやグループ単位でロールを割り当てることができるため、組織の変更にも柔軟に対応できます。

---

## 他の選択肢が適さない理由

この問題には他にもいくつかの選択肢がありますが、それぞれ適切ではありません。

まず、共有アクセス署名（SAS）は主に Azure Storage の一時的なアクセス制御に使用される仕組みであり、ユーザーごとの長期的な権限管理には向いていません。

次に、Resource Token はアプリケーションが特定のリソースにアクセスする際に利用されることがありますが、複数ユーザーのロール管理には管理が複雑になりやすいという問題があります。

また、Master Key を使用したアクセス管理はすべてのデータへの完全なアクセス権を与えるため、最小権限の原則に反してしまいます。そのためセキュリティ面でも推奨されません。

---

## 結論

このシナリオでは、ユーザーごとに細かくアクセス権限を設定し、セキュリティとコンプライアンスを維持する必要があります。

そのため、**Azure AD 認証と RBAC を使用してデータベースレベルでロールを割り当てる方法**が最も適切なアプローチになります。
