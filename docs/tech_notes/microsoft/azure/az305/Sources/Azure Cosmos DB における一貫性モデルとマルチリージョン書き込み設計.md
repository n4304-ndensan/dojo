# Azure Cosmos DB における一貫性モデルとマルチリージョン書き込み設計

## 1. 背景（シナリオ）

グローバルに展開されるクラウドアプリケーションでは、複数の地域にユーザーが存在するため、データベースを複数リージョンに分散配置することが一般的である。Azure Cosmos DB はこのような用途のために設計された分散データベースであり、世界中のリージョンにレプリカを配置して低遅延アクセスを提供できる。

しかし、グローバル分散システムでは常に **「一貫性（Consistency）」と「遅延（Latency）」のトレードオフ**が存在する。データの整合性を厳密に保証しようとすると、リージョン間の同期が必要になり、その結果として読み取りや書き込みの遅延が増加する。一方、遅延を最小化するためにローカル書き込みを許可すると、データの整合性を弱める必要がある。

今回のシナリオでは、Azure Cosmos DB を使用した **グローバル分散アプリケーション**を構築しており、「すべてのリージョンで強い一貫性を確保する」という要件が提示されている。さらに、「読み取りおよび書き込みの遅延を最小限に抑える」という要件も存在する。この問題は、Cosmos DB の **一貫性レベルとマルチリージョン書き込みの関係**を理解しているかどうかを問うものである。

---

## 2. 要件整理

この問題の設計要件を整理すると、次の三つの重要なポイントが見えてくる。

第一に、すべてのリージョンにおいて **強い一貫性（Strong Consistency）** を保証する必要がある。これは、あるクライアントが書き込みを行った後、どのリージョンから読み取っても必ずその書き込み結果が反映されている状態を意味する。つまり、データの読み取りが常に最新状態であることを保証する必要がある。

第二に、読み取りおよび書き込みの遅延を最小限に抑える必要がある。グローバル分散環境では、リージョン間通信が発生すると遅延が増えるため、アーキテクチャ設計によってできる限りその影響を小さくする必要がある。

第三に、Cosmos DB の設計上の制約として **強い一貫性とマルチリージョン書き込みは両立しない**という点を理解する必要がある。つまり、すべてのリージョンで強い一貫性を保証したい場合、書き込みリージョンは単一に限定する必要がある。

---

## 3. Azure Cosmos DB の一貫性レベル

Azure Cosmos DB では、5つの一貫性レベルが提供されている。

最も厳密なものが **Strong Consistency（強い一貫性）**である。このレベルでは、書き込みが完了した後のすべての読み取りが常に最新データを返すことが保証される。これは分散システムにおける **線形化可能性（linearizability）** を保証するモデルである。

その次に厳密なのが **Bounded Staleness（束縛された古さ）**である。このモデルでは、読み取り結果が最大で一定数の更新または一定時間だけ古い状態になることが許容される。

さらに **Session Consistency（セッション一貫性）** は、同一クライアントセッション内では強い一貫性が保証されるが、他のクライアントからは最新状態が見えない可能性がある。

残りの **Consistent Prefix** と **Eventual Consistency** はさらに弱い保証であり、グローバル分散システムで遅延を最小化する代わりに整合性の保証が弱くなる。

---

## 4. Cosmos DB におけるマルチリージョン書き込み

Cosmos DB は **マルチリージョン書き込み（Multi-region writes）**をサポートしており、複数のリージョンで同時に書き込みを受け付けることができる。この機能を有効にすると、ユーザーに最も近いリージョンで書き込みが行われるため、書き込み遅延を大幅に削減できる。

しかし、この仕組みには重要な制約がある。複数リージョンで書き込みを許可する場合、リージョン間での競合解決が必要になるため、**Strong Consistency を維持することができない**。そのため、マルチリージョン書き込みを使用する場合は、Session Consistency や Bounded Staleness などのより弱い一貫性モデルを採用する必要がある。

逆に言えば、**Strong Consistency を選択した場合は単一リージョン書き込みのみが許可される**。書き込みは特定のリージョンで処理され、その後すべてのリージョンに同期されるため、どこから読み取っても同じ結果が保証される。

---

## 5. 設計判断（なぜこの構成か）

今回の問題では「すべてのリージョンで強い一貫性を確保する」という要件が最も重要である。この要件を満たすためには、Cosmos DB の最も厳密な一貫性モデルである **Strong Consistency** を使用する必要がある。

しかし Strong Consistency を使用すると、マルチリージョン書き込みは利用できない。その理由は、複数リージョンで同時に書き込みが行われると、すべてのリージョンで即座に同一の状態を保証することが技術的に困難になるためである。

そのため、強い一貫性を維持するには **単一リージョン書き込み（Single-region write）** を採用する必要がある。書き込みは一つのリージョンで行われ、その結果が他のリージョンへレプリケーションされる。この構成により、すべてのリージョンで常に最新データを読み取ることが保証される。

したがって、このシナリオでは **Strong Consistency + Single-region write** の組み合わせが最も適切な構成となる。

---

## 6. 他の選択肢が誤りな理由

束縛された古さ（Bounded Staleness）と複数リージョン書き込みの組み合わせは、高い一貫性と低遅延を両立できる構成ではあるが、読み取り結果が一定時間または一定更新数だけ古くなる可能性がある。そのため、「すべてのリージョンで強い一貫性」という要件を満たさない。

セッション一貫性とマルチリージョン書き込みの組み合わせは、グローバル分散アプリケーションで最も一般的な構成の一つであり、低遅延と高可用性を提供する。しかしこのモデルでは、他のクライアントからの読み取りが最新データである保証はなく、強い一貫性要件を満たさない。

強い一貫性と複数リージョン書き込みを組み合わせる選択肢は一見理想的に見えるが、Cosmos DB の設計上この組み合わせはサポートされていない。複数リージョン書き込みを有効にすると、強い一貫性を保証するためのグローバル同期が実現できないためである。

---

## 7. 最終回答

**A. 強い一貫性、単一地域が記述する**

---

## 8. まとめ

Azure Cosmos DB の設計では、一貫性・可用性・遅延のバランスを理解することが重要である。Strong Consistency は最も厳密な整合性モデルであり、どのリージョンから読み取っても常に最新データを保証する。しかしこの保証を実現するためには、書き込みリージョンを単一に制限する必要がある。

そのため、グローバル分散アプリケーションで **「すべてのリージョンで強い一貫性」を要求する場合は、Strong Consistency と Single-region write の組み合わせを使用する**ことになる。Cosmos DB の試験問題では、この「強い一貫性とマルチリージョン書き込みは両立しない」という設計原則を理解しているかが重要なポイントとなる。
