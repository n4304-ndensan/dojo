# Azure Database for MySQL Flexible Server のコンピューティング層選択

## 1. 背景（シナリオ）

ある企業では、オンプレミス環境で稼働している **MySQL データベース**を Azure に移行する計画があります。移行先として選択されているのは **Azure Database for MySQL Flexible Server** です。

このシステムは企業の重要なアプリケーションで利用されるため、単にクラウドに移行するだけではなく、**高可用性**を確保する必要があります。特に、データセンター障害が発生した場合でも、データベースサービスが継続して利用可能であることが求められています。

そのため、MySQL Flexible Server のコンピューティング層を選択する際には、可用性、パフォーマンス、コストのバランスを考慮する必要があります。

## 2. 要件整理

この問題のポイントは、移行先のデータベースが **データセンター障害時でも利用可能であること**です。

シナリオから読み取れる要件を整理すると次の通りです。

まず、データセンター障害に対応する高可用性が必要です。次に、MySQL Flexible Server を利用するため、Azure が管理するマネージドサービスを活用することが前提になります。

また、特別にメモリ集約型や高性能ワークロードが求められているわけではありません。そのため、バランスの取れた構成が適しています。

これらをまとめると、次の要件になります。

高可用性のサポート  
バランスの取れたパフォーマンス  
コスト効率  
Azure MySQL Flexible Server の利用  

## 3. 技術の基本概念

Azure Database for MySQL Flexible Server には、ワークロードに応じて複数のコンピューティング層が用意されています。

それぞれの層は、CPU、メモリ、I/O パフォーマンスのバランスが異なります。

主なコンピューティング層には次のものがあります。

まず **Burstable（バースト可能）** は、低コストで小規模なワークロード向けの構成です。CPU 使用量が低いワークロードに適しています。

次に **General Purpose（汎用）** は、CPU・メモリ・I/O のバランスが取れた標準的な構成です。ほとんどのアプリケーションワークロードに対応します。

さらに **Memory Optimized（メモリ最適化）** は、インメモリ処理や分析処理など、メモリを大量に使用するワークロード向けです。

## 4. アーキテクチャまたは設計のポイント

MySQL Flexible Server の高可用性は、**ゾーン冗長構成（Zone-redundant HA）** を利用することで実現できます。

この構成では、プライマリサーバーとスタンバイサーバーが異なる可用性ゾーンに配置されます。これにより、1つのデータセンターが障害を起こした場合でも、別のゾーンのサーバーが自動的に引き継ぎます。

この高可用性構成は、主に **General Purpose 層**で利用される一般的な構成です。

この層は、ほとんどの本番ワークロードに対応できるため、Azure のベストプラクティスでも標準的な選択肢になっています。

## 5. 設計判断（なぜこの構成になるか）

General Purpose 層は、性能とコストのバランスが取れた構成であり、多くのアプリケーションに適しています。

この層は以下の理由で最適です。

まず、可用性ゾーンを利用した高可用性構成に対応しています。次に、CPU とメモリのバランスが取れているため、一般的なデータベースワークロードに適しています。

また、Burstable 層のようにパフォーマンスが不安定になる可能性もなく、Memory Optimized 層ほど高コストでもありません。

そのため、特別なワークロード要件がない場合は General Purpose が最も合理的な選択になります。

## 6. 他の選択肢が誤りな理由

### B メモリ最適化

メモリ最適化層は、インメモリ処理やリアルタイム分析など、大量のメモリを必要とするワークロード向けです。通常のデータベース移行には過剰な構成となる可能性があります。

### C バースタブル

Burstable 層はコストは低いですが、CPU クレジットモデルに依存するため、安定したパフォーマンスを必要とする本番環境には適していません。

### D 高性能

この選択肢は MySQL Flexible Server の標準コンピューティング層ではなく、問題の前提に適合しません。

## 7. 最終回答

A. **一般目的（General Purpose）**

## 8. まとめ

Azure Database for MySQL Flexible Server では、General Purpose コンピューティング層が最も一般的な本番環境向け構成です。

この層は CPU・メモリ・ストレージのバランスが取れており、ゾーン冗長構成による高可用性を実現できます。

試験では **「Flexible Server」「高可用性」「データセンター障害」「一般的ワークロード」** というキーワードが出た場合、**General Purpose** が正解になるケースが多いです。
