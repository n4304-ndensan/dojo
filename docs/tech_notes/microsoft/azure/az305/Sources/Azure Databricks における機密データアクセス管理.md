# Azure Databricks における機密データアクセス管理

## 1. 背景（シナリオ）
Azure Data Lake Storage Gen2 に保存された機密データを処理する財務分析アプリケーションに Azure Databricks を導入する必要があります。アプリケーションは厳格なアクセス制御が求められ、特定のディレクトリに特定の役割のみがアクセスできるようにし、すべてのデータアクセスがコンプライアンス目的で記録される必要があります。

## 2. 要件整理
- **厳密なアクセス制御**: 特定ディレクトリへのアクセスを特定ロールに限定
- **コンプライアンスのためのデータアクセスログ**: 誰がいつどのデータにアクセスしたか追跡可能
- **Azureサービスとの統合**: ユーザーID管理やポリシー管理を統合

## 3. 技術の基本概念
**ロールベースアクセス制御 (RBAC)** は、Azure サービスにおけるアクセス制御の基本メカニズムで、ユーザーやグループに対してきめ細かなアクセス権を付与できます。Azure Databricks と Azure Data Lake Storage Gen2 の組み合わせにより、以下が可能です。

- ディレクトリやファイル単位でアクセス権を制御
- Azure Active Directory と統合してユーザーやグループを管理
- データアクセスイベントをログとして記録・監査

## 4. アーキテクチャまたは設計のポイント
- RBAC を利用することで、最小特権の原則に基づきユーザーに必要最小限のアクセス権を付与
- Azure Monitor や Azure Log Analytics と統合することでアクセスログを保存し、コンプライアンス監査に対応
- Databricks ワークスペース、クラスター、ノートブック単位でロール割り当てを行い、細かく制御可能

## 5. 設計判断（なぜこの構成になるか）
- 機密財務データへのアクセスをきめ細かく制御可能
- 誰がいつデータにアクセスしたかのログを保持し、規制要件に準拠
- Azure AD との統合によりユーザー管理の効率化
- クラスタやノートブック単位のスコープで柔軟なアクセス制御を実現

## 6. 他の選択肢が誤りな理由
- **セキュアクラスター接続**: 接続の安全性を確保するが、アクセス制御や監査ログは提供されない
- **クラスタスコープのinitスクリプト**: クラスタ構成やライブラリ導入用であり、アクセス制御には対応していない
- **プライベートリンク連携**: ネットワーク的セキュリティを向上させるが、役割ベースのアクセス制御や監査ログは提供しない

## 7. 最終回答
**C. ロールベースアクセス制御(RBAC)**

## 8. まとめ
Azure Databricks における RBAC の導入により、機密データへのアクセスを許可されたユーザーのみに制限し、最小特権の原則を適用できます。また、アクセスログを記録することで、財務データのコンプライアンス要件にも対応可能です。これにより、安全で規制準拠したデータ分析環境を構築できます。
