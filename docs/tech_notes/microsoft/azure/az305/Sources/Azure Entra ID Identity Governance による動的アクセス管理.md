# Azure Entra ID Identity Governance による動的アクセス管理

## 1. 背景（シナリオ）

ある企業の Azure サブスクリプションには、Azure Active Directory（現在は **Azure Entra ID**）と統合された **10個のWebアプリケーション**があります。これらのアプリケーションは複数のプロジェクトで使用されており、ユーザーはプロジェクト間を頻繁に移動します。

そのため、ユーザーの所属プロジェクトに応じて、アクセス可能な Web アプリを変更する必要があります。つまり、プロジェクトが変わればアクセス権も変更される **動的なアクセス管理**が求められています。

このような状況では、単純な認証強化ではなく、**ユーザーの役割や所属に基づいたアクセス管理（Identity Governance）**が重要になります。

## 2. 要件整理

問題文から読み取れる要件を整理すると、アクセス管理の目的が明確になります。単なるログイン制御ではなく、**ユーザーの所属プロジェクトに基づいてアプリケーションアクセスを管理する仕組み**が必要です。

本シナリオで求められる要件は次の通りです。

まず、ユーザーは複数チームにまたがって作業し、プロジェクト間を頻繁に移動します。このため、アクセス権を固定的に設定するのではなく、状況に応じて変更できる仕組みが必要になります。

その要件を整理すると次のようになります。

・ユーザーの所属プロジェクトに応じたアクセス管理  
・Webアプリケーションへのアクセス制御  
・ユーザーの役割変更に応じたアクセス更新  
・アクセスの管理・監査の仕組み  

このような要件は、Azure Entra ID の **Identity Governance 機能**によって実現できます。

## 3. 技術の基本概念

Azure Entra ID には、アイデンティティとアクセス管理を実現するための複数の機能があります。その中でも **Identity Governance** は、ユーザーアクセスのライフサイクルを管理するための仕組みです。

Identity Governance には主に次の機能があります。

まず、ユーザーに対して複数のリソースへのアクセスをまとめて付与できる **Entitlement Management（エンタイトルメント管理）**があります。これは、アプリケーションやグループ、SharePoint などをまとめて **アクセスパッケージ**として提供できる仕組みです。

次に、ユーザーが現在も適切なアクセス権を持っているかを定期的に確認する **Access Reviews（アクセスレビュー）**があります。これにより、不要なアクセス権が残り続けることを防ぎます。

さらに、ユーザーのライフサイクルに応じてアクセスを管理する **Lifecycle Management** の機能も提供されています。

## 4. アーキテクチャまたは設計のポイント

この問題のように、ユーザーがプロジェクト間を頻繁に移動する環境では、手動でアクセス権を管理するのは非常に非効率です。そのため、Identity Governance を利用してアクセス管理を自動化します。

具体的には、次のような設計を行います。

まず、プロジェクトごとに必要な Web アプリケーションをまとめた **アクセスパッケージ**を作成します。そして、ユーザーが特定のプロジェクトに所属した場合、そのアクセスパッケージを付与します。

この設計では次のような管理が可能になります。

・プロジェクト単位のアクセスパッケージ  
・自動プロビジョニング  
・アクセスレビューによる定期確認  
・監査ログによる追跡  

この仕組みにより、ユーザーがプロジェクトを変更した場合でも、アクセス権を簡単に変更できます。

## 5. 設計判断（なぜこの構成になるか）

Identity Governance が最適な理由は、**動的なアクセス管理を実現できる点**にあります。

特に以下の機能が重要です。

まず、Entitlement Management によって、複数のアプリケーションへのアクセスをまとめて管理できます。これにより、プロジェクト単位のアクセス管理が容易になります。

次に、Access Reviews を利用することで、ユーザーが現在もそのアクセス権を必要としているかを定期的に確認できます。これにより、不要なアクセス権を削除できます。

さらに、監査ログやレポート機能によって、コンプライアンスやセキュリティ監査にも対応できます。

## 6. 他の選択肢が誤りな理由

問題では、他にもいくつかの Azure セキュリティサービスが提示されていますが、それぞれ目的が異なります。

### A Azure AD 条件付きアクセス

条件付きアクセスは、ログイン時のセキュリティポリシーを適用する機能です。例えば、次のような条件を設定できます。

・特定の場所からのアクセスのみ許可  
・多要素認証（MFA）の強制  
・デバイス準拠チェック  

しかし、この機能は **誰がどのアプリにアクセスできるかを管理するものではありません。**

### B Azure AD Identity Protection

Identity Protection は、ユーザーアカウントのリスクを検出するセキュリティ機能です。具体的には次のような用途で使われます。

・不審なログインの検出  
・漏洩した資格情報の検出  
・リスクベース認証

つまり、**アクセス管理ではなくリスク検出の機能です。**

### D Microsoft Defender for Identity

Microsoft Defender for Identity は、主にオンプレミス Active Directory に対する脅威検出を目的としたセキュリティソリューションです。

このサービスの主な目的は次の通りです。

・攻撃の検出  
・不正なユーザー行動の分析  
・インサイダー脅威の検出

したがって、プロジェクト単位のアクセス管理には適していません。

## 7. 最終回答

C. **Azure AD Identity Governance**

## 8. まとめ

Azure Entra ID Identity Governance は、ユーザーアクセスのライフサイクルを管理するための重要な機能です。特に、ユーザーが頻繁に役割やプロジェクトを変更する環境では非常に有効です。

Identity Governance の重要なポイントを整理すると次の通りです。

まず、アクセスパッケージによって複数のリソースをまとめて管理できます。次に、アクセスレビューによって不要な権限を定期的に確認できます。さらに、ライフサイクル管理によってアクセスの付与と削除を自動化できます。

試験では、**「ユーザーの役割変更」「プロジェクト単位のアクセス管理」「アクセスレビュー」**といったキーワードが出てきた場合、Azure Entra ID Identity Governance が正解になるケースが多いです。
