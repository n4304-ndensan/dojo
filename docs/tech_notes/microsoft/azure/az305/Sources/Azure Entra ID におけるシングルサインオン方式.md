# Azure Entra ID におけるシングルサインオン方式

Azure Entra ID（旧 Azure Active Directory）では、ユーザーが一度の認証で複数のアプリケーションにアクセスできる **シングルサインオン（SSO）** を提供しています。SSO を実現する方法はいくつかあり、アプリケーションの種類や変更可否、既存の認証方式によって適切な方式を選択する必要があります。

特に企業環境では、レガシーアプリケーション、モダンWebアプリケーション、SaaSアプリケーションなどが混在するため、それぞれに適したSSO方式を理解することが重要です。

Azure Entra ID で利用される代表的なSSO方式には次のものがあります。

- SAMLベースのSSO  
- OpenID Connect  
- パスワードベースSSO  
- リンクベースSSO  
- ヘッダーベースSSO  

これらは用途やアプリケーション構造によって使い分けられます。

---

# SAMLベースのSSO

SAML（Security Assertion Markup Language）は、企業向けのWebアプリケーションで広く利用されている認証フェデレーションプロトコルです。Azure Entra ID が **Identity Provider（IdP）** としてユーザーを認証し、その結果を **SAML Assertion** と呼ばれるトークンとしてアプリケーションに送信します。

アプリケーションはこのアサーションを検証し、ユーザーをログイン状態として扱います。重要な点は、ユーザーの認証は Azure Entra ID が行い、アプリケーションはその認証結果を信頼するという点です。

この方式は、既存のレガシーWebアプリケーションとSSOを統合する際によく利用されます。特に、アプリケーションのコードや認証方式を変更できない場合でも、SAMLフェデレーションを利用することでSSOを実現できる場合があります。

企業環境では、多くのオンプレミスアプリケーションや古いWebシステムがSAMLに対応しているため、SAMLは現在でも広く使用されています。

---

# OpenID Connect

OpenID Connect は OAuth 2.0 を基盤としたモダンな認証プロトコルです。現在のクラウドアプリケーションやモバイルアプリケーションでは、この方式が最も一般的に利用されています。

OpenID Connect では、Azure Entra ID がユーザーを認証した後に **IDトークン（JWT）** を発行します。アプリケーションはこのトークンを検証することでユーザーを認証します。

この方式は、APIやSPA（Single Page Application）などのモダンアプリケーションに適しています。しかし、アプリケーション側でトークン処理を実装する必要があるため、既存アプリケーションのコードを変更できない場合には導入が難しい場合があります。

そのため、新規開発アプリケーションでは OpenID Connect が推奨される一方、レガシーアプリケーションでは SAML が選択されることが多くなります。

---

# パスワードベースSSO

パスワードベースSSOは、Azure Entra ID がユーザーの資格情報を安全に保存し、ユーザーの代わりにログインフォームへ資格情報を入力することでSSOを実現する方式です。

この方式では、ユーザーは Azure Entra ID にログインした後、Azure ポータルやアプリランチャーから対象アプリを起動します。Azure Entra ID は保存された資格情報を使ってログインフォームに自動入力し、ユーザーはアプリケーションにログインします。

この方法は、SAMLやOpenID Connectなどのフェデレーションプロトコルに対応していないアプリケーションで利用されることがあります。ただし、資格情報を保存する仕組みであるため、セキュリティ面や管理面の観点からはフェデレーション方式ほど推奨される方法ではありません。

---

# リンクベースSSO

リンクベースSSOは、Azure Entra ID のアプリランチャーからアプリケーションへのリンクを提供する方式です。ユーザーはポータルからリンクをクリックすることでアプリケーションにアクセスできます。

この方式では Azure Entra ID が実際の認証を行うわけではなく、単にアクセスリンクを提供するだけです。そのため厳密な意味でのSSOではなく、アプリケーションへのアクセスをまとめて管理するための方法と考えることができます。

主にAzureポータルから外部アプリケーションへのアクセスを簡単にする目的で利用されます。

---

# ヘッダーベースSSO

ヘッダーベースSSOは、リバースプロキシやアプリケーションゲートウェイなどの中間システムがユーザー認証を行い、その結果をHTTPヘッダーとしてアプリケーションへ渡す方式です。

この方式では、ユーザーはまずプロキシで認証されます。その後、プロキシがユーザー情報をHTTPヘッダーに追加してアプリケーションへリクエストを送信します。アプリケーションはこのヘッダー情報を信頼してユーザーを認証します。

この方式は古いWebアプリケーションや特定のネットワーク環境で利用されることがありますが、Azure Entra ID の標準的なSSO方式ではありません。多くの場合、追加のプロキシサーバーやゲートウェイが必要になります。

---

# SSO方式の使い分け

Azure Entra ID でSSO方式を選択する際には、アプリケーションの種類や変更可能性が重要になります。

レガシーWebアプリケーションで認証方式を変更できない場合は **SAMLベースSSO** がよく利用されます。  
新規のクラウドアプリケーションやモダンWebアプリケーションでは **OpenID Connect** が推奨されます。

フェデレーションプロトコルに対応していないアプリケーションでは **パスワードベースSSO** が利用される場合があります。  
また、単純にアプリケーションリンクをポータルから提供する場合は **リンクベースSSO** が使用されます。

このように、Azure Entra ID ではアプリケーションの種類や技術要件に応じて複数のSSO方式を使い分けることができます。
