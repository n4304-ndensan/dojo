# Azure Entra ID 外部ユーザーアクセス（試験・実務ミニ整理）

Azure AD（現在は **Azure Entra ID**）では、他の会社のユーザーが自社のアプリやリソースにアクセスするための仕組みがいくつか用意されています。Azureの認証・ID管理に関する試験では、外部ユーザーのアクセス方法として次の仕組みの違いを理解していることが重要です。

- マルチテナントアプリ  
- Azure AD B2B  
- Entitlement Management  
- フェデレーション（SAML / 外部IdP）

これらを理解するうえで、まず重要になる判断基準があります。それは **外部ユーザーを自分のテナントにユーザーとして作成するかどうか**です。

外部ユーザーを自分のテナントに作成する場合は **Azure AD B2B** が利用されます。  
一方で、外部ユーザーを自分のテナントに作成せず、そのユーザーが所属するテナントの認証をそのまま利用する場合は **マルチテナントアプリ**が利用されます。

この違いを理解しておくと、Azureの試験問題の多くを整理して判断できるようになります。

---

# マルチテナントアプリ

マルチテナントアプリとは、複数の Azure Entra ID テナントのユーザーが利用できるアプリケーションのことです。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#App Registration]]
通常、Azure のアプリ登録は **シングルテナント**として作成されます。この設定では、そのテナントに属するユーザーのみがアプリにログインできます。そのため、他の会社の Azure AD ユーザーはログインできません。

この制限を解除する方法が **マルチテナント設定**です。  
アプリ登録の **Supported account types** を次の設定に変更します。

```

Accounts in any organizational directory

```

この設定に変更すると、アプリは **マルチテナントアプリ**となり、他の Azure AD テナントのユーザーもログインできるようになります。

この方式では、ユーザーは **自分のテナントで認証を行います**。  
例えば次のような流れになります。

Fabrikamユーザー  
↓  
Fabrikam Azure ADで認証  
↓  
アクセストークン発行  
↓  
Contosoアプリへアクセス

この場合、Contoso テナントのディレクトリにはユーザーアカウントは作成されません。ユーザーは自分の会社の Azure AD を使ってログインし、その認証結果をアプリが受け取ります。

この方式のメリットは、外部ユーザーのアカウント管理が不要になることです。特に SaaS アプリや、複数企業が利用するアプリケーションではよく利用される方式です。

一方で、アプリケーション側では **複数テナントから発行されるトークンを正しく検証する必要があります**。そのため、シングルテナントアプリよりも認証処理の設計を考慮する必要があります。

試験では、次のような条件が提示された場合にマルチテナントが正解になることが多くなります。

- 外部企業ユーザーがログインする  
- 自分の Azure AD アカウントを使う  
- ゲストユーザーを作成しない

# Azure AD B2B

Azure AD B2B は、外部ユーザーを **ゲストユーザーとして自分のテナントに招待する仕組み**です。

例えば Contoso が Fabrikam のユーザーを招待すると、Contoso テナントのディレクトリに **Guest ユーザー**としてアカウントが作成されます。

ログインの流れは次のようになります。

Fabrikamユーザー招待  
↓  
Contosoテナントに Guest ユーザー作成  
↓  
ユーザーがログイン  
↓  
Contosoリソースへアクセス

この方式では、ユーザーは外部ユーザーですが、Contoso テナント内では **ユーザーアカウントとして管理されます**。

この仕組みのメリットは、Azure AD の管理機能をそのまま利用できることです。例えば次のような管理が可能になります。

- グループ割り当て  
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Role-Based Access Control (RBAC)]]
- RBAC（ロールベースアクセス制御）  
- 条件付きアクセス  
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Multi-Factor Authentication (MFA)]]
- MFA  

つまり、内部ユーザーとほぼ同じ管理方法で外部ユーザーを制御できます。

デメリットとしては、ゲストユーザーの管理が必要になる点です。外部ユーザーが増えると、ユーザー管理やアクセス管理の運用が必要になります。

主な利用ケースは次の通りです。

- 外部パートナーとのコラボレーション  
- ベンダーアクセス  
- SharePoint や Teams の外部共有  

試験では次のような条件が提示された場合、Azure AD B2B が正解になることが多くなります。

- 外部ユーザーを招待する  
- ゲストアカウント  
- 外部パートナーアクセス

# Entitlement Management

Entitlement Management は、Azure Entra ID の **アクセスライフサイクル管理機能**です。

これは認証方式ではなく、ユーザーのアクセス権限を管理するための仕組みです。ユーザーがどのリソースにアクセスできるかを管理し、アクセス申請や承認プロセスを自動化することができます。

Entitlement Management では **アクセスパッケージ**という仕組みを使用します。アクセスパッケージは、複数のリソースをまとめて付与する単位です。

例えば次のようなものをまとめて管理できます。

- Azure AD グループ  
- アプリケーション  
- SharePoint サイト  

ユーザーはアクセスパッケージを申請し、承認されると必要なリソースへのアクセス権が付与されます。また、一定期間が経過するとアクセス権を自動的に削除することも可能です。

この機能のメリットは、外部ユーザーやパートナーのアクセス管理を自動化できることです。アクセス申請、承認、期限管理を統合的に管理できるため、運用負荷を減らすことができます。

ただし、Entitlement Management は **認証方式ではありません**。そのため、B2B やマルチテナントの代わりになるものではありません。

主な利用ケースは次の通りです。

- 外部パートナーのアクセス管理  
- アクセス申請と承認の管理  
- アクセス期限の自動管理  

試験では次のような条件が出た場合、Entitlement Management が選択肢になることがあります。

- アクセス申請  
- 承認フロー  
- 自動期限

# フェデレーション（SAML / 外部IdP）

フェデレーションとは、外部の ID プロバイダー（IdP）を利用して認証を行う仕組みです。

Azure AD 以外の認証システムと連携する場合に使用されます。例えば次のような IdP が利用されることがあります。

- Okta  
- ADFS  
- Ping Identity  
- その他の企業 IdP  

この方式では、アプリケーションは外部の IdP を信頼し、SAML や OpenID Connect を利用して認証結果を受け取ります。

認証の流れは次のようになります。

ユーザー  
↓  
外部 IdP で認証  
↓  
トークン発行  
↓  
アプリへアクセス

この方式のメリットは、既存の認証基盤を利用してシングルサインオンを実現できることです。企業がすでに外部 IdP を導入している場合には有効な方法です。

一方で、設定や運用が複雑になる場合があります。そのため、Azure AD 同士の環境では、フェデレーションよりも **マルチテナントアプリや B2B の方がシンプルな構成になることが多い**です。

試験では次のような条件が出た場合にフェデレーションが選択肢になることがあります。

- 外部 IdP  
- SAML  
- ADFS

# 試験での判断ルール

まず次のポイントを確認します。

外部ユーザーを **自分のテナントにユーザーとして作るか？**

YES  
→ **Azure AD B2B**

NO  
→ **マルチテナントアプリ**

次に、アクセス管理の仕組みが問われているかを確認します。

アクセス申請や承認フローを管理する必要がある場合は **Entitlement Management** が利用されます。

また、Azure AD 以外の認証システムを利用する必要がある場合は **フェデレーション**が利用されます。
