# Azure Entra ID 条件付きアクセスによる場所・リスクベースのアクセス制御

## 1. 背景（シナリオ）

ある組織では、Azure Active Directory（現在は **Azure Entra ID**）によって保護されたリソースに対するリモートアクセスを管理するセキュリティポリシーを設計する必要があります。

この組織では機密データへのアクセスを厳格に管理する必要があり、ユーザーがどこからアクセスしているかを考慮したアクセス制御を導入することが求められています。特に、信頼できるネットワークからのアクセスのみを許可し、それ以外の場所からアクセスする場合には追加の認証を要求する必要があります。

さらに、このセキュリティポリシーは固定的なルールだけではなく、ユーザーの行動やリスク評価に基づいて動的に適応できる必要があります。これは、ゼロトラストセキュリティモデルにおいて重要な要件となります。

## 2. 要件整理

このシナリオでは、単なるネットワーク制御ではなく、**ユーザーのアイデンティティとリスク評価を組み合わせたアクセス制御**が必要になります。

問題文から読み取れる要件は次の通りです。

まず、特定の信頼できる場所からのみ機密データにアクセスできるようにする必要があります。次に、組織の内部ネットワーク外からアクセスするユーザーには多要素認証（MFA）を要求する必要があります。

さらに重要な点として、ポリシーはユーザーの行動やリスクに応じて動的に適応する必要があります。

これらの条件を整理すると次の要件になります。

・信頼できる場所に基づくアクセス制御  
・内部ネットワーク外からのアクセス時に MFA を要求  
・ユーザーの行動やリスクに基づく動的ポリシー  
・アイデンティティベースのアクセス制御  

これらを実現できる機能が **Azure Entra ID の条件付きアクセス（Conditional Access）**です。

## 3. 技術の基本概念

Azure Entra ID の条件付きアクセスは、ユーザーのサインイン状況に応じてアクセス制御ポリシーを適用する仕組みです。

この機能では、ユーザーがリソースにアクセスする際に複数の条件を評価し、その結果に応じてアクセスを許可または制限します。

条件付きアクセスで利用できる代表的な条件には次のようなものがあります。

まず、ユーザーの **場所（Location）** に基づくアクセス制御があります。管理者は信頼できる IP 範囲を登録し、その場所からのアクセスを特別扱いできます。

次に、**サインインリスク（Sign-in Risk）** や **ユーザーリスク（User Risk）** といったリスクベースの条件があります。これらは Azure Entra ID Identity Protection と連携してリスク評価を行います。

さらに、ポリシーでは MFA を要求したり、アクセスをブロックしたりするなどの制御を実施できます。

## 4. アーキテクチャまたは設計のポイント

このシナリオでは、信頼できる場所と信頼できない場所を区別する設計が重要になります。

まず、組織の内部ネットワークの IP 範囲を **Trusted Locations（信頼された場所）** として定義します。これにより、内部ネットワークからのアクセスは通常の認証のみで許可できます。

次に、内部ネットワーク以外の場所からアクセスするユーザーに対して MFA を要求する条件付きアクセスポリシーを設定します。

さらに、Identity Protection と連携することで、サインインのリスクレベルを評価し、必要に応じて追加の認証やアクセス制限を適用できます。

この設計によって、セキュリティとユーザー利便性を両立することができます。

## 5. 設計判断（なぜこの構成になるか）

条件付きアクセスが最適な理由は、**複数の条件を組み合わせた動的なアクセス制御**を実現できるためです。

この機能を利用することで、ユーザーの場所、デバイス、リスクレベルなどを組み合わせてアクセス制御を行うことができます。

また、リスクベースの条件を利用することで、ユーザーの異常な行動や不審なサインインが検出された場合に追加の認証を要求することができます。

これにより、固定的なアクセスルールではなく、リアルタイムのリスク評価に基づいた柔軟なセキュリティポリシーを実現できます。

## 6. 他の選択肢が誤りな理由

この問題ではいくつかの Azure セキュリティ機能が提示されていますが、それぞれ目的が異なります。

### B ネットワークセキュリティグループ（NSG）

NSG は仮想ネットワークレベルでトラフィックを制御するサービスです。IP アドレスに基づく通信制御は可能ですが、ユーザーのアイデンティティや MFA を利用した認証制御はできません。

### C Identity Protection ポリシー

Identity Protection はユーザーのリスクやサインインリスクを検出する機能です。しかし、この機能単体ではアクセス制御を直接実施することはできません。

通常は条件付きアクセスと組み合わせて使用されます。

### D Azure Firewall

Azure Firewall はネットワークトラフィックを制御するセキュリティサービスです。アプリケーションレベルのフィルタリングは可能ですが、ユーザー認証や MFA といったアイデンティティベースのアクセス制御には対応していません。

## 7. 最終回答

A. **立地条件およびリスクベースの条件付きアクセスポリシー**

## 8. まとめ

Azure Entra ID の条件付きアクセスは、ユーザーのアイデンティティ、場所、リスク評価などを組み合わせた高度なアクセス制御を実現する機能です。

この機能を利用することで、信頼できる場所からのアクセスを許可し、信頼できない場所からのアクセスには MFA を要求することができます。

さらに、Identity Protection と連携することで、リスクベースのアクセス制御を実現できます。

試験では **「場所ベースのアクセス制御」「MFA」「リスクベース認証」「ゼロトラスト」** といったキーワードが出てきた場合、Azure Entra ID の条件付きアクセスが正解になるケースが非常に多いです。
