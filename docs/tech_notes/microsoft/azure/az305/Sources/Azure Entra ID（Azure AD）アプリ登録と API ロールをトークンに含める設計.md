# Azure Entra ID（Azure AD）アプリ登録と API ロールをトークンに含める設計

## 1. 背景（シナリオ）

Azure Entra ID（旧 Azure Active Directory）では、アプリケーション同士が安全に通信するための仕組みとして **アプリ登録（App Registration）** が使用される。特に API を公開するアプリケーションと、その API を呼び出すクライアントアプリケーションが存在する場合、API 側で定義された権限やロールをトークンに含めることでアクセス制御を行うことができる。

今回のシナリオでは、2つのアプリケーション登録が存在している。1つ目のアプリケーションは **App1** であり、これは API として動作するアプリケーションである。App1 は API を公開しており、その API に対して **Writer というロール**を定義している。もう1つのアプリケーションは **App2** であり、これは App1 の API を呼び出すクライアントアプリケーションである。

このとき App2 が App1 を呼び出す際に取得するアクセストークンには、App1 で定義された **Writer ロールのクレーム（role claim）** を含める必要がある。つまり App2 のトークンの中に、呼び出す API に対してどのロールを持っているかという情報を埋め込む必要がある。この問題は、その設定を Azure ポータルのどのブレードで行うべきかを理解しているかを問う内容である。

---

## 2. 要件整理

問題の要件は次のように整理できる。まず App1 は API を公開するアプリケーションであり、その API に対して Writer というロールを提供している。次に App2 はその API を呼び出すクライアントアプリケーションであり、App1 の API に対するアクセス権限を持つ必要がある。

さらに重要なのは、App2 が取得するアクセストークンの中に Writer ロールの情報を含める必要があるという点である。Azure Entra ID のアクセストークンには、アプリケーションやユーザーが持つ権限やロールが **role クレーム**として格納される。したがって、API 側でロールを公開し、それをクライアントアプリが利用できるように設定する必要がある。

この要件を満たすためには、API 側でロールを定義し公開することと、クライアント側でその API に対するアクセス権を要求することの両方が必要になる。

---

## 3. Azure Entra ID における API 公開の仕組み

Azure Entra ID では、アプリケーションを API として公開する場合、**Expose an API（API を公開）** というブレードを使用する。この設定では、API の識別子（Application ID URI）を定義し、さらに API が提供する権限やロールを設定することができる。ここで定義されたロールは、API を利用するクライアントアプリケーションのトークンに含めることが可能になる。

例えば App1 が Writer というロールを公開している場合、そのロールは API を呼び出すクライアントに対して付与できる権限として登録される。これにより、App1 の API を利用するアプリケーションは、特定のロールを持つトークンを取得して API を呼び出すことができるようになる。

一方で、クライアントアプリケーション側では、呼び出したい API に対するアクセス権を要求する必要がある。この設定は **API permissions（API 権限）** のブレードで行う。ここで App2 は App1 の API を選択し、必要な権限やロールを要求する。これにより、App2 が取得するアクセストークンには App1 の API に対するロール情報が含まれるようになる。

---

## 4. 技術的な仕組み

Azure Entra ID では、クライアントアプリケーションが API を呼び出す場合、OAuth 2.0 または OpenID Connect のフローを使用してアクセストークンを取得する。このトークンは JWT（JSON Web Token）形式であり、トークン内部にはクレームと呼ばれる属性情報が含まれる。API ロールが設定されている場合、その情報は **roles クレーム**としてトークンに含まれる。

今回の構成では、App1 が API としてロールを公開し、App2 がその API に対する権限を要求することで、App2 が取得するアクセストークンに Writer ロールが含まれるようになる。この流れを簡単な構成図で表すと次のようになる。

```id="app-role-flow"
App2 (Client App)
      │
      │ OAuth Token Request
      ▼
Azure Entra ID
      │
      │ Access Token (roles: Writer)
      ▼
App1 (API)
```

このトークンには roles クレームとして Writer が含まれ、App1 の API はその情報を利用してアクセス制御を行うことができる。

---

## 5. 設計判断

この問題で重要なのは、API ロールを定義する場所と、それをクライアントアプリケーションに付与する場所を正しく理解することである。API のロールは API を提供するアプリケーション側で定義する必要がある。そのため App1 では **Expose an API（API を公開）** のブレードで API のロールを定義することになる。

一方で、App2 は API を呼び出すクライアントであるため、App1 の API を利用するための権限を設定する必要がある。この設定は **API permissions（API 権限）** のブレードで行う。ここで App1 の API を選択し、必要なロールを付与することで、App2 のトークンにそのロールが含まれるようになる。

この2つの設定を組み合わせることで、App2 が App1 の API を呼び出す際のトークンに Writer ロールを含めることができる。

---

## 6. 他の選択肢が誤りな理由

選択肢 A は、App1 でアプリの役割を設定し、App2 でトークン構成を変更するという内容である。確かに API 側でロールを定義することは重要だが、トークン構成を変更するだけではクライアントアプリに API のアクセス権を付与することにはならない。そのため、この選択肢では App2 が API を呼び出すための正しい権限付与が行われない。

選択肢 B は、App1 の API 権限と App2 のアプリロールを設定するという内容であるが、API ロールは API を公開するアプリケーションで定義する必要があり、クライアントアプリ側でロールを定義することは意味を持たない。そのため、この構成では App1 の Writer ロールが正しく設定されない。

選択肢 C は、App1 のトークン設定と App2 の API 権限を設定するという内容である。しかしトークン設定はクレームの追加などに使用されるものであり、API ロールを公開するための設定ではない。したがって、この選択肢も要件を満たさない。

---

## 7. 最終回答

**D.  
App1：API を公開する（Expose an API）  
App2：API 権限（API permissions）**

---

## 8. まとめ

Azure Entra ID で API を公開し、クライアントアプリケーションのトークンにロールクレームを含める場合、まず API 側のアプリケーションでロールを定義して公開する必要がある。この設定は「Expose an API」ブレードで行う。その後、クライアントアプリケーション側で API に対するアクセス権を要求する必要があり、この設定は「API permissions」ブレードで行う。

この仕組みによって、クライアントアプリケーションが取得するアクセストークンの中に API のロール情報が含まれ、API 側でロールベースのアクセス制御を行うことが可能になる。Azure のアプリ登録に関する問題では、この **API 公開と API 権限の関係**を理解しておくことが重要である。
