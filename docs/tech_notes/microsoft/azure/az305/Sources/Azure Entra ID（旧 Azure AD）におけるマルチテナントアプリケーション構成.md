# Azure Entra ID（旧 Azure AD）におけるマルチテナントアプリケーション構成

## 1. 背景（シナリオ）

ある組織では、Azure Active Directory（現在は **Microsoft Entra ID**）を利用して認証を行うアプリケーションを構築しています。このアプリケーションは単一組織だけではなく、複数の外部組織のユーザーが利用することを想定しています。

つまり、このアプリケーションは **マルチテナントアプリケーション**として設計されており、異なるAzure ADテナントに所属するユーザーが、それぞれの組織アカウントを使用して認証できる必要があります。

さらに、このシステムには以下の要件があります。

まず、外部Azure ADテナントのユーザーが **追加アカウント作成なしでログインできること**が求められています。  
また、認証は各組織のセキュリティポリシー（MFA、条件付きアクセスなど）を維持したまま行われる必要があります。

そのため、Azure Entra IDの **アプリ登録設定（App Registration）** において、適切な認証対象を構成する必要があります。

## 2. 要件整理

この問題の重要なポイントを整理すると、次のようになります。

まず、このアプリケーションは単一組織ではなく、**複数の外部Azure ADテナントからのユーザー**が利用する必要があります。

さらに、ユーザーはそれぞれの組織のAzure ADアカウントを使って **シームレスに認証できる必要**があります。

このシナリオで求められる要件は次の通りです。

- 外部Azure ADテナントユーザーの認証
- シームレスなログイン体験
- 既存Azure ADアカウントの利用
- セキュリティポリシーの維持
- 管理負荷の低減

これらの要件を満たすためには、Azure Entra IDの **マルチテナントアプリケーション設定**を使用する必要があります。

## 3. 技術の基本概念

Azure Entra IDでは、アプリケーション登録時に **サポートするアカウントの種類（Supported account types）** を設定できます。

この設定によって、どのユーザーがアプリケーションにサインインできるかが決まります。

ここでは主なアカウントタイプについて理解することが重要です。

まず、**Single Tenant（単一テナント）**があります。  
この設定では、アプリケーションは **自分の組織のAzure ADテナントのユーザーのみ**が利用できます。外部テナントのユーザーはアクセスできません。

次に、**Multi-tenant（マルチテナント）**があります。  
この設定では、**任意のAzure ADテナントのユーザー**がアプリケーションにサインインできます。

つまり、複数の企業や組織のユーザーが同じアプリケーションを利用できるようになります。

この構成では、ユーザーは次のような流れで認証します。

1. ユーザーがアプリケーションにアクセスする  
2. Azure Entra IDのログインページへリダイレクトされる  
3. 自分の組織の認証基盤でログイン  
4. トークンが発行されアプリへアクセス

この仕組みにより、各組織のセキュリティポリシーを維持したままアプリケーションを利用できます。

## 4. アーキテクチャまたは設計のポイント

マルチテナントアプリケーション設計では、**認証対象のディレクトリ範囲**を正しく設定することが重要です。

Azure Entra IDでは、アプリ登録の「サポートされるアカウントの種類」で次の設定を行います。

その際、以下の構成を選択する必要があります。

- 任意の組織ディレクトリ内のアカウント  
- （Any organizational directory – Multitenant）

この設定を行うことで、アプリケーションは **すべてのAzure ADテナントからのユーザーを受け入れることが可能**になります。

また、この構成では次のAzureセキュリティ機能をそのまま利用できます。

- 条件付きアクセス
- 多要素認証（MFA）
- セキュリティポリシー
- デバイス準拠ポリシー

これにより、各組織は自分のセキュリティポリシーを維持したままアプリケーションを利用できます。

## 5. 設計判断（なぜこの構成になるか）

この問題の要件は、**複数のAzure ADテナントからのユーザーがシームレスに認証できること**です。

この要件を満たす唯一の設定は、アプリケーションを **マルチテナントアプリケーションとして構成すること**です。

具体的には、次の設定を使用します。

任意の組織ディレクトリ内のアカウント  
（Azure ADディレクトリ – マルチテナント）

この設定により次の利点が得られます。

まず、ユーザー体験が向上します。  
ユーザーは自分の組織のアカウントを使用して直接ログインできます。

次に、セキュリティが維持されます。  
認証は各組織のAzure ADによって行われるため、条件付きアクセスやMFAなどのポリシーが適用されます。

さらに、運用管理が簡単になります。  
各テナントごとに個別アカウントを管理する必要がなくなります。

## 6. 他の選択肢が誤りな理由

### A. ユーザー同意を必須にする

この設定は **API権限の承認プロセス**に関するものです。

つまり、

- 権限管理
- 同意フロー

に関係する設定であり、**マルチテナントアクセス自体には関係ありません。**

そのため、この問題の要件を満たしません。

### C. パブリッククライアントフローを許可する

この設定は以下のようなアプリケーションで使用されます。

- モバイルアプリ
- デスクトップアプリ
- CLIツール

これは **認証フローの種類の設定**であり、テナントの範囲とは無関係です。

そのため、この問題の解決にはなりません。

### D. Azure AD B2C を使用する

Azure AD B2Cは主に **コンシューマー向けアプリケーション**で使用されます。

例えば次のような用途です。

- ECサイト
- 一般ユーザー向けアプリ
- ソーシャルログイン

今回のシナリオは **企業テナント間の認証（B2B）** なので適切ではありません。

## 7. 最終回答

複数の外部Azure ADテナントからのユーザー認証を実現するための構成は次です。

**任意の組織ディレクトリ（Azure ADディレクトリ – マルチテナント）のアカウントを許可する**

したがって **正解は B** です。

## 8. まとめ

Azure Entra IDでは、アプリケーションの認証範囲を **Supported account types** で設定します。

この設定はアプリケーションの利用対象を決定する重要な構成です。

主なポイントは次の通りです。

まず、Single Tenantは自組織のみ利用可能です。  
次に、Multi-tenantはすべてのAzure ADテナントからのアクセスを許可します。

今回の問題では、

- 外部テナントユーザー
- シームレス認証
- セキュリティ維持

という要件があるため、**マルチテナント設定が必須**になります。

そのため、最適な構成は **「任意の組織ディレクトリ内のアカウント（マルチテナント）」** となります。
