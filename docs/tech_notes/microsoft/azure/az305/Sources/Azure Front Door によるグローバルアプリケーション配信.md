# Azure Front Door によるグローバルアプリケーション配信

## 1. 背景（シナリオ）

ある組織では、複数の Azure リージョンに Azure App Service をデプロイし、グローバルに利用される Web アプリケーションを提供する予定です。ユーザーは世界中からアクセスするため、可用性とパフォーマンスの両方を確保する必要があります。

このようなグローバルアプリケーションでは、単一リージョンの障害がサービス停止につながらないように設計することが重要です。また、急激なトラフィック増加が発生した場合でも、アプリケーションのパフォーマンスを維持できる仕組みが必要になります。

そのため、次のような要件を満たすソリューションを選択する必要があります。

- 複数リージョンの App Service にトラフィックを分散
- リージョン障害時でもアプリケーションを利用可能にする
- トラフィック急増時にレート制限を適用
- グローバルユーザー向けの高性能アクセス

---

## 2. 要件整理

この問題の要件を整理すると、主に次の3つのポイントがあります。

まず、トラフィック分散の要件です。

- 複数リージョンの App Service にトラフィックを均等に分散
- グローバルロードバランシング

次に、可用性の要件です。

- リージョン障害時のフェールオーバー
- 高可用性の確保

最後に、セキュリティおよびパフォーマンスの要件です。

- レート制限（Rate limiting）
- トラフィック急増への対応

これらの条件から、**グローバルレベルのロードバランシングとトラフィック管理が可能なサービス**を選択する必要があります。

---

## 3. 技術の基本概念

Azure には、トラフィック分散やアプリケーション配信を実現する複数のサービスがあります。

まず、グローバルアプリケーション配信のためのサービスとして **Azure Front Door** が提供されています。

Azure Front Door は、Microsoft のグローバルエッジネットワークを利用して、ユーザーのリクエストを最適なバックエンドへルーティングするサービスです。

主な機能には次のものがあります。

- グローバルロードバランシング
- ヘルスチェックによる自動フェールオーバー
- レート制限
- SSL 終端
- Web Application Firewall（WAF）
- アプリケーションアクセラレーション

このように、Azure Front Door はグローバル Web アプリケーションの入口として機能します。

---

## 4. アーキテクチャまたは設計のポイント

Azure Front Door を使用すると、次のようなグローバルアーキテクチャを構築できます。

まず、ユーザーは最寄りの Microsoft エッジロケーションへ接続します。  
次に Azure Front Door がバックエンドの App Service を評価し、最適なリージョンへリクエストを転送します。

この設計のポイントは次の通りです。

- グローバルエッジネットワークを利用
- 複数リージョン App Service をバックエンドとして登録
- ヘルスチェックによる自動フェールオーバー
- トラフィック分散

さらに、Front Door では **レート制限ルール**を設定できるため、急激なトラフィック増加からアプリケーションを保護できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題の要件を満たすためには、次の機能が必要です。

- グローバルロードバランシング
- リージョン障害時の自動フェールオーバー
- レート制限

Azure Front Door はこれらすべてをネイティブで提供します。

特に重要なポイントは次の通りです。

まず、**グローバルロードバランシング**です。  
Front Door はエッジネットワークでトラフィックを処理し、最適なリージョンにルーティングします。

次に、**高可用性**です。  
バックエンドのヘルスチェックにより、障害リージョンを自動的に除外します。

最後に、**レート制限**です。  
WAF のカスタムルールによりトラフィック制御が可能です。

そのため、このシナリオのすべての要件を満たすことができます。

---

## 6. 他の選択肢が誤りな理由

まず **Azure Traffic Manager** です。

Traffic Manager は DNS ベースのルーティングサービスであり、リージョン間のトラフィック分散は可能ですが、次の機能は提供しません。

- レート制限
- SSL 終端
- エッジレベル処理

そのため、この問題の要件を完全には満たしません。

次に **Azure Load Balancer** です。

Load Balancer は主に **単一リージョン内**の VM やサービス間でトラフィックを分散するサービスです。グローバルロードバランシングには適していません。

最後に **Azure Application Gateway** です。

Application Gateway は Layer 7 ロードバランサーですが、通常は **単一リージョン内**で使用されるサービスです。

したがって、マルチリージョン構成には適していません。

---

## 7. 最終回答

**A. Azure Front Door**

---

## 8. まとめ

Azure のトラフィック分散サービスは用途によって使い分ける必要があります。

代表的なサービスを整理すると理解しやすくなります。

まず、グローバル Web アプリケーション配信です。

- Azure Front Door  
  → グローバル HTTP/HTTPS ロードバランシング

次に、DNS レベルのルーティングです。

- Azure Traffic Manager  
  → DNS ベーストラフィック管理

さらに、リージョン内の負荷分散です。

- Azure Load Balancer  
  → L4 負荷分散

- Azure Application Gateway  
  → L7 負荷分散

このため、**複数リージョンの App Service に対してトラフィック分散、フェールオーバー、レート制限を提供する場合は Azure Front Door が最適なサービス**となります。
