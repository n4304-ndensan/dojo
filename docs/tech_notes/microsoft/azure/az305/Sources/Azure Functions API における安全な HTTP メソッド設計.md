# Azure Functions API における安全な HTTP メソッド設計

## 1. 背景（シナリオ）

ある企業では、Azure Functions を使用して顧客向け Web アプリケーションのバックエンド API を構築しています。この API は HTTP ベースで公開され、クライアントアプリケーションからリソースへのアクセスを提供します。

この API では、異なるアクセスレベルをサポートする必要があります。具体的には、一般ユーザーが認証なしで特定のリソースを読み取れるようにする匿名アクセスが必要です。一方で、新しいリソースを作成する操作については、認証されたユーザーのみが実行できるように制御する必要があります。

さらに、セキュリティ上の理由から、既存のリソースが不正に変更されたり削除されたりしないように API を設計することが重要です。つまり、API の設計段階で HTTP メソッドの利用範囲を制限することで、不正操作のリスクを最小化する必要があります。

## 2. 要件整理

問題文から読み取れる要件を整理すると、API 設計には次の条件が必要です。

まず、匿名ユーザーが特定のリソースを読み取ることができる必要があります。これは、公開情報などを閲覧する用途を想定しています。

次に、認証されたユーザーは新しいリソースを作成できる必要があります。つまり、データ登録などの操作をサポートする必要があります。

さらに、既存のリソースが不正に変更または削除されないようにする必要があります。これは API のセキュリティ設計として重要なポイントです。

これらの要件を整理すると、次のようになります。

- 匿名ユーザーがリソースを読み取れる  
- 認証済みユーザーが新しいリソースを作成できる  
- 不正なリソース変更を防止する  
- 不正なリソース削除を防止する  

## 3. 技術の基本概念

REST API では、HTTP メソッドを使用してリソースに対する操作を表現します。それぞれの HTTP メソッドには標準的な意味があります。

代表的な HTTP メソッドには次のようなものがあります。

GET はサーバーからデータを取得するためのメソッドです。サーバーの状態を変更しない安全な操作として設計されています。そのため、読み取り専用アクセスに適しています。

POST は新しいリソースを作成するためのメソッドです。クライアントがデータを送信し、サーバー側で新しいデータを生成する際に使用されます。

PUT は既存のリソースを完全に更新するために使用されるメソッドです。場合によってはリソースを上書きする可能性があるため、慎重なアクセス制御が必要です。

PATCH はリソースの一部を更新するためのメソッドです。完全な更新ではなく、部分的な変更を行う場合に使用されます。

DELETE はリソースを削除するためのメソッドです。適切な認可がない場合、不正なデータ削除のリスクがあります。

## 4. アーキテクチャまたは設計のポイント

このシナリオでは、API の安全性を確保するために使用する HTTP メソッドを制限することが重要です。具体的には、読み取り、作成、そして限定的な更新のみを許可し、削除や完全更新などの操作は許可しない設計が適しています。

この設計では、まず GET メソッドを使用して匿名ユーザーによる読み取りアクセスを提供します。GET はサーバーの状態を変更しないため、安全に公開できます。

次に、POST メソッドを使用して認証済みユーザーが新しいリソースを作成できるようにします。POST はリソース作成に使用される標準的な方法です。

さらに、必要に応じて PATCH を使用してリソースの一部を更新できるようにします。ただし、この操作は認証および認可によって厳格に管理される必要があります。

## 5. 設計判断（なぜこの構成になるか）

この問題の正解は **B. GET、POST、PATCH のみ** です。

GET メソッドを使用することで、匿名ユーザーが安全にリソースを読み取ることができます。このメソッドはデータ取得専用であり、サーバー状態を変更しません。

POST メソッドは、新しいリソースを作成するために使用されます。これにより、認証済みユーザーが新しいデータを追加できます。

PATCH メソッドは既存リソースの部分更新を可能にします。完全な更新ではなく特定のフィールドのみ変更できるため、細かい制御が可能になります。また、認証と認可を組み合わせることで、不正な変更を防ぐことができます。

DELETE メソッドを許可するとリソース削除が可能になるため、不正な削除のリスクが高まります。また PUT を許可するとリソース全体を上書きできるため、予期しない変更が発生する可能性があります。

そのため、GET・POST・PATCH の組み合わせが最も安全で柔軟な API 設計になります。

## 6. 他の選択肢が誤りな理由

GET と POST のみを許可する構成では、リソースの更新操作をサポートできません。そのため、アプリケーションの機能が制限されてしまいます。

GET、POST、DELETE の組み合わせでは削除操作が可能になります。これは問題文の要件である「既存リソースの不正な削除を防止する」という条件に反します。

GET、POST、PUT、DELETE を許可する構成では、リソース更新と削除の両方が可能になります。これにより、不正な変更や削除のリスクが高くなるため、要件を満たしません。

## 7. 最終回答

B. GET、POST、PATCH のみ

## 8. まとめ

この問題は、REST API 設計における HTTP メソッドの役割とセキュリティ設計を理解しているかを確認するものです。API 設計では、必要最小限の HTTP メソッドのみを公開することで、セキュリティリスクを低減できます。

GET は読み取り専用アクセス、POST は新規リソース作成、PATCH は部分更新に使用されます。一方で、DELETE や PUT を公開するとリソース削除や完全更新が可能になるため、不正操作のリスクが高まる可能性があります。

したがって、匿名読み取りアクセス、認証ユーザーによるリソース作成、そして安全な部分更新を実現する最適な HTTP メソッドの組み合わせは GET、POST、PATCH です。
