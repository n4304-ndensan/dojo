# Azure Functions API の認証レベル設計（Anonymous）

## 1. 背景（シナリオ）

顧客向け Web アプリケーションのバックエンドとして、HTTP ベースの API を設計する必要があります。この API はユーザーの機密データを扱うため、セキュリティを確保しながら適切なアクセス制御を行うことが重要です。

設計要件として、特定のリソースについてはユーザーが認証なしで **読み取り専用アクセス**できる必要があります。一方で、データの更新や作成などの **書き込み操作は認証されたユーザーのみに制限**する必要があります。

さらに、API の悪用や過剰なリクエストを防ぐために **レート制限（Rate Limiting）** を実装する必要があります。これらの要件を満たすためには、アクセシビリティとセキュリティのバランスを取った認証レベルを選択する必要があります。

## 2. 要件整理

このシナリオでは、API 設計において次の要件が提示されています。これらの条件を満たす認証レベルを選択する必要があります。

まず、API の特定のリソースに対して **匿名の読み取りアクセス**が必要です。これはユーザーがログインしていなくても API を利用できることを意味します。

次に、データの変更を伴う操作については **認証されたユーザーのみに制限**する必要があります。つまり、API 側で認証の有無によって操作を制御する必要があります。

最後に、匿名アクセスによる悪用を防ぐため **レート制限** を実装する必要があります。

このシナリオの要件は次の通りです。

- 読み取りは匿名アクセス可能  
- 書き込みは認証ユーザーのみ  
- API のレート制限を実装  
- セキュリティと公開性のバランスを維持  

## 3. 技術の基本概念

Azure Functions の HTTP トリガーでは、API のアクセス制御のために **Authorization Level（認証レベル）** を設定できます。

Azure Functions で使用できる主な認証レベルには次のものがあります。

Azure Functions の認証レベルには次の種類があります。

- Anonymous（匿名アクセス可能）
- Function（関数キーが必要）
- Admin（管理キーが必要）

この中で **Anonymous** を使用すると、クライアントは認証情報を持たなくても API にアクセスできます。

## 4. アーキテクチャまたは設計のポイント

このシナリオでは、読み取り操作と書き込み操作を API エンドポイントで分離する設計が一般的です。

例えば以下のような構成になります。

- GET エンドポイント：匿名アクセス可能  
- POST / PUT / DELETE：認証必須  

このようにすることで、公開可能なデータは誰でも取得できる一方で、データの変更は認証されたユーザーのみが実行できます。

また、API Gateway や API Management を利用して **レート制限** を設定することで、匿名アクセスによる過剰なリクエストを防ぐことができます。

## 5. 設計判断（なぜこの構成になるか）

このシナリオでは、読み取り操作について **匿名アクセスが必須条件**になっています。

そのため、Azure Functions の認証レベルとしては **Anonymous** を選択する必要があります。

匿名アクセスを許可した場合でも、次のような追加のセキュリティ対策を組み合わせることで安全性を維持できます。

- API Gateway でのレート制限  
- 認証が必要なエンドポイントの分離  
- API ログ監視  
- センシティブデータの公開禁止  

このように設計することで、公開 API の利便性とセキュリティの両方を確保できます。

## 6. 他の選択肢が誤りな理由

Function 認証レベルは API キーが必要になります。そのため、匿名ユーザーが読み取り API を利用できなくなり、要件を満たしません。

Admin 認証レベルは管理者専用のアクセスであり、公開 API には適していません。

User-assigned Managed Identity は Azure リソース間の認証に使用される仕組みであり、HTTP API のクライアント認証とは関係ありません。

## 7. 最終回答

B. 匿名（Anonymous）

## 8. まとめ

Azure Functions API で匿名ユーザーに読み取り専用アクセスを提供する必要がある場合、認証レベルとして **Anonymous** を使用するのが最適です。

書き込み操作は別のエンドポイントで認証を要求する設計にすることで、セキュリティとアクセシビリティを両立できます。また、API Gateway や Azure API Management を利用してレート制限を導入することで、匿名アクセスによる悪用も防止できます。
