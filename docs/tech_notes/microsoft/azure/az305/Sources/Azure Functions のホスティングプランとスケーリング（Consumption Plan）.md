# Azure Functions のホスティングプランとスケーリング（Consumption Plan）

## 1. 背景（シナリオ）

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Storage Account]]
ある企業では、Azure Storage Queue に格納されたメッセージを処理する Azure Functions をデプロイしています。キューにはバックエンド処理のための作業項目が格納されており、関数はそれらのメッセージを取得して処理します。

しかし、このシステムではメッセージの量が非常に変動します。通常はメッセージ数が少ない状態ですが、特定のタイミングで大量のメッセージがキューに投入されることがあります。

そのため、関数の実行環境には次のようなスケーリング特性が求められています。

まず、キューに大量のメッセージが追加された場合には、自動的にスケールアウトして複数のインスタンスでメッセージ処理を行う必要があります。これにより、大量のメッセージでも迅速に処理できます。

一方で、キューが空の状態では処理が不要になるため、実行環境は **ゼロインスタンスまでスケールダウン**できる必要があります。これにより、無駄なコストを削減できます。

このようなイベント駆動型で変動の大きいワークロードに適した Azure Functions のホスティングプランを選択する必要があります。

---

## 2. 要件整理

この問題では、Azure Functions のホスティングプランに関していくつかの重要な要件があります。これらを整理すると、適切なプランを選択しやすくなります。

まず、スケーリングに関する要件があります。キューのメッセージ量に応じて関数のインスタンス数を自動的に増減させる必要があります。

次に、コスト効率に関する要件があります。メッセージが存在しないときには処理を行わないため、インスタンスをゼロまでスケールダウンできる必要があります。

さらに、この処理はイベント駆動型のワークロードであるという点も重要です。Azure Storage Queue のメッセージが関数実行のトリガーとなります。

これらの条件をまとめると、次のようになります。

- キューメッセージ量に応じた自動スケール  
- 高いスケールアウト能力  
- アイドル時のスケールツーゼロ  
- イベント駆動型処理  

これらの条件を満たす Azure Functions のホスティングプランを選択する必要があります。

---

## 3. 技術の基本概念

### Azure Functions

Azure Functions は、サーバーレスコンピューティングサービスです。開発者はコードのみを実装すればよく、インフラ管理は Azure が行います。

Functions はイベント駆動型のアーキテクチャに適しており、HTTP リクエスト、ストレージイベント、キューメッセージなどをトリガーとして処理を実行できます。

特にストレージキューの処理では、キューに新しいメッセージが追加されると Azure Functions が自動的に起動して処理を行います。

---

### Azure Functions のホスティングプラン

Azure Functions には複数のホスティングプランが存在します。それぞれのプランはスケーリング動作や料金モデルが異なります。

代表的なプランとして次のものがあります。

- Consumption Plan  
- Premium Plan  
- App Service Plan  

それぞれの特徴を理解することが重要です。

---

### Consumption Plan

Consumption Plan は Azure Functions の最も一般的なサーバーレスプランです。このプランでは、関数はイベントが発生したときのみ実行されます。

このプランの最大の特徴は **自動スケールとスケールツーゼロ**です。

キューのメッセージ数が増えると、Azure が自動的に関数インスタンスを増やして並列処理を行います。逆に、メッセージが存在しない場合はインスタンスがゼロになります。

料金も実行ベースで課金されるため、関数が実行されていないときはコストが発生しません。

この特性により、変動の大きいイベント駆動型ワークロードに最適です。

---

## 4. アーキテクチャまたは設計のポイント

Azure Storage Queue と Azure Functions を組み合わせることで、イベント駆動型の非同期処理アーキテクチャを構築できます。

アプリケーションは処理タスクをキューに追加します。Azure Functions はキューを監視し、新しいメッセージを検出すると関数を実行します。

メッセージが大量に追加された場合、Azure Functions は自動的に複数のインスタンスを起動し、並列処理を行います。

一方で、キューが空になった場合にはすべてのインスタンスが停止し、実行環境はゼロになります。

この仕組みにより、スケーラブルでコスト効率の高いバックグラウンド処理システムを構築できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題の最も重要な要件は **スケールツーゼロ**です。

Consumption Plan は Azure Functions の中で唯一、完全なスケールツーゼロをサポートするプランです。

キューのメッセージ数が増えると自動的にスケールアウトし、メッセージが存在しない場合にはインスタンスがゼロになります。

また、実行ベースの課金モデルにより、アイドル状態のコストを削減できます。

このため、キュー処理のようなイベント駆動型ワークロードでは Consumption Plan が最も適した選択になります。

---

## 6. 他の選択肢が誤りな理由

### Azure Functions Premium Plan

Premium Plan は高性能な関数実行環境を提供しますが、常に最低1つのインスタンスが稼働します。そのため、完全なスケールツーゼロは実現できません。

### Azure Functions App Service Plan

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Virtual Machines]]
App Service Plan では専用の VM 上で関数が実行されます。インスタンスは常に稼働しているため、アイドル時にもコストが発生します。

### Azure Functions Dedicated Plan

Dedicated Plan も App Service Plan と同様に専用リソースを使用するため、ゼロまでスケールダウンすることはできません。

---

## 7. 最終回答

A. Azure Functions の Consumption Plan
