# Azure Functions を利用した読み取り専用 API 設計（GET メソッド）

## 1. 背景（シナリオ）

ある組織では、顧客向けのWebアプリケーションのためにHTTPベースのAPIを設計しています。このAPIはAzure Functionsを使用して実装される予定であり、サーバーレスアーキテクチャを採用しています。

APIは一般ユーザーがアクセスできるため、認証を必要としない **匿名アクセス（Anonymous Access）** をサポートする必要があります。

ただし、セキュリティ要件として、ユーザーはデータを閲覧することはできますが、データの変更や削除などの **書き込み操作は禁止** する必要があります。

そのため、APIのHTTPメソッドを適切に制限することが重要になります。

## 2. 要件整理

この問題の要件を整理すると次のようになります。

- Azure Functions で API を実装
- 匿名アクセスを許可
- 読み取り専用 API
- 不正な書き込み操作を防止

REST APIでは、HTTPメソッドによって操作の種類が決まります。そのため、適切なHTTPメソッドを選択することがセキュリティの観点でも重要になります。

## 3. 技術の基本概念

HTTPベースのAPIでは、一般的に以下のメソッドが使用されます。

まず **GET** です。  
GETはサーバーからデータを取得するためのメソッドです。これは読み取り専用操作であり、リソースを変更しません。

次に **POST** です。  
POSTは新しいリソースを作成するために使用されます。

次に **PUT** です。  
PUTは既存リソースを更新するために使用されます。

最後に **DELETE** です。  
DELETEはリソースを削除するために使用されます。

このように、POST、PUT、DELETEはすべて **データを書き込む操作** に該当します。

## 4. アーキテクチャまたは設計のポイント

このシナリオではAPIが **読み取り専用** である必要があります。

そのため、APIはデータ取得のみを許可し、データ作成・更新・削除の操作を許可してはいけません。

REST API設計の観点では、読み取り専用のエンドポイントには **GETメソッドのみを許可する** のがベストプラクティスです。

Azure FunctionsではHTTPトリガーを使用する際に、許可するHTTPメソッドを指定できます。

この設定により、APIはGETリクエストのみを受け付けるようになります。

## 5. 設計判断（なぜこの構成になるか）

この問題の重要なポイントは **不正な書き込み操作を防ぐこと** です。

もしPOST、PUT、DELETEなどのメソッドを許可すると、次のような操作が可能になります。

- POST：新しいデータ作成
- PUT：データ更新
- DELETE：データ削除

これらはすべて書き込み操作であり、要件に違反します。

一方、GETメソッドはデータ取得のみを行うため、データを変更することはできません。

そのため、セキュリティ要件を満たす最も適切な設定は **GETメソッドのみを許可すること**です。

## 6. 他の選択肢が誤りな理由

### B. GET と POST のみ

POSTは新しいリソースを作成するためのメソッドです。

そのため、不正な書き込み操作を防止するという要件に違反します。

### C. GET、POST、PUT

POSTはデータ作成、PUTはデータ更新に使用されます。

どちらも書き込み操作のため、この構成は許可できません。

### D. GET、POST、および DELETE

POSTはデータ作成、DELETEはデータ削除を行うメソッドです。

どちらもデータ変更操作であり、要件を満たしません。

## 7. 最終回答

読み取り専用のAPIを実装し、不正な書き込み操作を防ぐために設定するHTTPメソッドは次です。

**GET のみ**

したがって **正解は A** です。

## 8. まとめ

HTTPベースのAPIでは、HTTPメソッドによって操作の種類が決まります。

主なメソッドの役割は次の通りです。

- GET：データ取得
- POST：データ作成
- PUT：データ更新
- DELETE：データ削除

読み取り専用APIを設計する場合は、GETメソッドのみを許可することでデータ変更を防ぐことができます。

そのため、Azure Functionsで匿名アクセス可能な **読み取り専用APIを実装する場合は GET メソッドのみを許可するのが最も安全な設計**になります。
