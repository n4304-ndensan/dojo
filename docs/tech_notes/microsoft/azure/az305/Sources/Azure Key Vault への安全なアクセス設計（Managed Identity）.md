# Azure Key Vault への安全なアクセス設計（Managed Identity）

## 1. 背景（シナリオ）

ある組織では **App3** というWebアプリケーションをAzure上にデプロイしています。このアプリケーションは、Azure Key Vault に保存された機密情報（シークレット）にアクセスする必要があります。

しかし、この環境では高いセキュリティ要件があり、アプリケーションコードに認証情報（パスワード、クライアントシークレット、接続文字列など）を埋め込むことは禁止されています。

また、セキュリティコンプライアンスの観点から、以下の要件を満たす必要があります。

まず、Key Vaultへのアクセスは最小権限の原則に従う必要があります。つまり、アプリケーションは必要なシークレットだけにアクセスできるようにする必要があります。

さらに、Key Vaultへのアクセスは監査および監視が可能でなければなりません。これにより、不正アクセスやセキュリティインシデントを検出できます。

## 2. 要件整理

この問題では、App3がAzure Key Vaultに安全にアクセスする方法を設計する必要があります。

シナリオから読み取れる要件を整理すると以下の通りです。

- Key Vaultのシークレットへアクセスする必要がある
- アプリケーションコードに認証情報を埋め込まない
- 最小権限の原則を守る
- セキュリティ監査・監視が可能
- 運用管理がシンプルである

これらの要件を満たすには、Azureの**Managed Identity（マネージドID）**を利用する方法が最適です。

## 3. 技術の基本概念

Azureでは、アプリケーションが他のAzureサービスへ安全にアクセスするための仕組みとして **Managed Identity** が提供されています。

Managed Identityは、Azureリソースに対して自動的にIDを割り当てる仕組みです。このIDはAzure Active Directory（現在のMicrosoft Entra ID）で管理されます。

Managed Identityを使用すると、アプリケーションは次のような利点を得られます。

まず、認証情報の管理が不要になります。Azureが自動的に認証情報を管理するため、開発者がパスワードやシークレットを管理する必要がありません。

次に、セキュリティが向上します。認証情報をコードや設定ファイルに保存しないため、情報漏洩のリスクが大幅に減少します。

Managed Identityには2種類あります。

まず **System-assigned managed identity（システム割り当てID）** です。このIDはAzureリソースに直接関連付けられ、そのリソースのライフサイクルと連動します。

次に **User-assigned managed identity（ユーザー割り当てID）** です。これは独立したAzureリソースとして作成され、複数のAzureサービスから共有できます。

## 4. アーキテクチャまたは設計のポイント

このシナリオでは、Key Vaultに安全にアクセスするための設計が重要です。

ユーザー割り当てマネージドIDを使用すると、アプリケーションはAzure ADを通じてKey Vaultに認証できます。

その際、Key Vault側では**アクセスポリシー**または**RBAC**を使用して、マネージドIDに必要な権限を付与します。

具体的には次のような構成になります。

まず、ユーザー割り当てマネージドIDを作成します。

次に、そのマネージドIDをApp3に割り当てます。

その後、Key Vaultのアクセスポリシーで次のような権限を付与します。

- 特定のシークレットの読み取り
- 必要最小限の操作

この構成により、App3は安全にKey Vaultのシークレットへアクセスできます。

## 5. 設計判断（なぜこの構成になるか）

この問題の最大のポイントは次の3つです。

- 認証情報をコードに埋め込まない
- 最小権限の原則
- 監査可能なアクセス

ユーザー割り当てマネージドIDを利用すると、これらすべてを満たすことができます。

まず、認証情報が不要になります。AzureがIDを管理するため、アプリケーションは安全に認証できます。

次に、Key Vaultのアクセスポリシーにより、必要なシークレットだけにアクセス権を付与できます。

さらに、Azure Monitor や Key Vault のログ機能を利用することで、アクセス履歴を監査できます。

そのため、最適な方法は **ユーザー割り当てマネージドIDを使用すること**です。

## 6. 他の選択肢が誤りな理由

### B. サービスプリンシパル + Owner 権限

この方法には大きな問題があります。

まず、接続文字列やシークレットを管理する必要があります。これは「認証情報をコードに埋め込まない」という要件に反します。

さらに、Owner権限は過剰な権限であり、最小権限の原則に違反します。

### C. AADアプリケーション + クライアントシークレット

この方法でもクライアントシークレットを管理する必要があります。

つまり、シークレットの管理が必要になるため、セキュリティリスクが増加します。

### D. Azure Function を中継する方法

この方法ではAzure Functionを中継として利用します。

しかし、これは以下の問題があります。

- 不要なアーキテクチャの複雑化
- 追加のレイテンシ
- 障害ポイントの増加

App3が直接Key Vaultへアクセスする方がシンプルで安全です。

## 7. 最終回答

App3がAzure Key Vaultにアクセスする最適な方法は次です。

**ユーザー割り当てマネージドIDを使用し、Key Vaultアクセスポリシーで必要なシークレットへのアクセスを付与する**

したがって **正解は A** です。

## 8. まとめ

Azure Key Vaultを安全に利用するためには、認証情報をアプリケーションに埋め込まない設計が重要です。

この問題のポイントは次の通りです。

まず、**Managed Identity** を使用すると、Azureリソースが安全に他のサービスへアクセスできます。

次に、Key VaultのアクセスポリシーやRBACを利用することで、最小権限の原則を実現できます。

さらに、Azure Monitorや監査ログを利用することで、Key Vaultアクセスを追跡できます。

そのため、AzureアプリケーションからKey Vaultへアクセスする際には **Managed Identityを利用する方法が最も安全で推奨される設計**になります。
