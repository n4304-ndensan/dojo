# Azure Logic Apps によるワークフローオーケストレーション

## 1. 背景（シナリオ）

ある企業では、Azure 上で **注文処理システム（Order Processing System）** を構築しています。このシステムでは、顧客からの注文を処理するために複数のクラウドサービスが連携して動作します。

注文の受付は **Azure Functions** を通じて行われます。その後、注文データは複数のサービスに送信され、処理が進められます。

また、コンプライアンスや監査のために、注文処理のすべてのステップを **Azure Blob Storage** に記録する必要があります。

このようなシステムでは、複数のクラウドサービスを順序立てて実行し、全体の処理フローを管理する仕組みが必要になります。

---

## 2. 要件整理

問題文から読み取れる要件を整理すると、次のようになります。

まず、注文処理の開始です。

- Azure Functions で注文を受信

次に、データ処理です。

- 注文データを 2 つの下流サービスに送信
- 並列処理または順序処理の管理

さらに、監査ログです。

- 各処理ステップのログを保存
- Azure Blob Storage に記録

最後に、アーキテクチャ要件です。

- 複数コンポーネント間のワークフロー管理
- 処理フローのオーケストレーション

このような要件には **ワークフローオーケストレーションサービス**が必要です。

---

## 3. 技術の基本概念

Azure には複数サービスを統合するための **ワークフローオーケストレーションサービス** が存在します。

その代表的なサービスが **Azure Logic Apps** です。

Azure Logic Apps は、アプリケーションやサービス間の処理フローを自動化するためのサービスです。

Logic Apps の特徴を整理すると次の通りです。

まず、ワークフロー自動化です。

- イベントをトリガーに処理開始
- 条件分岐
- 並列処理

次に、サービス統合です。

- Azure Functions
- Azure Storage
- SaaS アプリケーション

さらに、ビジュアルワークフローです。

- GUIベースのワークフローデザイナー
- コード不要で構築可能

これにより、複雑な処理フローを簡単に設計できます。

---

## 4. アーキテクチャまたは設計のポイント

この注文処理システムでは、Logic Apps を中心にワークフローを構築します。

処理の流れは次のようになります。

まず、注文が Azure Functions に送信されます。

次に、Logic Apps がトリガーされます。

その後、ワークフローの処理が実行されます。

この処理では次のステップが実行されます。

- 注文データの取得
- 下流サービスへの送信
- 処理ログの記録

Logic Apps は **並列処理** をサポートしているため、注文情報を複数のサービスへ同時送信することも可能です。

さらに、Blob Storage コネクタを使用することでログを簡単に保存できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題のポイントは **ワークフローのオーケストレーション**です。

Azure Logic Apps は次の機能を提供します。

まず、サービス統合です。

- Azure Functions
- Azure Storage
- 外部サービス

次に、ワークフロー管理です。

- 条件分岐
- 並列処理
- リトライ制御

さらに、監査ログです。

- 各処理のトラッキング
- Azure Monitor 統合

このため、複数コンポーネントの処理を統合する用途に最適です。

---

## 6. 他の選択肢が誤りな理由

まず **Azure Service Bus（トピック）**です。

Service Bus はメッセージングサービスであり、ワークフロー管理機能はありません。

次に **Azure Event Grid** です。

Event Grid はイベントルーティングサービスであり、複数ステップのワークフローを管理する用途には適していません。

最後に **Azure Data Factory** です。

Data Factory は主に **ETL（データ統合）** 用のサービスであり、リアルタイムの注文処理ワークフローには向いていません。

---

## 7. 最終回答

**A. Azure Logic Apps**

---

## 8. まとめ

Azure には複数の統合サービスがありますが、用途によって使い分ける必要があります。

主なサービスは次の通りです。

まず、ワークフローオーケストレーションです。

- **Azure Logic Apps**
  → サービス間の処理フロー管理

次に、メッセージングです。

- **Azure Service Bus**
  → 非同期メッセージ処理

さらに、イベントルーティングです。

- **Azure Event Grid**
  → イベント通知

そして、データ統合です。

- **Azure Data Factory**
  → ETL処理

そのため、**複数のAzureサービス間で注文処理ワークフローを管理する場合は Azure Logic Apps が最適なサービス**になります。
