# Azure PaaS サービスへのプライベート接続設計（Private Endpoint）

## 1. 背景（シナリオ）

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Key Vault]]
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Storage Account]]
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure SQL Database]]
ある企業では、Azure 上にアプリケーションを構築しており、そのアプリケーションは Azure の PaaS サービス（例：Azure Storage、Azure SQL Database、Azure Key Vault など）を利用する予定です。PaaS サービスは通常、パブリックエンドポイントを持ち、インターネット経由でアクセス可能な設計になっています。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Virtual Network (VNet)]]
しかし、この企業のネットワーク設計では、セキュリティ上の理由から重要な要件があります。まず、Azure 仮想ネットワーク（VNet）から PaaS サービスへの通信は **パブリックインターネットを経由してはならない**という要件があります。つまり、通信は Azure のバックボーンネットワーク内部で完結する必要があります。

さらに、もう一つ重要な要件があります。PaaS サービスは **オンプレミスネットワークからアクセスできないようにする必要**があります。企業の内部ネットワークから直接 PaaS サービスにアクセスできてしまうと、セキュリティポリシーに違反する可能性があるためです。

このような要件を満たすためには、Azure のネットワーク機能を正しく理解し、PaaS サービスへのアクセス経路を制御する必要があります。

---

## 2. 要件整理

この問題を正しく理解するためには、まずネットワーク要件を整理することが重要です。問題文には複数の条件が含まれており、それぞれが設計判断に影響します。

まず、Azure 仮想ネットワークから PaaS サービスへの通信に関する要件があります。この通信はセキュリティの観点からインターネットを経由してはならず、Azure の内部ネットワークを利用する必要があります。

- VNet から PaaS への通信は **パブリックインターネットを経由しない**

次に、アクセス制御に関する要件があります。企業はオンプレミス環境から PaaS サービスにアクセスできないようにする必要があります。

- **オンプレミスネットワークからのアクセスを禁止**

この2つの条件を同時に満たすソリューションが必要になります。

---

## 3. 技術の基本概念

### Azure Private Endpoint

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Private Link]]
Azure Private Endpoint は、Azure Private Link の機能の一つであり、PaaS サービスに対して **仮想ネットワーク内のプライベート IP アドレスを割り当てる仕組み**です。

通常、Azure の PaaS サービスはパブリックエンドポイントを持っています。これはインターネット経由でアクセスできるように設計されています。しかし、Private Endpoint を使用すると、PaaS サービスは仮想ネットワーク内の **プライベート IP アドレス**を通じてアクセスされるようになります。

この仕組みによって、アプリケーションは次のような通信経路で PaaS サービスに接続します。

- アプリケーション  
- 仮想ネットワーク  
- Private Endpoint  
- Azure PaaS サービス  

この通信は Azure のバックボーンネットワーク内部で処理されるため、インターネットを経由しません。

---

### Private Link の特徴

Private Endpoint を使用すると、PaaS サービスへのアクセス方法が大きく変わります。通常のパブリックエンドポイントではインターネット経由でアクセスできますが、Private Endpoint を使用するとアクセス経路が制限されます。

Private Endpoint の主な特徴には次のようなものがあります。

- PaaS サービスへ **プライベート IP 経由で接続**
- Azure バックボーンネットワークのみを使用
- パブリックインターネットを経由しない
- パブリックアクセスを無効化可能

これらの特徴により、PaaS サービスへの通信を完全にプライベートネットワーク内に閉じることができます。

---

## 4. アーキテクチャまたは設計のポイント

今回のネットワーク設計では、PaaS サービスを安全に利用するために Private Endpoint を使用します。Private Endpoint を作成すると、対象の PaaS サービスに対して仮想ネットワーク内に専用のプライベート IP アドレスが割り当てられます。

この構成により、アプリケーションから PaaS サービスへの通信は仮想ネットワーク内部の通信として扱われます。そのため、インターネットを経由することなく Azure のバックボーンネットワーク上で通信が行われます。

さらに、PaaS サービスのパブリックアクセスを無効化することで、外部ネットワークからのアクセスを遮断することも可能です。これにより、オンプレミスネットワークから直接 PaaS サービスへアクセスすることを防止できます。

この設計は、Azure のセキュアネットワークアーキテクチャにおいて非常に一般的なパターンであり、多くの企業が機密データを扱うワークロードで採用しています。

---

## 5. 設計判断（なぜこの構成になるか）

この問題で重要なのは **通信経路とアクセス制御を同時に満たすこと**です。

まず、VNet から PaaS への通信がインターネットを経由しないようにする必要があります。この要件は Private Endpoint を使用することで満たされます。Private Endpoint を使用すると通信は Azure バックボーン内で完結します。

次に、オンプレミスネットワークから PaaS サービスへのアクセスを防ぐ必要があります。Private Endpoint は特定の仮想ネットワーク内に配置されるため、そのネットワークに接続していない環境からはアクセスできません。

このように、Private Endpoint は **プライベート接続とアクセス制御を同時に実現する機能**であり、今回の要件に最も適したソリューションとなります。

## 6. 他の選択肢が誤りな理由

### A サービスエンドポイントポリシー

サービスエンドポイントは、VNet から Azure サービスへの接続を Azure バックボーン経由にする機能です。しかしサービスエンドポイントを使用しても、PaaS サービスは依然としてパブリックエンドポイントを持ちます。そのため、オンプレミスネットワークからパブリックエンドポイント経由でアクセスできてしまう可能性があります。

### C 強制トンネル付き VPN

強制トンネルは、インターネットトラフィックをオンプレミスにルーティングするための機能です。しかしこれは PaaS サービスへのプライベート接続を提供するものではありません。今回の要件である PaaS へのプライベートアクセス制御とは目的が異なります。

### D ネットワークセキュリティグループ

NSG は仮想ネットワーク内の通信を制御するための機能です。しかし NSG は PaaS サービスのパブリックエンドポイントへのアクセスを完全に制御することはできません。また NSG だけではプライベート接続を提供することもできません。
