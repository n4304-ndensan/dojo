# Azure Policy による TDE 強制と自動修復（DeployIfNotExists）

## 1. 背景（シナリオ）

あるAzureサブスクリプションには **200個のAzure SQL Database** が存在しています。組織のセキュリティポリシーにより、すべてのデータベースに対して **透過データ暗号化（Transparent Data Encryption: TDE）** を有効化する必要があります。

TDEはデータベースに保存されるデータを暗号化する仕組みであり、保存データ（Data at Rest）の保護において非常に重要なセキュリティ機能です。

さらに、この環境では次のようなガバナンス要件があります。

- 新規作成されるデータベースもポリシーに準拠する必要がある  
- 既存のデータベースも準拠している必要がある  
- 準拠していないデータベースは **自動的に修復（Remediation）** される必要がある  

このような要件を満たすためには、Azureのガバナンス機能である **Azure Policy** を利用する必要があります。

## 2. 要件整理

この問題で重要なポイントを整理すると次の通りです。

まず、すべてのAzure SQL Databaseで **TDEを有効化する必要**があります。

次に、既存のデータベースだけでなく **新しく作成されるデータベースにもポリシーが適用される必要**があります。

さらに重要なのが **自動修復（Remediation）** です。つまり、TDEが無効になっているデータベースを検出した場合、システムが自動的に設定を修正する必要があります。

このような要件では、Azure Policyの **Effect（効果）** の選択が重要になります。

## 3. 技術の基本概念

Azure Policyでは、ポリシーが違反を検出した場合にどのような動作を行うかを **Effect（効果）** として定義します。

代表的なPolicy Effectには次の種類があります。

まず **Deny** です。  
これはポリシーに違反するリソースの作成や変更を拒否します。

次に **Append** です。  
これはリソース作成時に追加の設定を付与するために使用されます。

次に **Modify** です。  
これは既存のリソース設定を変更してポリシーに準拠させることができます。

最後に **DeployIfNotExists** です。  
これは特定の設定やリソースが存在しない場合に、自動的にそれをデプロイします。

## 4. アーキテクチャまたは設計のポイント

このシナリオでは、次の2つの条件を満たす必要があります。

- TDEが有効であること
- 無効な場合は自動的に修復されること

DeployIfNotExistsを使用すると、Azure Policyは対象リソースを評価し、必要な設定が存在しない場合に自動的に設定を適用できます。

つまり、次のような動作になります。

まず、Azure PolicyがすべてのAzure SQL Databaseを評価します。

次に、TDEが有効かどうかを確認します。

もしTDEが無効になっている場合、Policyが自動的にTDEを有効化する設定をデプロイします。

この仕組みにより、既存リソースと新規リソースの両方に対してコンプライアンスを維持できます。

## 5. 設計判断（なぜこの構成になるか）

この問題の重要なキーワードは **自動修復（Automatic Remediation）** です。

Denyポリシーは、ポリシー違反のリソース作成を防ぐことはできますが、既存リソースを修正することはできません。

Appendはリソース作成時に設定を追加するだけであり、既存リソースの修復には適していません。

Modifyは設定変更に使用できますが、リソースが存在しない場合に設定を展開する用途ではありません。

一方、DeployIfNotExistsは以下の機能を提供します。

- リソース構成のチェック
- 不足設定の自動デプロイ
- 自動修復

そのため、このシナリオでは DeployIfNotExists が最適な選択になります。

## 6. 他の選択肢が誤りな理由

### A. Deny

Denyポリシーは非準拠リソースの作成を拒否します。

しかし、この問題では **既存の200個のデータベース** を自動修復する必要があります。

Denyでは既存リソースを修正できません。

### C. Modify

Modifyはリソースのプロパティ変更に使用されますが、TDEのような設定を自動デプロイする用途では一般的ではありません。

また、この問題ではDeployIfNotExistsの方が適切です。

### D. Append

Appendはリソース作成時に設定を追加するだけです。

既存リソースを修復する機能はありません。

## 7. 最終回答

この要件を満たすAzure Policyの効果は次です。

**DeployIfNotExists**

したがって **正解は B** です。

## 8. まとめ

Azure Policyはクラウド環境のガバナンスとコンプライアンスを維持するための重要なサービスです。

Policy Effectには複数の種類がありますが、用途によって使い分ける必要があります。

主なポイントは次の通りです。

まず、Denyは非準拠リソースの作成を防ぎます。

次に、Appendはリソース作成時に設定を追加します。

そして、DeployIfNotExistsは不足している設定やリソースを自動的にデプロイします。

今回のように **既存リソースを自動修復する必要がある場合は DeployIfNotExists を使用するのがベストプラクティス**です。
