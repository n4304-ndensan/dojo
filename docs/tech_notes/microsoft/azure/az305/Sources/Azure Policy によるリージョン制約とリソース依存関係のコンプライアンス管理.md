# Azure Policy によるリージョン制約とリソース依存関係のコンプライアンス管理

## 1. 背景（シナリオ）

ある企業では、Azure 上で **Azure App Service と Azure SQL Database** を使用した多層アプリケーションを設計しています。このアプリケーションは規制要件に厳格に準拠する必要があり、リソースの配置場所や構成について明確な制約があります。

特に、すべてのリソースは特定の承認された Azure リージョン内に配置される必要があります。また、各 App Service インスタンスは同じリージョン内にある対応する SQL Database とペアリングされる必要があります。

さらに重要なのは、これらの要件が **デプロイ時だけでなく、リソースのライフサイクル全体を通じて継続的に監視および強制される必要がある**という点です。そのため、ガバナンスとコンプライアンスを自動的に管理できる仕組みが必要になります。

## 2. 要件整理

このシナリオではクラウドガバナンスの観点から、いくつかの重要な要件があります。

まず、リソースは承認された Azure リージョンにのみデプロイされる必要があります。これはデータレジデンシーや規制要件に関連する重要な制約です。

次に、App Service と SQL Database のリソースは同一リージョンに配置される必要があります。これはアプリケーションアーキテクチャの整合性とパフォーマンスの両方に影響します。

さらに、これらの要件はデプロイ時だけでなく継続的に監視される必要があります。つまり、コンプライアンス違反のリソースが作成された場合、自動的に検出または拒否できる仕組みが必要です。

この問題の要件は次のように整理できます。

・リソースのリージョン制限  
・App Service と SQL Database の同一リージョン配置  
・リソース依存関係の制御  
・継続的なコンプライアンス監視  
・自動的なポリシー適用  

## 3. 技術の基本概念

Azure ではクラウドガバナンスを実現するために **Azure Policy** が提供されています。このサービスは、Azure リソースに対してルールを定義し、そのルールに基づいてリソースの作成や更新を制御する仕組みです。

Azure Policy の基本的な役割を理解することが重要です。

まず、ポリシーを使用すると、リソースのプロパティや構成に対してルールを定義できます。たとえば「特定のリージョンでのみリソースを作成できる」といった制約を設定できます。

また、Azure Policy はリソース作成時にルールを評価するだけでなく、既存リソースのコンプライアンス状態も継続的に評価します。これにより、組織のポリシー違反を検出できます。

さらに、Azure Policy ではカスタムポリシー定義を作成できるため、複雑な条件やリソース間の関係を制御することも可能です。

## 4. アーキテクチャまたは設計のポイント

今回のシナリオでは、組織の規制要件を自動的に強制する必要があります。そのため、Azure Policy を使用してリソースの配置ルールを定義します。

まず、ポリシーを使用してリソースのリージョンを制限できます。例えば、許可されたリージョンのリストを定義し、それ以外のリージョンでのリソース作成を拒否することが可能です。

次に、App Service と SQL Database の配置関係を制御するポリシーを作成できます。これにより、関連するリソースが同一リージョンに配置されるように制約を設定できます。

また、Azure Policy はコンプライアンス状態を継続的に監視するため、ポリシー違反が発生した場合に検出することができます。

## 5. 設計判断（なぜこの構成になるか）

Azure Policy は Azure 環境におけるガバナンスとコンプライアンス管理の中心的なサービスです。ポリシーを定義することで、組織のルールを自動的に適用できます。

特に今回のシナリオでは、リージョン制限やリソース依存関係などのルールを継続的に監視する必要があります。Azure Policy はこのような要件に対応するために設計されています。

さらに、ポリシーは管理グループ、サブスクリプション、リソースグループなどのさまざまなスコープで適用できます。そのため、大規模な環境でも一貫したガバナンスを実現できます。

## 6. 他の選択肢が誤りな理由

この問題では他にもいくつかの Azure サービスが提示されていますが、それぞれ役割が異なります。

ARM テンプレートはインフラストラクチャをデプロイするためのテンプレートですが、デプロイ後の継続的なコンプライアンス監視を行うことはできません。

Azure Blueprints は環境を標準化して展開するための仕組みですが、実際のコンプライアンス評価やポリシー適用は内部で Azure Policy を使用しています。

また、管理グループはサブスクリプションを整理するための構造であり、具体的なポリシーの実装や評価は Azure Policy が担当します。

## 7. 最終回答

A. リソースの配置およびリソース間依存関係に関するカスタム定義を含む Azure Policy

## 8. まとめ

クラウド環境では、組織のガバナンスルールを確実に適用するための仕組みが重要になります。特にリージョン制限やリソース構成ルールなどは、手動運用では管理が困難になります。

Azure Policy を使用することで、リソースの配置ルールや構成ポリシーを自動的に適用し、コンプライアンスを継続的に監視することができます。

これにより、アプリケーションのライフサイクル全体を通じて規制要件を確実に遵守するクラウドガバナンスを実現できます。
