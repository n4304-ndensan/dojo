# Azure Policy を使用したタグ付けコンプライアンスの管理

## 1. 背景（シナリオ）

ある組織ではAzureサブスクリプションを管理しており、すべてのAzureリソースに対して特定のタグを必ず付与するというガバナンスポリシーを導入しています。タグはコスト管理、所有者管理、環境分類（例: Production / Dev / Test）などの目的で利用されます。

しかし現在、既存のリソースの中にはタグ付けポリシーに準拠していないものが存在しています。そのため、まず既存のリソースの状態を確認し、タグが不足しているリソースをレポートとして把握する必要があります。

さらに、今後作成される新しいリソースについては、必要なタグが付いていない場合に作成できないようにし、タグ付けポリシーを強制的に適用する必要があります。

このような要件を満たすためには、Azure Policyの適切な「Effect（効果）」を組み合わせて利用する必要があります。

---

## 2. 要件整理

この問題では、Azure Policyを利用して次の2つの要件を満たす必要があります。

まず既存のリソースのコンプライアンス状況を確認する必要があります。既存リソースを削除したり変更したりする必要はありませんが、タグが不足しているリソースをレポートとして確認できるようにする必要があります。

次に、新しく作成されるリソースが必ずタグ付けルールに従うようにする必要があります。タグがないリソースの作成は許可されないようにする必要があります。

この要件を整理すると次の通りです。

- 既存リソースのコンプライアンス状況をレポート  
- 新規リソースのタグ付けを強制  
- ポリシー違反を防止  
- Azureガバナンスの強化  

---

## 3. 技術の基本概念

Azure Policyは、Azureリソースが組織のガバナンスルールに従っているかを評価し、必要に応じて制御するサービスです。

Azure Policyでは「Effect」と呼ばれる動作を指定することができます。代表的なポリシー効果には次のようなものがあります。

まずAuditです。Auditは、リソースがポリシーに準拠しているかを評価し、違反しているリソースをレポートとして記録します。この効果は既存リソースの状態を確認する場合に適しています。

次にDenyです。Denyは、ポリシー条件を満たさないリソースの作成または更新を拒否します。これにより、ポリシー違反のリソースが作成されることを防ぐことができます。

この2つの効果を組み合わせることで、既存リソースの可視化と新規リソースの制御を同時に実現できます。

---

## 4. アーキテクチャまたは設計のポイント

Azureのガバナンス設計では、ポリシーを段階的に適用することが推奨されます。

最初のステップとして、Auditポリシーを使用して既存リソースの状態を確認します。これにより、タグが不足しているリソースの一覧を取得できます。

次に、新規リソースに対してはDenyポリシーを適用します。これにより、タグが存在しないリソースの作成がブロックされます。

この構成では次のような管理が可能になります。

- 既存リソースのコンプライアンスレポート  
- 新規リソース作成時のポリシー強制  
- 組織のガバナンス強化  
- タグベースのコスト管理  

このようなポリシー設計により、Azure環境全体のタグ管理を統一できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題では「既存リソースのレポート」と「新規リソースの制御」という2つの異なる目的があります。

Auditポリシーは既存リソースを評価してレポートを生成するために最適です。既存リソースを変更することなく、コンプライアンス状況を確認できます。

一方で、Denyポリシーはポリシー違反のリソース作成を防ぐために使用されます。これにより、タグ付けポリシーを組織全体で強制できます。

この2つのポリシー効果を組み合わせることで、次のメリットが得られます。

- 既存リソースの可視化  
- 新規リソースのコンプライアンス強制  
- ガバナンスの自動化  
- 運用管理の効率化  

---

## 6. 他の選択肢が誤りな理由

Appendポリシーは、リソース作成時に指定した値を追加するために使用されます。しかし、この方法ではタグがなくてもリソースが作成される可能性があり、ポリシー違反を完全に防ぐことはできません。

Modifyポリシーはリソース作成時に自動修正を行うことができますが、この問題の要件では「作成を防止する」ことが求められているため、Denyの方が適しています。

また、既存リソースにDenyを適用することはできません。Denyは新規リソースの作成や更新時にのみ適用されます。

---

## 7. 最終回答

A.  
既存リソースの **Audit** と新規リソースの **Deny（Enforce）**

---

## 8. まとめ

Azure Policyを利用することで、Azure環境全体のガバナンスとコンプライアンスを管理できます。

この問題のように、既存リソースの状況確認と新規リソースの制御を同時に行う場合には、次のポリシー効果を組み合わせます。

Audit  
既存リソースのコンプライアンス状態をレポートする。

Deny  
ポリシー違反の新規リソース作成をブロックする。

この組み合わせにより、**タグ付けポリシーを継続的に維持するAzureガバナンス環境**を構築できます。
