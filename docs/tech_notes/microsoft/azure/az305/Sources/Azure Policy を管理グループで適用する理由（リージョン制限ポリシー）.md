# Azure Policy を管理グループで適用する理由（リージョン制限ポリシー）

## 1. 背景（シナリオ）

ある企業では、Azure 環境におけるガバナンスとコンプライアンスを強化するために、リソースの配置リージョンを制限することを検討しています。企業のポリシーにより、すべての Azure リソースは **West Europe（西ヨーロッパ）** または **North Europe（北ヨーロッパ）** のみで作成される必要があります。

このようなリージョン制限は、データレジデンシーや法規制の要件を満たすためによく使用されます。また、運用やサポートを特定の地域に集中させるためにも有効です。

企業の Azure 環境は複数のサブスクリプションで構成されています。これらのサブスクリプションは異なる部門やプロジェクトに使用されていますが、すべてのサブスクリプションに共通のガバナンスポリシーを適用する必要があります。

さらに、このポリシーは次の条件を満たす必要があります。

- 既存のリソースにも適用される  
- 新しく作成されるリソースにも適用される  
- 複数のサブスクリプション全体に適用される  

このような要件を満たす Azure Policy の割り当て方法を選択する必要があります。

---

## 2. 要件整理

この問題では、Azure Policy の適用範囲と管理方法に関する要件が重要になります。

まず、ポリシーは **複数のサブスクリプションにまたがるリソース**に適用される必要があります。そのため、単一のサブスクリプションやリソースグループだけを対象とする方法では不十分です。

次に、ポリシーは **既存リソースと新規リソースの両方**に適用される必要があります。Azure Policy は作成時の評価だけでなく、既存リソースのコンプライアンス評価も行うことができます。

さらに、運用管理の観点から **ポリシーの管理はできるだけ簡単であるべき**です。サブスクリプションごとに個別の設定を行う方法では管理負荷が高くなります。

これらの条件をまとめると、次のようになります。

- 複数サブスクリプションを一括管理  
- 新規リソースと既存リソースの両方を評価  
- ガバナンスを階層的に適用  
- 管理オーバーヘッドを最小化  

このような要件を満たす Azure のガバナンス機能を利用する必要があります。

---

## 3. 技術の基本概念

### Azure Policy

Azure Policy は、Azure 環境におけるリソースの構成やコンプライアンスを強制するためのサービスです。組織のルールや標準をポリシーとして定義し、それをリソースに適用することができます。

例えば次のようなポリシーを定義できます。

- 特定リージョンのみ許可  
- 特定の SKU のみ使用可能  
- タグの必須化  
- 暗号化の強制  

Azure Policy はリソースの作成時に評価されるだけでなく、既存のリソースについても定期的にコンプライアンスチェックを行います。

---

### Azure 管理スコープ

Azure には階層的な管理スコープがあります。ポリシーはこの階層のどのレベルにも割り当てることができます。

Azure の主なスコープ階層は次のようになっています。

- Management Group  
- Subscription  
- Resource Group  
- Resource  

この階層構造により、上位スコープで設定したポリシーは下位スコープに自動的に継承されます。

例えば、管理グループにポリシーを割り当てると、その管理グループ配下のすべてのサブスクリプションとリソースにポリシーが適用されます。

---

## 4. アーキテクチャまたは設計のポイント

複数のサブスクリプションを持つ企業では、Azure のガバナンスを効率的に管理するために **Management Group** を使用することが一般的です。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Role-Based Access Control (RBAC)]]
管理グループはサブスクリプションをまとめて管理するための論理コンテナです。ポリシーや RBAC 権限などを管理グループレベルで設定すると、その設定はすべてのサブスクリプションに適用されます。

例えば、このシナリオでは次のような構成になります。

- Management Group  
  - Subscription A  
  - Subscription B  
  - Subscription C  

この管理グループに「許可されたリージョン」ポリシーを割り当てると、すべてのサブスクリプションでそのポリシーが有効になります。

これにより、新しいサブスクリプションが追加された場合でも、同じポリシーが自動的に適用されます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題の重要なポイントは **複数のサブスクリプションにまたがるポリシー適用**です。

サブスクリプション単位でポリシーを設定することも可能ですが、その場合はサブスクリプションごとにポリシー設定を管理する必要があります。

一方、管理グループにポリシーを割り当てると、その管理グループ配下のすべてのサブスクリプションに自動的にポリシーが適用されます。

また、新しいサブスクリプションが追加された場合でも、自動的にポリシーが適用されるため、ガバナンスを一貫して維持することができます。

そのため、このシナリオでは **管理グループレベルでポリシーを割り当てる方法**が最も適しています。

---

## 6. 他の選択肢が誤りな理由

### 各リソースグループでポリシーを割り当てる

この方法では、すべてのリソースグループでポリシーを設定する必要があります。新しいリソースグループが作成されるたびに設定が必要になり、管理負荷が大きくなります。

### 各サブスクリプションでポリシーを割り当てる

サブスクリプションごとにポリシーを設定する方法は可能ですが、サブスクリプション数が増えると管理が複雑になります。また、新しいサブスクリプションを追加した場合にポリシー設定を忘れるリスクがあります。

### Azure Automation スクリプト

スクリプトを使用してリソースの場所をチェックすることも可能ですが、これはリアルタイムのポリシー強制ではありません。Azure Policy を使用する方がガバナンス管理に適しています。

---

## 7. 最終回答

C. サブスクリプションを含む管理グループで、単一のポリシー課題を作成します。
