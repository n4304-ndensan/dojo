# Azure RBAC とスコープ階層によるリソース作成権限の判断

## 1. 背景（シナリオ）

Azure では、ユーザーやグループに対してリソース操作の権限を付与するために **Azure RBAC（Role Based Access Control）** が使用される。RBAC では、ユーザーやグループに対してロールを割り当てることで、どのリソースに対してどの操作を行えるかを制御する。

Azure の RBAC には重要な概念として **スコープ（Scope）** がある。スコープとは、そのロールが適用される範囲のことであり、Azure のリソース階層と密接に関係している。Azure のリソースは階層構造で管理されており、上位のスコープで付与された権限は下位のスコープに継承されるという特徴がある。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Management Group]]
今回のシナリオでは、管理グループ、サブスクリプション、リソースグループという Azure の階層構造の中で、ユーザーがどのリソースを作成できるかを判断する問題である。特に、管理グループレベルの権限がどこまで継承されるのか、またサブスクリプションレベルのロールがどの範囲まで有効なのかを理解することが重要になる。

---

## 2. Azure の階層構造

Azure のリソース管理は、次のような階層構造で構成されている。

```
Management Group
      │
      ▼
Subscription
      │
      ▼
Resource Group
      │
      ▼
Resource
```

この構造では、上位スコープで割り当てられたロールは下位スコープに継承される。つまり、管理グループレベルで付与された権限は、その管理グループに属するすべてのサブスクリプションとリソースグループに適用される。また、サブスクリプションレベルのロールは、そのサブスクリプションに属するすべてのリソースグループとリソースに適用される。

この「権限の継承」が RBAC を理解するうえで最も重要なポイントである。

---

## 3. 問題の構成

問題に登場する要素を整理すると、次のような構成になっている。

管理グループ MG1 の下にサブスクリプション Sub1 が存在している。そして Sub1 の中には RG1 と RG2 というリソースグループが存在する。

ユーザーとロールの関係は次の通りである。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Virtual Machines]]
User1 は Group1 に所属しており、その Group1 には **MG1 レベルで Virtual Machine Contributor ロール**が割り当てられている。つまり User1 は管理グループレベルで VM 作成に関する権限を持っている。

User2 は Group2 に所属しており、その Group2 には **Sub1 レベルで Contributor ロール**が割り当てられている。

User3 は **Sub1 レベルで直接 Contributor ロール**が割り当てられている。

この状態で、次の二つの操作が可能かどうかを判断する必要がある。

1つ目は、User1 が RG1 で仮想マシンを作成できるかどうかである。  
2つ目は、User3 が RG2 でストレージアカウントを作成できるかどうかである。

---

## 4. Virtual Machine Contributor ロールの意味

Virtual Machine Contributor は、Azure における組み込みロールの一つであり、仮想マシンの管理に関する操作を許可するロールである。このロールを持つユーザーは、仮想マシンの作成、開始、停止、削除などの操作を行うことができる。ただし、ネットワークやストレージの管理など、仮想マシン以外のリソースについては完全な管理権限を持たない。

今回の問題では、このロールが **管理グループ MG1 のスコープで割り当てられている**。Azure RBAC のルールでは、管理グループで付与されたロールは、その配下のすべてのサブスクリプションとリソースグループに継承される。そのため、MG1 に属する Sub1 の中にある RG1 でも、この権限が有効になる。

つまり、User1 は RG1 の中で仮想マシンを作成する権限を持つことになる。

---

## 5. Contributor ロールの意味

Contributor ロールは Azure RBAC の代表的な組み込みロールであり、リソースの作成や管理を広く許可するロールである。このロールを持つユーザーは、リソースの作成、変更、削除などのほとんどの操作を行うことができる。ただし、RBAC のロール割り当てを変更する権限は持たない。

今回の問題では、User3 が **Sub1 レベルで Contributor ロール**を持っている。この場合、Sub1 に属するすべてのリソースグループとリソースに対して Contributor 権限が適用される。したがって、Sub1 の中にある RG2 で新しいストレージアカウントを作成することも可能である。

つまり、User3 は RG2 の中でストレージアカウントを作成できる。

---

## 6. 技術的なポイント

この問題の重要なポイントは、Azure RBAC における **権限の継承**である。管理グループで付与された権限は、その管理グループに属するすべてのサブスクリプション、リソースグループ、リソースに適用される。また、サブスクリプションレベルで付与された権限は、そのサブスクリプション内のすべてのリソースグループとリソースに適用される。

これを図で表すと次のようになる。

```
Management Group (MG1)
      │
      │ VM Contributor
      ▼
Subscription (Sub1)
      │
      │ Contributor
      ▼
Resource Groups
   ├── RG1
   └── RG2
```

MG1 のロールは Sub1 に継承され、さらに RG1 や RG2 にも適用される。同様に、Sub1 の Contributor ロールは RG1 と RG2 の両方に適用される。

---

## 7. 他の選択肢が誤りな理由

選択肢 B は、User1 が RG1 で仮想マシンを作成できないとしているが、管理グループレベルの VM Contributor ロールはサブスクリプションとリソースグループに継承されるため、この判断は誤りである。また User3 についても Contributor ロールを持っているため、ストレージアカウントの作成は可能である。

選択肢 C は、User1 の仮想マシン作成を「はい」としている点は正しいが、User3 のストレージ作成を「いいえ」としている点が誤っている。Contributor ロールはストレージアカウントの作成を許可するため、この判断は正しくない。

選択肢 D は、User3 がストレージアカウントを作成できる点は正しいが、User1 が仮想マシンを作成できないとしている点が誤っている。管理グループレベルのロールは下位スコープに継承されるため、User1 にはその権限がある。

---

## 8. 最終回答

**A.  
User1：RG1 の仮想マシン作成 – はい  
User3：RG2 のストレージアカウント作成 – はい**

---

## 9. まとめ

Azure RBAC を理解するうえで最も重要な概念は **スコープと権限の継承**である。管理グループで付与されたロールは、その下にあるサブスクリプションやリソースグループに継承される。また、サブスクリプションレベルで付与されたロールは、そのサブスクリプション内のすべてのリソースグループに適用される。

そのため、管理グループに割り当てられた Virtual Machine Contributor ロールは、下位のリソースグループで仮想マシンを作成する権限を与える。また、サブスクリプションレベルの Contributor ロールは、そのサブスクリプション内のすべてのリソース作成を許可する。Azure 試験では、このような RBAC の階層構造と権限継承の仕組みが頻繁に問われるため、管理グループ、サブスクリプション、リソースグループの関係を正しく理解しておくことが重要である。
