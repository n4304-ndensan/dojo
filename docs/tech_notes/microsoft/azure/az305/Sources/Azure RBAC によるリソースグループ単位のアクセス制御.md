# Azure RBAC によるリソースグループ単位のアクセス制御

## 1. 背景（シナリオ）

ある企業では、Azure 上で複数のアプリケーションを開発しています。開発チームは Azure 環境にアプリケーションをデプロイする必要がありますが、開発者のアクセス権限は慎重に管理する必要があります。特に、開発者が誤って他のプロジェクトのリソースに影響を与えることを防ぐことが重要です。

そのため、開発者には特定のリソースグループ内でのみリソースを作成および管理できる権限を与える必要があります。一方で、サブスクリプション内の他のリソースグループやリソースに対してはアクセスや変更を行えないようにする必要があります。

このようなシナリオでは、Azure のアクセス制御機能を利用して権限の範囲を適切に制限することが求められます。特に、最小特権の原則に基づいた権限管理が重要になります。

## 2. 要件整理

問題文から読み取れる要件を整理すると、以下の条件を満たすアクセス管理が必要になります。

まず、開発者は特定のリソースグループ内でリソースを作成および管理できる必要があります。これはアプリケーション開発やテストのために必要な権限です。

次に、開発者はそのリソースグループ以外のリソースを変更できないようにする必要があります。つまり、サブスクリプション全体へのアクセスは許可されません。

さらに、誤操作や不正な変更によって他のリソースに影響が出ないようにする必要があります。

これらをまとめると、次のような要件になります。

- 開発者は特定のリソースグループのみ管理できる  
- 他のリソースグループへのアクセスは許可しない  
- サブスクリプション全体への権限は不要  
- 最小特権の原則に従ったアクセス制御  

## 3. 技術の基本概念

Azure ではアクセス制御を実現するために **RBAC（Role-Based Access Control）** が使用されます。RBAC はユーザーやグループに対して特定のロールを割り当てることで、リソースへのアクセス権限を管理する仕組みです。

RBAC の重要な特徴は、ロールを割り当てる **スコープ（範囲）** を指定できる点です。スコープには次の階層があります。

- 管理グループ  
- サブスクリプション  
- リソースグループ  
- 個別リソース  

例えば、サブスクリプションレベルでロールを割り当てると、そのサブスクリプション内のすべてのリソースに対する権限が付与されます。一方、リソースグループレベルでロールを割り当てると、そのリソースグループ内のリソースにのみ権限が適用されます。

また、Azure では既定ロールだけでなく、独自の権限セットを定義した **カスタムロール** を作成することもできます。

## 4. アーキテクチャまたは設計のポイント

このシナリオでは、開発者が特定のリソースグループ内のリソースのみ管理できるようにする必要があります。そのためには、RBAC を使用してアクセス権限をリソースグループ単位で制御する設計が適しています。

まず、開発者が必要とする操作を定義したカスタムロールを作成します。このロールには、リソースの作成、読み取り、更新、削除など、開発作業に必要な権限のみを含めます。

次に、そのカスタムロールを対象のリソースグループに対して割り当てます。これにより、開発者はそのリソースグループ内のリソースを管理できますが、他のリソースにはアクセスできません。

この方法により、アクセス範囲を最小限に抑えながら、必要な操作のみを許可できます。

## 5. 設計判断（なぜこの構成になるか）

この問題の正解は **A. 特定のリソースグループに限定された権限を持つカスタムロールを作成し、それを開発者に割り当てる** です。

カスタムロールを作成することで、開発者が実行できる操作を細かく制御できます。また、ロールをリソースグループレベルで割り当てることで、権限の範囲をそのリソースグループに限定できます。

このアプローチは「最小特権の原則」に基づいたアクセス制御のベストプラクティスです。ユーザーには必要最低限の権限のみを付与し、それ以外の操作は許可しません。

その結果、開発者は自分のプロジェクトに必要なリソースを管理できる一方で、サブスクリプション内の他のリソースに影響を与えることはできません。

## 6. 他の選択肢が誤りな理由

サブスクリプションレベルで Contributor ロールを割り当てると、そのサブスクリプション内のすべてのリソースに対して変更権限が与えられます。これは問題の要件である「特定のリソースグループのみ管理する」という条件を満たしません。

Azure Policy はリソース構成のガバナンスを目的とした機能であり、ユーザー権限を直接制御するものではありません。そのため、アクセス権限を制限する方法としては適していません。

Owner ロールは最も強力な権限を持ち、アクセス管理の変更も可能になります。これは開発者に対して過剰な権限であり、セキュリティリスクが高くなります。

## 7. 最終回答

A. 特定のリソースグループに限定された権限を持つカスタムロールを作成し、それを開発者に割り当てます。

## 8. まとめ

この問題は、Azure における RBAC（Role-Based Access Control）の基本的な設計原則を理解しているかを確認するものです。Azure ではロールを割り当てるスコープを指定することで、アクセス権限を細かく制御できます。

特定のリソースグループにのみアクセスを許可する場合は、RBAC を使用してそのリソースグループにロールを割り当てることが最も効果的です。また、カスタムロールを利用することで、必要な権限のみを付与することができます。

したがって、開発者が特定のリソースグループ内のリソースのみを管理できるようにする最適な方法は、リソースグループスコープでカスタムロールを作成して割り当てることです。
