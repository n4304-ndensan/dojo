# Azure Resource Mover によるリージョン間リソース移行

## 1. 背景（シナリオ）

ある組織では、Azure 仮想マシンを含むインフラストラクチャを別のリージョンへ移行する必要があります。対象となるのは仮想マシンだけではなく、それに関連するインフラリソースも含まれます。

具体的には以下のリソースが対象です。

・Azure Virtual Machines  
・Managed Disks  
・Network Interface  
・Public IP Address  

これらのリソースは互いに依存関係を持っており、移行の際にはその依存関係や構成が維持される必要があります。また、移行先は**異なるリージョンの別のリソースグループ**になります。

このような条件のもとで、Azure リソースを安全かつ効率的に移行するためのサービスを選択する必要があります。

## 2. 要件整理

このシナリオでは単純なバックアップやレプリケーションではなく、**リソース構成を維持したままの移行（Migration）**が求められています。

問題の要件を整理すると次のようになります。

まず、仮想マシンだけでなく関連リソースもまとめて移行する必要があります。また、仮想マシンとネットワークインターフェース、ディスクなどの依存関係を維持しなければなりません。

そのため必要な条件は次の通りです。

・Azure リージョン間のリソース移行  
・リソースグループ間の移動  
・VM と関連リソースの依存関係維持  
・構成を保持したまま移行  

これらの条件を満たす Azure サービスが **Azure Resource Mover** です。

## 3. 技術の基本概念

Azure Resource Mover は、Azure リソースを **リージョン間またはリソースグループ間で移動するための専用サービス**です。特に複数の関連リソースを同時に移動する場合に使用されます。

このサービスは、仮想マシンの移行だけでなく、仮想マシンが依存するネットワークやストレージなどのリソースも一緒に移動できる点が特徴です。

Azure Resource Mover が提供する主な機能には次のものがあります。

まず、移行対象のリソースを登録すると、サービスが自動的に依存関係を検出します。そのため、仮想マシンだけでなく、その VM が使用しているディスクやネットワークインターフェースも自動的に関連付けられます。

次に、移行前に検証プロセスが実行されます。これにより、移行先リージョンで問題なく動作するかを事前に確認できます。

さらに、移行プロセスは段階的に実行できるため、大規模なインフラでも安全に移行できます。

## 4. アーキテクチャまたは設計のポイント

Azure Resource Mover を使用したリージョン移行では、リソースの依存関係を正しく管理することが重要です。Azure Resource Mover はこの依存関係を自動的に検出し、グループとして移行します。

移行プロセスは一般的に次のようなステップで進みます。

まず、移行対象のリソースを選択して Resource Mover に追加します。次に、依存関係の検出と検証が行われます。その後、リソースのコピーや準備が行われ、最終的に移行先リージョンでリソースが作成されます。

このプロセスの中で重要なポイントは次の通りです。

・リソース依存関係の自動検出  
・移行前の検証チェック  
・段階的移行プロセス  
・移行後の検証  

これにより、仮想マシンのネットワーク設定やディスク構成を維持したまま移行できます。

## 5. 設計判断（なぜこの構成になるか）

このシナリオでは、仮想マシン単体ではなく **関連リソースを含めた移行**が必要です。そのため、依存関係管理ができるツールが必要になります。

Azure Resource Mover を使用することで、次のメリットがあります。

まず、仮想マシンの依存関係を自動検出できるため、手動で構成を再作成する必要がありません。

次に、リージョン間移行をネイティブにサポートしているため、移行プロセスを簡素化できます。

さらに、移行前の検証機能により、移行失敗のリスクを減らすことができます。

## 6. 他の選択肢が誤りな理由

問題には他の Azure サービスも提示されていますが、それぞれ目的が異なります。

### B Azure Site Recovery

Azure Site Recovery は、主に **災害復旧（Disaster Recovery）** のためのサービスです。

このサービスでは、仮想マシンを別リージョンにレプリケーションできますが、目的は障害発生時のフェイルオーバーです。そのため、リソース構成を保持したままの移行には最適ではありません。

### C Azure Migrate

Azure Migrate は、オンプレミス環境から Azure への移行を計画・評価するためのサービスです。

このツールは主に次の用途で使用されます。

・オンプレミス環境の評価  
・移行計画  
・Azure への移行支援  

Azure 内のリージョン間移行には適していません。

### D Azure Backup

Azure Backup はデータ保護のためのサービスです。バックアップと復元を提供しますが、インフラリソースの移行機能はありません。

そのため、リソース構成や依存関係を維持した移行には使用できません。

## 7. 最終回答

A. **Azure Resource Mover**

## 8. まとめ

Azure Resource Mover は、Azure リソースをリージョン間またはリソースグループ間で移動するための専用サービスです。

特に仮想マシンと関連リソースをまとめて移行する場合に最適です。

重要なポイントを整理すると次の通りです。

まず、VM と関連リソースの依存関係を自動検出します。次に、リージョン間移行をサポートします。さらに、移行前の検証機能によって安全に移行できます。

試験では **「Azure 内のリージョン移行」「依存関係を維持」「VM と関連リソースをまとめて移行」** というキーワードが出てきた場合、Azure Resource Mover が正解になる可能性が非常に高いです。
