# Azure SQL Database におけるマルチテナント設計（Elastic Pool）

## 1. 背景（シナリオ）

ある企業では、複数の顧客（テナント）が同一のアプリケーションを利用する **マルチテナント型Webアプリケーション** を開発しています。このアプリケーションでは、各テナントごとにデータを分離するため、個別のデータベースを使用する設計になっています。

今回の設計では、以下の条件があります。

・30個の SQL データベース  
・各データベースのサイズは約25GB  
・テナントごとに異なるパフォーマンス要件  
・最低 99.95% の可用性  
・需要に応じたスケーリング  
・データベース間でリソースを共有しコスト最適化  

このような要件を満たす Azure SQL Database の構成を選択する必要があります。

## 2. 要件整理

まず、この問題で重要となる要件を整理します。単にデータベースを配置するだけではなく、**マルチテナント環境特有のリソース効率と柔軟性**が求められている点がポイントです。

本シナリオで満たす必要がある要件は次の通りです。

・マルチテナント構成（テナントごとにDB分離）  
・30個のデータベース  
・テナントごとに異なるパフォーマンス要求  
・99.95%の可用性  
・スケーラビリティ  
・リソース共有によるコスト最適化  

これらの条件から、**複数のデータベースでリソースを共有できる仕組み**が必要になります。

## 3. 技術の基本概念

Azure SQL Database には、複数データベースを効率よく運用するためのいくつかのモデルがあります。その中でも、マルチテナントアプリケーションでよく利用されるのが **Elastic Pool（エラスティックプール）** です。

### Elastic Pool

Elastic Pool は、複数の Azure SQL Database が **CPU・メモリ・IO などのリソースを共有できる仕組み**です。

この仕組みは特に以下のような状況で効果を発揮します。

・多数のデータベースが存在する  
・各データベースの負荷が時間によって変動する  
・すべてのデータベースが常に高負荷ではない  

Elastic Pool を利用すると、リソースはプール単位で確保され、データベースはそのリソースを動的に共有します。

そのため、ピーク時のデータベースは多くのリソースを使用し、アイドル状態のデータベースは少ないリソースで動作します。

## 4. アーキテクチャまたは設計のポイント

マルチテナントアプリケーションでは、各テナントの利用状況が異なるため、個別にリソースを割り当てると無駄が多くなります。その問題を解決するために Elastic Pool が利用されます。

Elastic Pool を利用した場合の設計ポイントは次の通りです。

・複数のデータベースを1つのリソースプールに配置  
・DTU または vCore をプール単位で割り当て  
・各データベースが必要に応じてリソースを利用  
・ワークロードの変動に柔軟に対応  

この構成により、各テナントの負荷が異なるマルチテナント環境でも効率的にリソースを利用できます。

## 5. 設計判断（なぜこの構成になるか）

Elastic Pool が最適な理由は、**マルチテナント環境に特化したリソース共有モデル**を提供しているためです。

Elastic Pool を採用することで得られるメリットは次の通りです。

・複数のデータベースでリソースを共有できる  
・ピーク負荷のデータベースにリソースを自動配分  
・未使用リソースの無駄を削減  
・データベースを追加しても管理が容易  
・Azure SQL Database の高可用性（99.95%）を利用可能  

これにより、**スケーラビリティ・コスト最適化・可用性**を同時に満たすことができます。

## 6. 他の選択肢が誤りな理由

問題では他の選択肢も提示されていますが、それぞれ重要な要件を満たしていません。

### A プロビジョニングされた Azure SQL Database を30個作成

この構成では、各データベースが個別のリソースを持つことになります。

その結果、次の問題が発生します。

・リソースが固定割り当てになる  
・使用率が低いデータベースでもコストが発生  
・リソース共有ができない  

そのため、マルチテナント環境ではコスト効率が悪くなります。

### C Azure VM 上の SQL Server

この構成では、SQL Server を仮想マシンで管理する必要があります。

この方式には次の課題があります。

・パッチ管理  
・バックアップ管理  
・OS 管理  
・可用性設計

つまり、**PaaS の Azure SQL Database より管理負担が大きくなります。**

### D Azure SQL Managed Instance

Managed Instance は SQL Server 互換性が高いサービスですが、Elastic Pool のような効率的なリソース共有モデルは提供されません。

そのため、次の点で不利になります。

・コスト最適化が難しい  
・データベース単位のスケール柔軟性が低い  

## 7. 最終回答

B. パフォーマンス階層が異なる30の Azure SQL Database を含む **Elastic Pool**

## 8. まとめ

Azure SQL Database でマルチテナントアプリケーションを設計する場合、Elastic Pool は非常に重要なサービスです。

特に以下の特徴を覚えておくことが重要です。

・複数データベースでリソース共有  
・コスト最適化  
・負荷変動への柔軟な対応  
・高可用性（99.95% SLA）  
・マルチテナントアーキテクチャに最適  

試験では **「多数のデータベース + 異なるワークロード + コスト最適化」** という条件が出た場合、Elastic Pool が正解になるケースが非常に多いです。
