# Azure SQL Database に対する TDE 適用と自動修復のポリシー設計

## 1. 背景（シナリオ）
Azure サブスクリプション内に 200 個の Azure SQL データベースが存在します。新規および既存のデータベースすべてに以下の要件を適用する必要があります。

- 透過データ暗号化 (TDE) を有効にする
- TDE が有効でないデータベースは自動的に修復される

## 2. 要件整理
- **セキュリティ要件**: 保存データの暗号化
- **自動修復**: 非準拠データベースは自動的に TDE を有効化
- **スケーラビリティ**: サブスクリプション内のすべてのデータベースに適用
- **管理負荷の最小化**: 自動化と監視機能を利用

## 3. 技術の基本概念
- **透過データ暗号化 (TDE)**: SQL データベース上のデータをディスクレベルで暗号化
- **Azure Policy**: リソースに対してルールや効果を強制する仕組み
  - ポリシー定義: 適用するルールや自動修復の効果を指定
  - 割り当てスコープ: ポリシーが適用されるリソースの範囲を定義

## 4. アーキテクチャまたは設計のポイント
- サブスクリプションレベルでポリシーを割り当てることで、既存および新規データベースに一括適用
- ポリシー定義で「TDE を有効にする」と「非準拠時は自動修復」の効果を設定
- Azure ポリシーがデータベースを継続的に評価し、準拠状態を監視

## 5. 設計判断（なぜこの構成になるか）
- 割り当て範囲を正しく設定することで、すべての SQL データベースにポリシーを適用可能
- 自動修復機能により、手動での介入なしに TDE が有効化される
- ポリシー割り当てのスコープが最も重要な要素であり、ポリシーの適用漏れやコンプライアンスギャップを防ぐ

## 6. 他の選択肢が誤りな理由
- **カスタムロール (オーナー権限)**: アクセス管理には関係するが、TDE の適用や自動修復を保証しない
- **マネージドID (コントリビューター権限)**: リソース操作は可能だが、コンプライアンス強制は行えない
- **Key Vault アクセスポリシー**: 暗号化キー管理用であり、TDE の自動適用には直接関係しない

## 7. 最終回答
**B. ポリシー割り当ての範囲**

## 8. まとめ
Azure Policy の割り当てスコープを正しく設定することで、サブスクリプション内のすべての SQL データベースに対し TDE を適用し、非準拠データベースを自動修復可能です。このアプローチにより、セキュリティ要件の遵守と管理オーバーヘッドの最小化を両立できます。
