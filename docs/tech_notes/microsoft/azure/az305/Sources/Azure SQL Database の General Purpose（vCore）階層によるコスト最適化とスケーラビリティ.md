# Azure SQL Database の General Purpose（vCore）階層によるコスト最適化とスケーラビリティ

## 1. 背景（シナリオ）

ある企業では、Azure SQL Database を使用して大量データを処理するアプリケーションを運用しています。このアプリケーションでは、通常時の負荷は比較的低いものの、月に一度のレポート生成期間中には大量のデータ処理が発生します。

このピーク期間中は高いパフォーマンスが必要ですが、それ以外の期間ではデータベースの負荷は大きくありません。そのため、常に高性能なデータベース構成を維持するとコストが無駄に高くなる可能性があります。

さらに、データ量は年間約10%のペースで増加すると予測されているため、将来的なスケーラビリティも考慮する必要があります。

## 2. 要件整理

このシナリオでは、パフォーマンスとコストのバランスを取ることが重要な設計ポイントになります。

まず、月に一度のピークレポート処理に対応できる十分なパフォーマンスが必要です。大量のデータ処理を行うため、データベースの計算リソースを柔軟にスケールできる必要があります。

次に、ピーク期間以外はデータベースの利用率が低いため、不要なリソースに対して高額なコストを支払わないようにする必要があります。

また、データ量は今後も増加するため、将来的な拡張に対応できる柔軟なスケーリングモデルが必要になります。

この問題の要件は次のように整理できます。

・ピーク時の高いパフォーマンス  
・通常時のコスト最適化  
・データ増加への対応  
・柔軟なスケーリング  

## 3. 技術の基本概念

Azure SQL Database には複数のサービス階層があり、それぞれ異なる用途に最適化されています。主に vCore モデルと DTU モデルの2種類の課金モデルが存在します。

vCore モデルは CPU、メモリ、ストレージを個別に管理できるため、リソースの調整やコスト管理がしやすいという特徴があります。

Azure SQL Database の主な vCore サービス階層には次のものがあります。

まず、General Purpose（汎用）階層は、バランスの取れたパフォーマンスとコスト効率を提供する階層です。ほとんどのビジネスアプリケーションに適しています。

次に、Business Critical 階層は、低レイテンシと高 I/O パフォーマンスを必要とするミッションクリティカルなアプリケーション向けです。

さらに、Hyperscale 階層は非常に大規模なデータベースや急速なスケーリングが必要なワークロード向けに設計されています。

## 4. アーキテクチャまたは設計のポイント

今回のシナリオでは、月に一度だけ高負荷が発生するワークロードが特徴です。このような場合には、必要なときだけリソースをスケールアップできる構成が最適です。

vCore ベースのサービス階層では、必要に応じて CPU やメモリをスケールアップまたはスケールダウンすることができます。これにより、ピーク期間中にはパフォーマンスを確保し、通常時にはコストを抑えることが可能になります。

General Purpose 階層は、標準的なストレージと計算リソースを提供するため、多くの業務アプリケーションに適しています。また、必要に応じてリソースをスケールできるため、データ量の増加にも柔軟に対応できます。

## 5. 設計判断（なぜこの構成になるか）

General Purpose（vCore）階層は、パフォーマンスとコスト効率のバランスを重視するワークロードに最適です。

ピークレポート期間中には vCore を増やしてパフォーマンスを確保し、通常時にはリソースを縮小することでコストを最適化できます。

また、DTU モデルよりも vCore モデルの方がリソース構成が明確であり、将来的なスケーリングやコスト管理がしやすいという利点があります。

このため、変動するワークロードを持つデータベースに対して最も適したサービス階層となります。

## 6. 他の選択肢が誤りな理由

Hyperscale 階層は非常に大規模なデータベースや極端なスケーラビリティを必要とするワークロード向けです。今回のシナリオではそこまでのスケールは必要ないため、コストが過剰になります。

DTU ベースの Premium 階層は高いパフォーマンスを提供しますが、リソースの柔軟な調整が難しく、使用率が低い期間でも高いコストが発生する可能性があります。

Business Critical 階層は低レイテンシと高可用性を提供する高性能な階層ですが、主にミッションクリティカルな OLTP ワークロード向けであり、今回のコスト最適化要件には適していません。

## 7. 最終回答

D. vCore ベースの汎用（General Purpose）

## 8. まとめ

ワークロードにピークとアイドル期間があるデータベースでは、柔軟にリソースを調整できるサービス階層を選択することが重要です。

Azure SQL Database の General Purpose（vCore）階層は、パフォーマンスとコスト効率のバランスを提供するため、多くのアプリケーションに適しています。

必要に応じてリソースをスケールアップ・スケールダウンできるため、ピーク時のパフォーマンス要件と通常時のコスト最適化の両方を満たすことができます。
