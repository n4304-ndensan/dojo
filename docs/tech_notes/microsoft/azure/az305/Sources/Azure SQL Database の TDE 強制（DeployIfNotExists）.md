# Azure SQL Database の TDE 強制（DeployIfNotExists）

## 1. 背景（シナリオ）

ある Azure サブスクリプションでは、約 200 の Azure SQL データベースが運用されています。  
組織ではセキュリティ強化のため、すべてのデータベースに対して **Transparent Data Encryption（TDE）** を必須とするコンプライアンス要件を定めています。

TDE は Azure SQL Database において保存データを暗号化する機能であり、次のようなセキュリティ上のメリットがあります。

- データベースファイルの暗号化
- バックアップの暗号化
- データ漏洩リスクの低減
- 規制コンプライアンスの遵守

しかし現在の環境では、すべてのデータベースで TDE が有効になっているとは限りません。

そのため、次のような要件を満たすソリューションが必要です。

- 既存のデータベースのコンプライアンスを修正
- 新規データベースでも TDE を強制
- 手動対応なしで自動修復

---

## 2. 要件整理

このシナリオでは Azure Policy を利用してセキュリティポリシーを実装する必要があります。

問題文から読み取れる要件は次の通りです。

まず、既存リソースに関する要件があります。

- 既存の SQL データベースの TDE を確認
- TDE が無効な場合は自動修復

次に、新規リソースに関する要件です。

- 新規 SQL データベース作成時に TDE を自動適用
- 手動設定を不要にする

つまり、この問題では **コンプライアンス違反を自動的に修復できる Azure Policy 効果**を選択する必要があります。

---

## 3. 技術の基本概念

Azure Policy には、リソース構成を評価した結果に対して実行される **Policy Effect（ポリシー効果）** が存在します。

主要なエフェクトを理解しておくことが重要です。

まず、コンプライアンス違反を検出するだけのものがあります。

- **Audit**  
  違反を記録するだけ

次に、リソース作成を制御するものです。

- **Deny**  
  ポリシー違反のリソース作成を拒否

さらに、リソース構成を変更するものがあります。

- **Modify**  
  リソースの設定を変更

最後に、自動修復を行うものです。

- **DeployIfNotExists**

DeployIfNotExists は、ポリシー条件に一致する構成が存在しない場合に **必要な構成を自動デプロイ**します。

---

## 4. アーキテクチャまたは設計のポイント

このシナリオでは Azure Policy を利用して、TDE の自動適用を実現します。

その設計は次のようになります。

まず、Azure SQL Database に対するポリシーを定義します。

ポリシーでは次の条件を設定します。

- SQL Database に TDE が有効か確認
- 無効の場合は修復処理を実行

このとき使用するポリシー効果が **DeployIfNotExists** です。

この構成では、次のような処理が行われます。

1. Azure Policy が SQL Database の設定を評価  
2. TDE が有効か確認  
3. 無効の場合  
4. TDE 有効化設定を自動デプロイ  

この仕組みにより、既存データベースと新規データベースの両方に対してコンプライアンスを維持できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題では次の2つの要件が重要です。

- 既存リソースの修復
- 将来の違反の自動修正

この2つを同時に実現できる Azure Policy 効果が **DeployIfNotExists** です。

DeployIfNotExists の特徴は次の通りです。

- 必要な構成が存在しない場合に自動デプロイ
- 既存リソースの修復（Remediation）
- 新規リソースの自動修正

そのため、TDE のような **セキュリティ設定を自動適用する用途**に非常に適しています。

---

## 6. 他の選択肢が誤りな理由

まず **Modify** です。

Modify は既存リソースのプロパティ変更に使用されますが、必要な構成を新たにデプロイする用途には適していません。

次に **Audit** です。

Audit はコンプライアンス違反を検出するだけであり、修復処理は行いません。

つまり、監査はできますが **自動修復はできません**。

そのため、この問題の要件を満たすことはできません。

---

## 7. 最終回答

**A. DeployIfNotExists**

---

## 8. まとめ

Azure Policy を利用すると、クラウド環境のセキュリティとコンプライアンスを自動的に管理できます。

特に覚えておくべき Policy Effect は次の通りです。

まず、違反検出のみを行うものです。

- **Audit**  
  違反を記録

次に、違反を防止するものです。

- **Deny**  
  リソース作成を拒否

さらに、自動修復を行うものです。

- **DeployIfNotExists**  
  必要な構成を自動デプロイ

このため、**Azure SQL Database の TDE をすべてのデータベースで強制し、自動修復まで行う場合は DeployIfNotExists を使用するのが最適な設計**となります。
