# Azure SQL Database のサービス層設計（Business Critical）

## 1. 背景（シナリオ）

ある企業では、新しいアプリケーションのために Azure SQL Database を使用する計画を立てています。このアプリケーションはビジネスにとって非常に重要なシステムであり、データベースには高い可用性と高性能が求められています。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Availability Zone]]
特に重要な要件として、Azure の **可用性ゾーン（Availability Zone）全体が障害になった場合でも、データベースは利用可能であり続ける必要があります。**

Azure の可用性ゾーンは、同じリージョン内でも物理的に分離されたデータセンター群です。これにより、電源障害やネットワーク障害などが発生した場合でも、他のゾーンが影響を受けないように設計されています。

しかし、アプリケーションがミッションクリティカルな場合には、1つのゾーンが完全に停止してもデータベースが継続して稼働するような構成が必要になります。

さらに、このアプリケーションはトランザクション処理が多く、**できるだけ低い I/O 遅延**が求められています。

このような要件を満たす Azure SQL Database のサービス層を選択する必要があります。

---

## 2. 要件整理

この問題では、Azure SQL Database のサービス層を選択する際に重要な要件があります。これらの要件を整理すると、最適な選択肢が明確になります。

まず、可用性に関する要件があります。データベースは **可用性ゾーン障害を耐える必要があります**。つまり、ゾーン冗長構成が必要になります。

次に、パフォーマンスに関する要件があります。アプリケーションはトランザクション処理を多く行うため、ストレージ I/O の遅延はできるだけ低くなければなりません。

また、データベースはミッションクリティカルなアプリケーションを支えるため、高い信頼性とパフォーマンスが求められます。

これらの条件をまとめると次のようになります。

- 可用性ゾーン障害に耐えられること  
- 非常に低い I/O レイテンシ  
- 高可用性のデータベースアーキテクチャ  
- ミッションクリティカルなワークロード対応  

このような要件を満たす Azure SQL Database のサービス層を選択する必要があります。

---

## 3. 技術の基本概念

### Azure SQL Database のサービス層

Azure SQL Database には複数のサービス層が存在します。これらのサービス層は、性能・可用性・スケーラビリティの観点で異なる特徴を持っています。

代表的なサービス層には次のようなものがあります。

- General Purpose  
- Business Critical  
- Hyperscale  

それぞれのサービス層は異なるアーキテクチャで設計されています。

---

### General Purpose

General Purpose 層は、バランスの取れたパフォーマンスとコスト効率を提供するサービス層です。

この層ではストレージとコンピュートが分離されており、リモートストレージを使用します。そのため、I/O レイテンシは Business Critical 層よりも高くなります。

一般的な業務アプリケーションには適していますが、非常に低いレイテンシを必要とするワークロードには最適ではありません。

---

### Hyperscale

Hyperscale 層は、非常に大規模なデータベース向けに設計されています。データベースサイズが数十テラバイトに成長する可能性がある場合に適しています。

ストレージは分散アーキテクチャで管理され、大規模データのスケーラビリティに優れています。

しかし、この層の主な目的は **大規模スケール**であり、最低レイテンシを提供することではありません。

---

### Business Critical

Business Critical 層は、ミッションクリティカルなワークロード向けに設計されています。

この層では、ローカル SSD ストレージを使用するため、非常に低い I/O レイテンシを実現できます。また、複数のレプリカを使用する高可用性アーキテクチャを採用しています。

さらに、この層ではゾーン冗長構成を使用することで、可用性ゾーン障害にも耐えることができます。

そのため、高性能と高可用性の両方を必要とするアプリケーションに適しています。

---

## 4. アーキテクチャまたは設計のポイント

Business Critical 層では、Always On 可用性グループに似たアーキテクチャが採用されています。

この構成では、複数のレプリカが同期的にデータを保持しています。1つのレプリカが障害になった場合でも、別のレプリカがすぐにプライマリとして昇格することができます。

また、ゾーン冗長オプションを使用することで、これらのレプリカを異なる可用性ゾーンに配置できます。

その結果、1つのゾーンが完全に停止しても、別のゾーンのレプリカが処理を継続します。

さらに、Business Critical 層ではローカル SSD を使用するため、ストレージ I/O の遅延が非常に低くなります。

---

## 5. 設計判断（なぜこの構成になるか）

この問題の最も重要なポイントは **可用性ゾーン障害への耐性と低 I/O レイテンシの両方**です。

General Purpose 層はコスト効率が高いですが、リモートストレージを使用するため I/O レイテンシが高くなります。

Hyperscale 層は大規模データベース向けであり、低レイテンシを最優先とする設計ではありません。

一方、Business Critical 層はミッションクリティカルなワークロード向けに設計されており、ローカル SSD ストレージによる低レイテンシと、複数レプリカによる高可用性を提供します。

そのため、このシナリオでは Business Critical 層が最も適した選択になります。

---

## 6. 他の選択肢が誤りな理由

### General Purpose

General Purpose 層はコスト効率の高い構成ですが、リモートストレージを使用するため I/O レイテンシが Business Critical 層より高くなります。

### Hyperscale

Hyperscale 層は大規模データベース向けのスケーラビリティを提供しますが、最小レイテンシを提供する設計ではありません。

### Standard

Standard 層は基本的なワークロード向けのサービス層であり、ミッションクリティカルな可用性やパフォーマンス要件には適していません。

---

## 7. 最終回答

C. Business Critical
