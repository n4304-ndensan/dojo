# Azure SQL Database のマルチリージョン高可用性設計（Auto Failover Group）

## 1. 背景（シナリオ）

ある企業では、Azure SQL Database をバックエンドとして利用する重要な Web アプリケーションを運用しています。このアプリケーションはユーザー数が多く、ビジネスにとって重要なシステムであるため、サービス停止の影響を最小限に抑える必要があります。

特に、リージョン全体の障害（リージョン障害）が発生した場合でもアプリケーションが継続して利用できるように、**マルチリージョン構成**による高可用性アーキテクチャを設計する必要があります。

さらに、このシステムには次のような要件があります。

・地域障害時でもサービスを継続できる高可用性  
・自動的にフェイルオーバーできる仕組み  
・データ損失を最小限に抑えるレプリケーション  
・フェイルオーバーの管理と監視が容易であること  

このような条件を満たす Azure SQL Database の高可用性ソリューションを選択する必要があります。

## 2. 要件整理

シナリオから読み取れる重要な要件を整理すると、以下の通りです。

この問題では、単なるバックアップや読み取りスケールではなく、**災害発生時の自動切り替えと継続運用**が重要になります。

・マルチリージョン構成  
・高可用性  
・自動フェイルオーバー  
・最小限のデータ損失  
・監視や管理が容易  

これらの要件から、**Azure SQL Database の災害対策（Disaster Recovery）機能**の中でも、自動フェイルオーバーを提供するサービスを選択する必要があります。

## 3. 技術の基本概念

Azure SQL Database では、リージョン間の可用性を確保するためにいくつかのレプリケーション技術が提供されています。これらの仕組みを理解することが、問題の解答を導くために重要です。

### アクティブ ジオレプリケーション

アクティブジオレプリケーションは、別リージョンに読み取り可能なセカンダリデータベースを作成する機能です。

この方式では以下の特徴があります。

・最大4つのセカンダリを作成可能  
・非同期レプリケーション  
・読み取り専用ワークロードに利用可能  
・フェイルオーバーは**手動**

このため、**自動フェイルオーバー要件がある場合には適していません。**

### Auto Failover Group（自動フェイルオーバーグループ）

Auto Failover Group は、Azure SQL Database の **マルチリージョン災害対策の標準機能**です。

この機能の特徴は次の通りです。

・プライマリリージョンとセカンダリリージョンを構成  
・リージョン障害時に**自動フェイルオーバー**  
・DNS エンドポイントによる接続先の自動切替  
・複数データベースをグループ管理  
・Azure Monitor と統合した監視

この仕組みにより、アプリケーションは接続文字列を変更することなくフェイルオーバーを行うことができます。

## 4. アーキテクチャまたは設計のポイント

Azure SQL Database をマルチリージョンで運用する場合、Auto Failover Group を利用すると以下の構成になります。

まず、プライマリリージョンにメインの SQL Database を配置します。そして別リージョンにセカンダリデータベースを作成し、データを継続的にレプリケーションします。

この構成の重要なポイントは次の通りです。

・プライマリリージョン  
・セカンダリリージョン  
・DNSベースの接続エンドポイント  
・自動フェイルオーバーポリシー  
・Azure Monitor による監視

この構造により、リージョン障害が発生した場合でも Azure が自動的にセカンダリへ切り替え、アプリケーションは継続して動作できます。

## 5. 設計判断（なぜこの構成になるか）

今回の問題では、特に **「自動フェイルオーバー」と「管理の容易さ」** が重要な要件です。

Auto Failover Group を採用する理由は次の通りです。

・リージョン障害時に自動的にフェイルオーバーできる  
・接続エンドポイントが DNS で自動切替される  
・複数データベースをグループ単位で管理できる  
・Azure Monitor と統合した監視が可能  
・災害対策（DR）シナリオに最適

このように、Auto Failover Group は **高可用性 + 災害復旧 + 自動管理**を同時に実現できる設計となっています。

## 6. 他の選択肢が誤りな理由

この問題では、他の選択肢がどの要件を満たさないのかを理解することが重要です。

### A 手動フェイルオーバーによるアクティブジオレプリケーション

この機能はリージョン間レプリケーションを提供しますが、フェイルオーバーは手動で行う必要があります。

そのため、以下の問題があります。

・自動フェイルオーバー要件を満たさない  
・人の操作が必要になる  
・復旧時間が長くなる可能性がある  

### C Elastic Pools

Elastic Pools は複数のデータベース間でリソースを共有するための機能です。

この機能の目的は以下です。

・コスト最適化  
・リソース共有  
・スケーラビリティ

つまり **高可用性や災害対策とは関係ありません。**

### D 読み取りスケールアウト（Geo Replication）

読み取りスケールアウトは、セカンダリデータベースを読み取り専用に利用して負荷分散するための機能です。

この機能は次の用途に使われます。

・レポート処理  
・分析クエリ  
・読み取り負荷の分散

しかし、この機能は **書き込みワークロードのフェイルオーバーや自動復旧を目的としていません。**

## 7. 最終回答

B. 異なるリージョンのセカンダリデータベースを持つ **自動フェイルオーバーグループ（Auto Failover Group）**

## 8. まとめ

Azure SQL Database のマルチリージョン高可用性設計では、**Auto Failover Group** が最も重要な機能の一つです。

この機能のポイントを整理すると次の通りです。

・リージョン障害に対応する DR ソリューション  
・自動フェイルオーバーを提供  
・DNS による接続先の自動切替  
・複数データベースをまとめて管理  
・Azure Monitor と統合した監視

試験対策として覚えておくべき重要なポイントは、**「Azure SQL Database の自動フェイルオーバー = Auto Failover Group」** という対応関係です。

マルチリージョン、災害対策、自動フェイルオーバーというキーワードが出た場合、この機能が最も適切な解答になるケースが多いです。
