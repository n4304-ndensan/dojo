# Azure SQL Database の機密データ保護（Always Encrypted）

## 1. 背景（シナリオ）

企業がクラウド上のデータベースに個人情報（PII: Personally Identifiable Information）を保存する場合、データ漏洩や不正アクセスを防ぐために強力なセキュリティ対策が必要になります。特に、医療情報、顧客情報、金融情報などの機密データは、GDPR や HIPAA などの規制の対象になることが多く、厳格な保護が求められます。

Azure SQL Database においても同様であり、単にデータを保存するだけではなく、「誰がデータを見ることができるか」を制御することが重要です。この問題では、データベースに保存されている **個人識別情報（PII）を特権ユーザーのみが閲覧できるようにする方法**を問われています。

つまり、データベースにアクセスできるユーザーであっても、必要な権限を持っていない場合には **実際のデータを閲覧できない仕組み**が必要になります。

## 2. 要件整理

問題文のセキュリティ要件を整理すると、次のようなポイントが重要になります。

まず、このシナリオでは単なるアクセス制御ではなく、**機密データそのものを保護する仕組み**が求められています。

具体的には、次の要件が存在します。

・Azure SQL Database に機密データ（PII）が保存されている  
・特権ユーザーのみがデータを閲覧できる必要がある  
・データベースにアクセスできるユーザーでも、許可されていなければ閲覧できない  
・機密データを安全に保護する仕組みが必要  

このような要件では、単なる認証やマスキングではなく、**強力な暗号化によるデータ保護**が重要になります。

## 3. 技術の基本概念

Azure SQL Database では、データ保護のために複数のセキュリティ機能が提供されています。これらの機能はそれぞれ目的が異なるため、違いを理解することが重要です。

まず、代表的なデータ保護機能には次のようなものがあります。

・Transparent Data Encryption (TDE)  
・Dynamic Data Masking  
・Azure Active Directory 認証  
・Always Encrypted  

これらはすべてセキュリティに関連していますが、保護する対象やレイヤーが異なります。

特に **Always Encrypted** は、機密データを列単位で暗号化し、暗号化された状態でデータベースに保存する機能です。この仕組みでは、データの暗号化と復号化が **クライアント側（アプリケーション側）で行われる**という特徴があります。

そのため、データベース管理者やサーバー管理者であっても、暗号化キーを持っていない限り、実際のデータを見ることはできません。

## 4. アーキテクチャまたは設計のポイント

Always Encrypted を使用すると、データベースに保存される機密情報は常に暗号化された状態になります。この仕組みでは、暗号化キーを利用できるユーザーやアプリケーションのみがデータを復号できます。

Always Encrypted の重要な特徴には次のようなものがあります。

・列単位で機密データを暗号化できる  
・データベース管理者でも平文データを閲覧できない  
・暗号化と復号化はクライアント側で実行される  
・Column Master Key と Column Encryption Key を使用する  
・Azure Key Vault と連携してキー管理が可能  

この設計により、仮にデータベースにアクセスできるユーザーがいたとしても、暗号化キーを持っていなければ機密データを閲覧することはできません。

## 5. 設計判断（なぜこの構成になるか）

この問題のポイントは、「特権ユーザーのみがPIIを閲覧できるようにする」という要件です。

この要件を満たすためには、次のようなセキュリティ機能が必要になります。

・データそのものを暗号化する  
・暗号化キーを持つユーザーのみ復号できる  
・データベース管理者でも平文データを見られない  

これらの条件を満たす Azure SQL Database の機能が **Always Encrypted** です。

Always Encrypted を使用すると、機密データは暗号化された状態で保存され、適切なキーを持つユーザーやアプリケーションのみが復号できます。そのため、PIIを強力に保護することができます。

## 6. 他の選択肢が誤りな理由

この問題では、Azure SQL Database の他のセキュリティ機能も選択肢として提示されていますが、それぞれ目的が異なります。

まず Transparent Data Encryption（TDE）について説明します。TDE はデータベース全体を暗号化する機能であり、保存されているデータファイルを保護します。しかし、データベースにアクセスできるユーザーは復号されたデータを見ることができるため、ユーザー単位の閲覧制御には使用できません。

次に Dynamic Data Masking です。この機能は、クエリ結果の表示時にデータをマスクする機能です。しかし、適切な権限を持つユーザーは元のデータを見ることができるため、強力なデータ保護の仕組みとは言えません。

Azure Active Directory 認証は、ユーザーの認証を管理する仕組みです。これはログイン方法を管理するものであり、データ自体を暗号化する機能ではありません。そのため、PIIの保護には直接関係しません。

## 7. 最終回答

D. Always Encrypted（カラムマスターキーを使用した暗号化）

## 8. まとめ

Azure SQL Database には複数のセキュリティ機能がありますが、それぞれ役割が異なります。TDE は保存データを暗号化する機能であり、Dynamic Data Masking は表示データをマスクする機能です。

一方で Always Encrypted は、機密データを列単位で暗号化し、暗号化キーを持つユーザーのみが復号できる仕組みです。この機能により、データベース管理者であっても機密データを閲覧できない強力なセキュリティを実現できます。

そのため、Azure SQL Database に保存された PII を特権ユーザーのみが閲覧できるようにするには **Always Encrypted** を使用することが最も適切です。
