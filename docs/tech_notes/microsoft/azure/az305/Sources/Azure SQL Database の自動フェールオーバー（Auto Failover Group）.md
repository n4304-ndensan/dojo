# Azure SQL Database の自動フェールオーバー（Auto Failover Group）

## 1. 背景（シナリオ）

ある企業では、Azure SQL Database をバックエンドとして使用する重要な Web アプリケーションを運用しています。このアプリケーションはビジネスにとって重要なシステムであるため、サービス停止のリスクを最小限に抑える必要があります。

特に問題となるのは **リージョン障害（地域的な停電）**です。もし Azure のプライマリリージョンが利用できなくなった場合でも、アプリケーションは継続して動作する必要があります。そのためには、データベースの高可用性構成と自動フェールオーバーが不可欠です。

このシナリオでは、Azure SQL Database において **リージョン障害時に自動フェールオーバーを実現できる機能**を選択する必要があります。

## 2. 要件整理

問題の要件を整理すると、次のポイントが重要になります。

まず、Web アプリケーションは **高可用性**を維持する必要があります。つまり、障害が発生してもサービスが停止しない構成が求められます。

次に、**リージョン障害に対応できる構成**が必要です。これは単一リージョンの冗長化ではなく、複数リージョン間の冗長化を意味します。

さらに重要なのは **自動フェールオーバー**です。障害発生時に手動操作を必要とせず、自動的にセカンダリリージョンへ切り替わる必要があります。

これらの要件をまとめると次の通りです。

- Azure SQL Database の高可用性  
- マルチリージョン冗長化  
- 自動フェールオーバー  
- アプリケーションの継続稼働  

## 3. 技術の基本概念

Azure SQL Database には、リージョン間の冗長性を確保するための複数の機能があります。その中でも **Auto Failover Group（自動フェールオーバーグループ）** は、高可用性と自動フェールオーバーを実現するための機能です。

Auto Failover Group は、複数の Azure SQL Database をグループ化し、プライマリリージョンとセカンダリリージョン間でデータをレプリケーションします。

この機能には次の特徴があります。

- 複数データベースのグループ管理  
- リージョン間のデータレプリケーション  
- 自動フェールオーバー  
- 読み取り専用セカンダリの利用  
- アプリケーション接続の自動切り替え  

このような機能により、リージョン障害時でもアプリケーションの可用性を維持できます。

## 4. アーキテクチャまたは設計のポイント

Auto Failover Group を利用した高可用性アーキテクチャでは、通常次のような構成を採用します。

まず、プライマリリージョンに Azure SQL Database を配置します。次に、別リージョンにセカンダリデータベースを作成します。このセカンダリデータベースはプライマリデータベースのレプリカとして同期されます。

この構成では次のような動作が行われます。

- プライマリリージョンが通常トラフィックを処理する  
- データはセカンダリリージョンへ継続的にレプリケーションされる  
- ヘルスチェックによりプライマリの状態を監視する  
- プライマリリージョンが停止した場合、自動的にセカンダリへフェールオーバーする  

また、アプリケーションは **フェールオーバーグループのエンドポイント**を使用するため、接続文字列を変更する必要がありません。

## 5. 設計判断（なぜこの構成になるか）

Auto Failover Group が最適な理由は、**自動フェールオーバー機能を提供する Azure SQL Database の唯一のマネージド機能であるため**です。

この機能では Azure がデータレプリケーションを管理し、プライマリリージョンの障害を検知すると自動的にセカンダリリージョンへ切り替えます。

また、複数のデータベースをグループとして管理できるため、アプリケーション全体のデータベース構成を一括でフェールオーバーできます。これにより、重要な Web アプリケーションの可用性を確保できます。

さらに、アプリケーション側では単一の接続エンドポイントを使用するため、フェールオーバー時の設定変更が不要になります。

## 6. 他の選択肢が誤りな理由

Azure SQL Database の他の機能も高可用性に関連しますが、今回の要件には完全には一致しません。

アクティブジオレプリケーションは、最大4つの読み取り可能なセカンダリを作成できます。しかし、フェールオーバーは手動で実行する必要があります。そのため、自動フェールオーバー要件を満たしません。

Basic ティアの可用性は、主に単一リージョン内の可用性を提供するものであり、リージョン障害対策としては十分ではありません。

SQL Database のバックアップは災害復旧には重要ですが、リアルタイムの高可用性や自動フェールオーバーを提供するものではありません。

## 7. 最終回答

C. 自動フェールオーバーグループ（Auto Failover Group）

## 8. まとめ

Azure SQL Database でリージョン障害に対応するためには、マルチリージョン冗長化と自動フェールオーバーを組み合わせる必要があります。

Auto Failover Group は Azure SQL Database の高可用性機能であり、リージョン間レプリケーションと自動フェールオーバーを提供します。これにより、プライマリリージョンが利用できなくなった場合でも、セカンダリリージョンへ自動的に切り替えることができます。

そのため、重要な Web アプリケーションのデータベース可用性を確保するには **Auto Failover Group** を使用するのが最適です。
