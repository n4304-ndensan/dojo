# Azure SQL Database の診断設定とログ送信先の制約

## 1. 背景（シナリオ）

Azure 環境では、アプリケーションやデータベースの状態を監視し、問題発生時に迅速に原因を特定するためにログの収集が非常に重要である。Azure SQL Database も例外ではなく、パフォーマンスやクエリ実行状況、リソース使用状況などを確認するためのログを出力することができる。これらのログは、運用監視、セキュリティ監査、パフォーマンスチューニングなどの用途で利用される。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Log Analytics]]
Azure では、こうしたログを外部の分析基盤や保存先へ送信するための仕組みとして **診断設定（Diagnostic Settings）** が提供されている。診断設定を利用することで、Azure SQL Database を含む Azure リソースが生成するログを Log Analytics ワークスペースに送信して分析したり、ストレージアカウントにアーカイブして長期間保存したりすることが可能になる。

今回のシナリオでは、Azure SQL Database に対してすでに SQLInsights というログカテゴリの診断設定が構成されており、そのログは Log Analytics ワークスペースに送信されると同時に、ストレージアカウントにもアーカイブされている。ここで新しい要件として、SQLInsights ログを **別の Log Analytics ワークスペースや別のストレージアカウントにも送信するための2つ目の診断設定を追加できるか**という点が問われている。この問題は、Azure の診断設定が同一ログカテゴリを複数の宛先へ送信できるかどうかという仕組みを理解しているかどうかを確認する内容である。

---

## 2. 要件整理

この問題のポイントは、Azure SQL Database のログカテゴリである SQLInsights を既存の宛先とは異なる場所に送信できるかどうかという点にある。すでに存在している診断設定では、SQLInsights のログが Log Analytics ワークスペースとストレージアカウントの両方に送信されている。ここで新しい診断設定を追加して、別の Log Analytics ワークスペースや別のストレージアカウントにも同じログを送信できるかどうかを判断する必要がある。

つまり、検討すべきポイントは二つある。一つは、同じログカテゴリを複数の Log Analytics ワークスペースに送信できるかどうかである。もう一つは、同じログを複数のストレージアカウントへ送信できるかどうかである。この二つの制約を理解していないと、正しい選択肢を判断することができない。

---

## 3. Azure 診断設定の仕組み

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Event Hubs]]
Azure の診断設定は、Azure リソースが生成するログやメトリックを外部のサービスへ送信するための機能である。診断設定を構成すると、リソースから出力されるログを複数の送信先に転送することができる。主な送信先としては、Log Analytics ワークスペース、ストレージアカウント、そして Event Hub がある。

Log Analytics ワークスペースはログの分析や検索を目的としたサービスであり、Kusto Query Language（KQL）を使ってログを検索したり、アラートやダッシュボードを作成したりすることができる。ストレージアカウントはログの長期保存やアーカイブ用途に利用されることが多く、比較的低コストでログデータを保存できるという特徴がある。

Azure SQL Database の SQLInsights ログは、クエリパフォーマンスなどの詳細な情報を含むログであり、通常は Log Analytics ワークスペースに送信してパフォーマンス分析や監視に利用される。しかし、コンプライアンスや監査の観点からログを長期間保存する必要がある場合には、ストレージアカウントへのアーカイブも同時に構成されることが多い。

---

## 4. 技術的な制約

Azure の診断設定では、1つのリソースに対して複数の診断設定を作成することができる。しかし、ログ送信先の種類によっては制約が存在する。

Log Analytics ワークスペースの場合、同一のログカテゴリを複数のワークスペースへ同時に送信することはできない。つまり、SQLInsights のログは一つの Log Analytics ワークスペースにのみ送信することができる。もし別のワークスペースへ送信したい場合は、既存の設定を変更する必要があり、新しいワークスペースへ追加で送信することはできない。

一方で、ストレージアカウントについてはこのような制約は存在しない。同じログカテゴリを複数のストレージアカウントへ送信することが可能であり、新しい診断設定を作成して別のストレージアカウントにログを保存することができる。これはストレージアカウントがログ分析ではなく単なる保存先として機能するためである。

この関係を簡単に図で表すと次のようになる。

```
Azure SQL Database
      │
      │ Diagnostic Settings
      │
      ├─ Log Analytics Workspace（1つのみ）
      │
      └─ Storage Account（複数可能）
```

---

## 5. 設計判断

この設計の考え方を理解するためには、Log Analytics とストレージアカウントの役割の違いを理解することが重要である。Log Analytics はログの分析基盤であり、ログのスキーマ管理やインデックス管理が行われる。そのため同じログを複数のワークスペースに送信することは、データの重複管理や運用上の複雑さを生む可能性がある。Azure の設計では、このような問題を避けるために、同じログカテゴリを複数のワークスペースに送信することを制限している。

一方でストレージアカウントは、ログデータをファイルとして保存するだけのシンプルなストレージである。そのため同じログを複数のストレージアカウントに保存することに技術的な問題はなく、バックアップやリージョン分散などの目的で複数のストレージに保存することが可能になっている。

---

## 6. 他の選択肢が誤りな理由

選択肢 A は、Log Analytics ワークスペースとストレージアカウントの両方に追加できるとしている。しかし、Log Analytics ワークスペースは同じログカテゴリを複数のワークスペースへ送信できないため、この選択肢は誤りである。

選択肢 B は、ワークスペースには追加できるがストレージには追加できないとしている。しかし実際には逆であり、ストレージアカウントは複数設定することが可能であるため、この選択肢も正しくない。

選択肢 D は、両方とも追加できないとしているが、ストレージアカウントへの追加は可能であるため誤りとなる。

---

## 7. 最終回答

**C. ワークスペースには「いいえ」、ストレージには「はい」**

---

## 8. まとめ

Azure の診断設定は、リソースが生成するログを外部サービスに送信するための重要な機能であるが、送信先の種類によっていくつかの制約が存在する。特に Log Analytics ワークスペースは同一ログカテゴリに対して一つしか設定できないという制限があり、複数のワークスペースへ同時に送信することはできない。一方でストレージアカウントはログの保存先として利用されるため、複数のストレージアカウントへ同じログを送信することが可能である。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Monitor]]
このようなログ送信の制約は Azure Monitor の設計を理解するうえで重要であり、Azure のアーキテクチャ試験でも頻繁に問われるポイントである。
