# Azure SQL Database の透過的データ暗号化（TDE）を自動修復付きで強制するガバナンス設計

## 1. 背景（シナリオ）

ある企業では、本番環境に存在するすべての Azure SQL Database に対して、データ保護の観点から **透過的データ暗号化（Transparent Data Encryption : TDE）** を必ず有効にするポリシーを導入しようとしています。TDE はデータベースファイルを自動的に暗号化する仕組みであり、ストレージレベルでのデータ漏洩リスクを低減するための重要なセキュリティ機能です。

しかし、実際のクラウド環境では、開発チームや運用チームが新しいデータベースを作成するたびに手動で設定を確認するのは現実的ではありません。そのため、企業は **ガバナンスレベルで TDE を強制し、設定漏れがあった場合には自動的に修復する仕組み**を必要としています。

つまり今回の要件は単なるポリシー監査ではなく、次のような特徴を持つソリューションが求められています。

まず、Azure 環境内の SQL Database が TDE を有効化しているかどうかを **継続的に評価できること**が必要です。次に、もし TDE が無効になっているリソースが発見された場合には **自動的に修復（Remediation）できること**が求められています。さらに、これらの処理は企業の IT 管理者が個別にスクリプトを管理するのではなく、Azure のネイティブなガバナンス機能によって管理される必要があります。

---

## 2. 要件整理

この問題を理解するためには、まず要件を整理する必要があります。問題文から読み取れる要件は主に3つの観点に分類できます。

まず、セキュリティ要件です。企業は本番環境のデータベースに対して強制的に暗号化を適用する必要があります。これはデータ保護およびコンプライアンスの観点から重要です。

- Azure SQL Database で **TDE を必ず有効化する必要がある**

次に、ガバナンス要件です。クラウド環境ではリソースが頻繁に作成されるため、設定の監査を継続的に実行する必要があります。

- Azure リソースの **継続的なコンプライアンス評価**

最後に、自動化要件があります。今回のシナリオでは単に非準拠リソースを検出するだけでは不十分です。企業は **自動的に修復できる仕組み**を求めています。

- 非準拠リソースの **自動修復（Auto remediation）**

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Policy]]
これらの要件を満たす Azure サービスとして最も適切なのが **Azure Policy** です。

---

## 3. 技術の基本概念

### Azure Policy

Azure Policy は Azure のガバナンスサービスであり、リソースの構成が組織のルールに準拠しているかどうかを評価し、必要に応じて制御する機能を提供します。

Azure Policy は単なる監査ツールではなく、組織のクラウドガバナンスを実現するための重要なサービスです。ポリシーを使用すると、Azure 環境全体に対して設定ルールを適用し、リソースの構成が企業の標準に準拠しているかどうかを継続的に確認できます。

Azure Policy の主な用途には次のようなものがあります。

- セキュリティ設定の強制  
- リソース構成の標準化  
- コンプライアンス監査  
- 自動修復によるガバナンス強化  

これらの機能により、企業はクラウド環境全体の構成を統一的に管理できます。

---

### Policy Initiative

Azure Policy には **Policy Initiative** という概念があります。これは複数のポリシーをまとめて管理するための仕組みです。

企業のガバナンスでは、単一のポリシーではなく複数のセキュリティルールを同時に管理することが多いため、ポリシーをグループ化することで管理を簡素化できます。

例えば SQL Database のガバナンスでは、次のようなポリシーをまとめて管理できます。

- TDE の有効化  
- TLS の強制  
- パブリックアクセス制限  

このようなポリシーをまとめて **SQL Security Initiative** のような形で適用することができます。

---

### DeployIfNotExists エフェクト

Azure Policy には複数の **Policy Effects** が存在します。ポリシーエフェクトとは、ポリシー違反が発生した場合にどのような動作をするかを定義するものです。

Azure Policy には次のようなエフェクトがあります。

- Deny  
- Audit  
- DeployIfNotExists  
- Modify  

それぞれのエフェクトには異なる目的があります。今回の問題では **DeployIfNotExists** が重要になります。

DeployIfNotExists は、リソースがポリシーに準拠していない場合に **必要な構成を自動的にデプロイする**仕組みです。つまり、Azure Policy が非準拠リソースを検出した場合、ポリシー定義に基づいて設定を修復できます。

この仕組みによって、企業はリソースの構成ドリフトを自動的に修正できるようになります。

---

## 4. アーキテクチャまたは設計のポイント

今回の設計では、Azure SQL Database に対して TDE を強制し、さらに非準拠リソースを自動修復する仕組みを実装する必要があります。そのためには Azure Policy を中心としたガバナンス構成を使用します。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Management Group]]
具体的には、Azure Policy の **組み込みポリシー（Built-in policy）** を利用して TDE の状態を評価します。そのポリシーを **Policy Initiative** としてまとめ、本番環境のサブスクリプションまたは管理グループに割り当てます。

さらに、ポリシーエフェクトとして DeployIfNotExists を設定することで、非準拠の SQL Database が検出された場合には TDE を自動的に有効化する処理が実行されます。

この構成により、企業は次のようなガバナンスモデルを実現できます。

まず、Azure 環境内のすべての SQL Database を継続的に評価できます。次に、TDE が無効なデータベースが検出された場合でも、管理者が手動で修正する必要はありません。Azure Policy が自動的に修復処理を実行するため、環境のセキュリティ状態を常に維持できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題で重要なのは **「自動修復が可能であること」**です。Azure Policy は単なる監査ツールではなく、DeployIfNotExists エフェクトを使用することで **検出と修復を同時に実行できる**点が大きな特徴です。

もしポリシーが Audit エフェクトのみであれば、非準拠リソースは検出できますが修復は手動になります。しかし DeployIfNotExists を使用すれば、ポリシー違反が検出された時点で必要な設定が自動的にデプロイされます。

このような仕組みは、クラウド環境における **Policy as Code** の考え方と非常に相性が良く、大規模環境でもスケーラブルに運用できます。

## 6. 他の選択肢が誤りな理由

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Blueprints]]
### B Azure Blueprints

Azure Blueprints は、環境の初期構築時にポリシーやリソーステンプレートをまとめて展開するためのサービスです。しかし Blueprints は主に **初期プロビジョニング**に焦点を当てています。既存リソースの継続的な修復を自動化する機能としては Azure Policy の方が適しています。

### C Azure Automation + PowerShell

PowerShell スクリプトを使用すれば TDE を有効化する処理を自動化することは可能です。しかしこの方法ではスクリプトの管理、スケジュール設定、エラー処理などを自分で実装する必要があります。また Azure Policy のような統合されたコンプライアンス管理ができません。

### D Azure Security Center

Azure Security Center（現在の Defender for Cloud）はセキュリティの推奨事項を提供するサービスです。セキュリティ状態を可視化するには優れていますが、ポリシーベースで構成を自動修復する主要な仕組みではありません。

---

## 7. 最終回答

A. Azure Policy イニシアチブで、TDE 用の組み込みポリシーと DeployIfNotExists エフェクトを使用する

---

## 8. まとめ

この問題の核心は **Azure 環境全体のセキュリティ設定を自動的に強制するガバナンス機能**を理解することです。Azure Policy はクラウドガバナンスの中心的なサービスであり、組織のセキュリティポリシーをコードとして定義できます。

DeployIfNotExists エフェクトを使用すると、非準拠リソースを検出するだけでなく自動的に修復できます。この仕組みにより、企業はクラウド環境全体のセキュリティ設定を一貫して維持できます。

特に Azure SQL Database のような重要なデータサービスでは、TDE のような暗号化機能をポリシーによって強制することが、セキュリティとコンプライアンスの両面で非常に重要になります。
