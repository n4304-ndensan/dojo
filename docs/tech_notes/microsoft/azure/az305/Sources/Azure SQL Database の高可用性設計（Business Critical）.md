# Azure SQL Database の高可用性設計（Business Critical）

## 1. 背景（シナリオ）

ある企業では、Azure SQL Database を使用した高可用性データベースソリューションを設計する必要があります。このデータベースはミッションクリティカルなワークロードを扱っており、システム停止やデータ損失は許容されません。

そのため、データベースは障害発生時にも継続して利用できる必要があります。特に、Azureの可用性ゾーンの障害が発生した場合でもデータベースが利用可能であることが求められています。

さらに、フェイルオーバーが発生する場合でもデータ損失が発生してはいけません。このような要件を満たしつつ、可能な限りコストを抑えたサービス層を選択する必要があります。

---

## 2. 要件整理

この問題では、Azure SQL Databaseのサービス層を選択することが求められています。シナリオから読み取れる重要な要件を整理すると次のようになります。

まず、フェイルオーバー時にデータ損失が発生しないことが必要です。これは同期レプリケーションが必要であることを意味します。

次に、可用性ゾーンの障害が発生してもデータベースが継続して利用可能である必要があります。つまり、ゾーン冗長構成が必要になります。

さらに、これらの要件を満たしながらコストを可能な限り抑える必要があります。

要件をまとめると次の通りです。

- フェイルオーバー時のデータ損失ゼロ  
- ゾーン障害への耐性  
- 高可用性  
- コスト効率  

---

## 3. 技術の基本概念

Azure SQL Databaseには複数のサービス層が存在します。それぞれの層はパフォーマンス、可用性、コストのバランスが異なります。

主なサービス層には次のようなものがあります。

Basic / Standard  
開発環境や小規模ワークロード向けであり、高度な可用性機能は提供されません。

Serverless  
使用量に応じて自動スケーリングするコスト最適化モデルですが、ミッションクリティカルな高可用性ワークロードには適していません。

Business Critical  
ミッションクリティカルなアプリケーション向けに設計されたサービス層であり、Always On可用性グループ技術を使用して高可用性を提供します。

この層では複数のレプリカが同期的に保持されるため、フェイルオーバー時にもデータ損失が発生しません。

---

## 4. アーキテクチャまたは設計のポイント

Business Criticalサービス層では、データベースは複数のレプリカで構成されます。

この構成では次の仕組みが採用されています。

- プライマリレプリカ  
- 複数のセカンダリレプリカ  
- 同期レプリケーション  
- 自動フェイルオーバー  

これらのレプリカは可用性ゾーンに分散配置することができます。

この構成により、次のようなメリットが得られます。

- ゾーン障害時のサービス継続  
- 自動フェイルオーバー  
- データ損失ゼロのフェイルオーバー  
- 高いパフォーマンス  

そのため、ミッションクリティカルなデータベースに適しています。

---

## 5. 設計判断（なぜこの構成になるか）

この問題の最大のポイントは「データ損失なしのフェイルオーバー」です。

これは同期レプリケーションが必要であることを意味します。Azure SQL Databaseの中で、この機能をネイティブに提供するのがBusiness Critical層です。

また、ゾーン冗長構成にも対応しているため、可用性ゾーン障害にも耐えることができます。

さらに、Premium層よりもコスト効率が良く、必要な高可用性機能を提供します。

そのため、以下の要件をすべて満たします。

- 同期レプリケーション  
- 自動フェイルオーバー  
- ゾーン冗長構成  
- コスト効率  

---

## 6. 他の選択肢が誤りな理由

Serverlessはコスト最適化を目的としたモデルであり、ミッションクリティカルな高可用性構成には適していません。

Premiumは高性能なサービス層ですが、この問題のシナリオではBusiness Critical層の方が高可用性機能とコスト効率のバランスが適しています。

Basicは開発やテスト用途向けの低価格サービス層であり、高可用性やゾーン冗長機能は提供していません。

---

## 7. 最終回答

B.  
Azure SQL Database **Business Critical**

---

## 8. まとめ

Azure SQL Databaseでミッションクリティカルなワークロードを運用する場合、高可用性設計が重要になります。

Business Critical層は次の特徴を持っています。

- Always On可用性グループ  
- 同期レプリケーション  
- 自動フェイルオーバー  
- ゾーン冗長構成  

これにより、フェイルオーバー時のデータ損失を防ぎながら高可用性を実現できます。

そのため、データ損失ゼロのフェイルオーバーとゾーン耐障害性を必要とするシナリオでは **Azure SQL Database Business Critical** が最適なサービス層です。
