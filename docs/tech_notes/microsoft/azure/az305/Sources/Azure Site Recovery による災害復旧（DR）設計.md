# Azure Site Recovery による災害復旧（DR）設計

## 1. 背景（シナリオ）

ある組織では、Azure 仮想マシン上で稼働する重要な多層アプリケーションを運用しています。このアプリケーションは事業運営の中核を担っており、システム停止が長時間発生すると大きなビジネス損失につながります。

そのため、組織では厳格な災害復旧（Disaster Recovery, DR）ポリシーを定義しています。

具体的には次のような復旧目標が設定されています。

- **RTO（Recovery Time Objective）: 10分以内**
- **RPO（Recovery Point Objective）: 5分以内**

さらに、プライマリリージョンで障害が発生した場合には、アプリケーションを別リージョンに切り替えて運用を継続する必要があります。

このような要件を満たすためには、Azure の災害復旧サービスを適切に構成する必要があります。

---

## 2. 要件整理

この問題では、災害復旧アーキテクチャの設計において重要な要件を理解する必要があります。

まず、復旧目標に関する要件です。

- **RTO（復旧時間）: 10分以内**
- **RPO（データ損失許容時間）: 5分以内**

次に、システム構成に関する要件です。

- Azure VM 上の多層アプリケーション
- 地理的なリージョン間冗長
- 自動フェールオーバー
- 災害時の迅速な復旧

このことから、**VM レベルでリージョン間レプリケーションとフェールオーバーを提供するサービス**が必要になります。

---

## 3. 技術の基本概念

災害復旧アーキテクチャでは、次の2つの指標が重要になります。

まず **RTO（Recovery Time Objective）** です。

RTO は、システム障害発生後にサービスを復旧させるまでの許容時間を意味します。

次に **RPO（Recovery Point Objective）** です。

RPO は、どの程度のデータ損失を許容できるかを示す指標であり、通常は時間で表されます。

これらの目標を満たすために Azure では **Azure Site Recovery（ASR）** というサービスが提供されています。

Azure Site Recovery は、Azure VM やオンプレミス VM を別リージョンへレプリケーションし、災害発生時にフェールオーバーできるサービスです。

主な機能は次の通りです。

- VM の継続的レプリケーション
- リージョン間フェールオーバー
- 自動フェールオーバー
- 災害復旧テスト
- オーケストレーション（復旧順序）

---

## 4. アーキテクチャまたは設計のポイント

Azure Site Recovery を使用すると、次のような災害復旧アーキテクチャを構築できます。

まず、プライマリリージョンの Azure VM を Site Recovery に登録します。

次に、VM データをセカンダリリージョンへ継続的にレプリケーションします。

この構成の特徴を整理すると次の通りです。

- VM データの継続的レプリケーション
- リージョン間の DR 構成
- フェールオーバー自動化
- 最小限のダウンタイム

この仕組みにより、プライマリリージョン障害時でもアプリケーションをセカンダリリージョンで迅速に再起動できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題の要件は **厳格な RTO / RPO とリージョン間フェールオーバー**です。

Azure Site Recovery は次の理由で最適なサービスです。

まず、レプリケーション機能です。

- 継続的データレプリケーション
- 数分レベルの RPO

次に、フェールオーバー機能です。

- 自動フェールオーバー
- リカバリープランによる復旧手順管理

さらに、DR テスト機能も提供されます。

これにより、アプリケーション停止時間を最小化しながら災害復旧を実現できます。

---

## 6. 他の選択肢が誤りな理由

まず **Azure Traffic Manager** です。

Traffic Manager は DNS ベースのトラフィックルーティングサービスであり、リージョン間の負荷分散は可能ですが、VM のレプリケーションや災害復旧機能は提供しません。

次に **Azure SQL Database Geo-replication** です。

これはデータベースのレプリケーション機能であり、アプリケーション全体の DR ソリューションではありません。

今回のシナリオでは **VM 上の多層アプリケーション全体**の復旧が必要です。

最後に **Azure Load Balancer** です。

Load Balancer は VM 間のトラフィック分散を行うサービスであり、災害復旧機能は提供しません。

---

## 7. 最終回答

**A. Azure Site Recovery**

---

## 8. まとめ

Azure における災害復旧設計では、サービスの役割を正しく理解することが重要です。

主なサービスの役割は次の通りです。

まず、災害復旧サービスです。

- **Azure Site Recovery**
  → VM のリージョン間レプリケーションと DR

次に、トラフィック制御です。

- **Azure Traffic Manager**
  → DNS ベースのルーティング

さらに、負荷分散です。

- **Azure Load Balancer**
  → VM 間のトラフィック分散

このため、**VM ベースの多層アプリケーションに対して厳格な RTO / RPO を満たす DR を実現する場合は Azure Site Recovery を使用する**のが最適な設計になります。
