# Azure Storage におけるきめ細かなアクセス制御（Azure RBAC）

## 1. 背景（シナリオ）

ある組織では、大規模なAzureストレージ環境を運用しています。このストレージ環境には複数のチームがアクセスする必要がありますが、すべてのチームが同じデータにアクセスできるわけではありません。

例えば、あるチームは特定のコンテナのみ読み取りアクセスが必要であり、別のチームは別のコンテナに書き込みアクセスが必要になる場合があります。このような状況では、ストレージアカウント全体にアクセス権を与えるのではなく、**必要なリソースだけにアクセスできるように制御する仕組み**が重要になります。

さらに、この環境では組織のセキュリティコンプライアンスを満たす必要があります。そのため、アクセス権限は最小権限の原則に従い、各チームに必要最小限の権限だけを付与する必要があります。

## 2. 要件整理

この問題では、Azure Storage のアクセス制御をどのように設計するかがポイントになります。

シナリオから読み取れる要件は以下の通りです。

- 複数のチームが同一ストレージ環境を利用する
- チームごとにアクセス可能なコンテナやブロックが異なる
- きめ細かなアクセス制御が必要
- セキュリティコンプライアンスを満たす必要がある
- 管理がスケーラブルである必要がある

これらの要件を満たすためには、Azureの**ロールベースアクセス制御（RBAC）**を利用するのが最適です。

## 3. 技術の基本概念

Azureでは、リソースへのアクセス制御を行うために **Azure RBAC（Role-Based Access Control）** が提供されています。

RBACは、ユーザー・グループ・アプリケーションに対して、特定のロールを割り当てることでアクセス権を管理する仕組みです。

RBACは次の3つの要素で構成されます。

- Security Principal（ユーザー・グループ・サービスプリンシパル）
- Role Definition（権限の集合）
- Scope（権限の適用範囲）

Scopeは階層構造になっており、以下のレベルで権限を設定できます。

- Management Group
- Subscription
- Resource Group
- Resource
- Resource Sub-resource（例: コンテナ）

この階層構造により、非常に細かいアクセス制御が可能になります。

## 4. アーキテクチャまたは設計のポイント

Azure Storageでチームごとのアクセス制御を実装する場合、RBACを利用して**カスタムロール**を作成することが有効です。

まず、チームごとのアクセス要件を整理します。例えば次のようなケースがあります。

- Team A は Container A の読み取りのみ
- Team B は Container B の読み書き
- Team C は Container C の読み取りと削除

このような要件に対応するために、RBACではカスタムロールを作成できます。

カスタムロールでは、次のような権限を定義できます。

- Blob 読み取り
- Blob 書き込み
- コンテナ作成
- データ削除

これらの権限を組み合わせてロールを作成し、それをAzure ADグループに割り当てることで、チーム単位のアクセス管理が可能になります。

この方法により、組織は最小権限の原則を維持しながら安全にストレージアクセスを管理できます。

## 5. 設計判断（なぜこの構成になるか）

このシナリオでは、**コンテナレベルやBlobレベルでの細かいアクセス制御**が必要です。

Azure RBACを利用することで、以下のメリットがあります。

まず、きめ細かなアクセス制御が可能になります。RBACでは特定のコンテナやリソースに対して権限を割り当てることができます。

次に、セキュリティコンプライアンスを満たすことができます。最小権限の原則に従ってアクセスを管理できるため、不正アクセスのリスクを減らせます。

さらに、管理が簡単になります。Azure ADグループとRBACを組み合わせることで、チームメンバーの追加や削除が容易になります。

そのため、最も効果的な方法は **Azure RBACを利用してカスタムロールを作成すること**です。

## 6. 他の選択肢が誤りな理由

### B. 共有アクセス署名（SAS）

SASはストレージリソースへの一時的なアクセスを提供する仕組みです。

しかし、大規模環境では以下の問題があります。

- トークン管理が複雑
- トークン漏洩リスク
- 長期的なアクセス管理に不向き

そのため、組織のアクセス管理には適していません。

### C. Azure Policy

Azure Policyは主に **ガバナンスとコンプライアンスの強制**に使用されます。

例えば、以下のような制御を行います。

- 特定リージョンの利用制限
- 必須タグの強制
- リソース構成制御

しかし、ユーザー単位のアクセス権管理には使用できません。

### D. Access Keys とファイアウォール

Access Keysはストレージアカウントへの**完全なアクセス権**を提供します。

そのため、キーを持つユーザーはストレージアカウント内のすべてのリソースにアクセスできます。

これは最小権限の原則に反するため、このシナリオには適していません。

## 7. 最終回答

このシナリオで最も適切なアクセス制御戦略は次です。

**Azure RBAC を使用して、各チームのストレージリソースに対してカスタムロールを作成する**

したがって **正解は A** です。

## 8. まとめ

Azure Storage のアクセス制御では、適切なガバナンスとセキュリティを維持することが重要です。

特に大規模環境では、チームごとに異なるアクセス権が必要になります。

このような場合、Azure RBAC を利用することで次のようなメリットがあります。

- コンテナレベルの細かいアクセス制御
- 最小権限の原則の実現
- Azure ADグループとの統合
- 管理の簡素化

そのため、Azure Storageのアクセス管理では **RBACによるロールベースアクセス制御が最も推奨される方法**になります。
