# Azure Storage アクセス制御と監査を実現する RBAC

## 1. 背景（シナリオ）

ある組織では、複数の **Azure Storage アカウント**を利用しており、それぞれが異なる部門のサービスを支えています。  
各部門は、自分たちのストレージリソースに対して誰がアクセスできるか、そしてどのような操作が許可されるのかについて独自の要件を持っています。

そのため、組織としては以下のような課題を解決する必要があります。

- 各部門ごとにアクセス権限を細かく制御する
- 不正アクセスを防止する
- 誰がいつどのリソースにアクセスしたかを追跡できる
- コンプライアンス要件を満たす監査ログを確保する

このような環境では、単なるデータ保護だけでなく **アクセス制御と監査を両立する仕組み**が必要になります。

---

## 2. 要件整理

問題文から読み取れる重要な要件を整理すると、以下の通りです。

まず、この問題のポイントは「アクセス制御」と「監査」の両方です。

その具体的な要件は次の通りです。

- Azure Storage アカウントへのアクセス管理
- 部門ごとに異なるアクセス権限
- 操作レベルの権限制御
- リソース操作の監査ログ
- セキュリティとコンプライアンス対応

これらの要件を満たすには、Azureの **IDベースアクセス管理機能**を利用する必要があります。

---

## 3. 技術の基本概念

Azureでは、アクセス管理を実現するために複数のセキュリティ機能が提供されています。その中でも最も重要なのが **Azure Role-Based Access Control (RBAC)** です。

### Azure RBAC（Role-Based Access Control）

Azure RBAC は、Azure リソースへのアクセスを **ロール（役割）ベースで管理する仕組み**です。

RBACでは、ユーザーやグループ、アプリケーションに対してロールを割り当てることで、実行可能な操作を制御します。

RBACの特徴として、次のようなポイントがあります。

- Azureリソース単位でアクセス制御が可能
- 最小権限の原則を適用できる
- 組み込みロールとカスタムロールの利用
- Azure Active Directoryと統合
- Azure Activity Log と連携した監査

これにより、組織内のユーザーや部門ごとに細かな権限設定が可能になります。

---

## 4. アーキテクチャまたは設計のポイント

このシナリオでは、複数の部門がそれぞれ異なるストレージリソースを使用しています。  
そのため、リソースごとに権限を分離する設計が必要になります。

Azure RBAC を使用すると、以下のような設計が可能になります。

まず、アクセス管理の基本構造として、以下の3要素が存在します。

- セキュリティプリンシパル（ユーザー、グループ、サービスプリンシパル）
- ロール定義
- スコープ（リソース範囲）

この仕組みを利用することで、たとえば以下のような運用が可能になります。

- 財務部門には特定ストレージの読み取り権限のみ付与
- 開発部門にはBlob書き込み権限を付与
- 管理者のみ削除操作を許可

このように **部門単位でアクセス制御を実装できる点がRBACの大きな特徴です。**

---

## 5. 設計判断（なぜこの構成になるか）

この問題では「きめ細かいアクセス制御」と「監査」が重要です。

Azure RBAC は次の理由で最適な選択となります。

まず、RBACは **最小権限の原則**を実現できます。  
これは、ユーザーに必要最小限の権限だけを与えることで、セキュリティリスクを最小化する考え方です。

次に、RBACは **Azure Activity Log と統合**されています。

この仕組みにより、次のような情報を追跡できます。

- 誰が
- どのリソースに
- いつ
- どの操作を行ったか

このログはコンプライアンスやセキュリティ監査において非常に重要です。

さらに、RBACは **Azure Active Directory と統合されているため、組織全体のID管理と連携できます。**

---

## 6. 他の選択肢が誤りな理由

この問題では、他のAzure機能がなぜ適していないかを理解することも重要です。

### B Azure Storage Service Encryption

この機能は、ストレージ内のデータを **保存時に暗号化する機能**です。

しかし、この機能は次の目的のためのものです。

- データ保護
- 保存時暗号化

そのため、次の機能は提供されません。

- ユーザーアクセス制御
- 操作権限管理
- 監査ログ管理

したがって、この問題の要件を満たしません。

---

### C Azure Managed Identity

Managed Identity は、Azure サービスが他の Azure サービスへ **安全に認証するための仕組み**です。

主な用途は以下です。

- アプリケーション認証
- シークレット管理の削減

しかし、この機能は **ユーザーのアクセス制御管理のための仕組みではありません。**

---

### D Azure Policy

Azure Policy は **リソースの構成ルールを強制するためのガバナンス機能**です。

例えば次のようなポリシーを設定できます。

- 特定リージョンのみ使用
- 特定タグの必須化
- 特定SKUの制限

しかし、Azure Policyは次の機能を提供しません。

- ユーザー権限管理
- 操作レベルのアクセス制御

したがって、このシナリオには適していません。

---

## 7. 最終回答

A. **Azure ロールベースアクセス制御 (RBAC)**

---

## 8. まとめ

Azure Storage のアクセス管理では、**Azure RBAC** が中心的な役割を担います。

RBACを利用することで、次のようなセキュリティ設計を実現できます。

- 部門ごとのアクセス制御
- 最小権限の原則の適用
- Azure ADとの統合
- リソース操作の監査ログ
- セキュリティおよびコンプライアンス対応

そのため、複数部門が利用するストレージリソースのアクセス管理では **Azure RBAC が最も適したソリューション**となります。
