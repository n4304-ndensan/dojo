# Azure Traffic Manager によるグローバルトラフィックルーティング

## 1. 背景（シナリオ）

ある組織では、複数の Azure リージョンに分散された Web アプリケーションを運用しています。このアプリケーションは世界中のユーザーからアクセスされるため、ユーザー体験を向上させるために遅延を最小限に抑える必要があります。

また、単一リージョンの障害が発生した場合でもアプリケーションが停止しないように、高可用性と障害耐性を確保する必要があります。そのため、ユーザーリクエストを適切なリージョンのアプリケーションインスタンスへルーティングする仕組みが重要になります。

さらに、トラフィックルーティングではユーザーの地理的位置だけでなく、アプリケーションインスタンスの健全性（ヘルス）も考慮する必要があります。これにより、障害が発生したインスタンスへのアクセスを防ぐことができます。

## 2. 要件整理

このシナリオでは、グローバルに分散された Web アプリケーションのトラフィック管理がテーマです。要件を整理すると、次のようなポイントが挙げられます。

まず、ユーザーの遅延を最小化する必要があります。そのためには、ユーザーに最も近いリージョンへリクエストをルーティングすることが重要です。

次に、アプリケーションの高可用性を確保する必要があります。もし特定のリージョンのインスタンスが障害を起こした場合には、自動的に別の健全なインスタンスへトラフィックを転送する必要があります。

さらに、トラフィックルーティングはインスタンスの健全性も考慮しなければなりません。つまり、障害のあるインスタンスへユーザーを誘導してはいけません。

このシナリオの要件をまとめると次の通りです。

- ユーザー遅延の最小化  
- 複数リージョンでの高可用性  
- インスタンスの健全性チェック  
- 自動フェイルオーバー  

## 3. 技術の基本概念

Azure Traffic Manager は DNS ベースのグローバルロードバランシングサービスです。ユーザーの DNS クエリに対して最適なエンドポイントを返すことで、トラフィックを複数のリージョンに分散します。

Traffic Manager には複数のルーティング方式があり、それぞれ異なる目的で使用されます。

まず、**地理ルーティング（Geographic Routing）** はユーザーの地域に基づいてトラフィックをルーティングする方式です。これにより、ユーザーを最も近いリージョンのサービスに誘導できます。

次に、**ヘルスプローブ（Health Probe）** はアプリケーションエンドポイントの状態を監視する機能です。もしエンドポイントが正常に応答しない場合、そのエンドポイントは自動的にトラフィック対象から除外されます。

## 4. アーキテクチャまたは設計のポイント

Azure Traffic Manager を使用したグローバルアーキテクチャでは、複数リージョンのアプリケーションエンドポイントを Traffic Manager プロファイルに登録します。

次に、Traffic Manager はユーザーの DNS リクエストを受け取ると、最適なエンドポイントを選択します。

この構成では次のような仕組みが実装されます。

- Traffic Manager が DNS クエリを受信する  
- ユーザーの地理的場所を判定する  
- 最も近いリージョンのエンドポイントを選択する  
- ヘルスプローブでエンドポイントの健全性を確認する  
- 健全なインスタンスのみにトラフィックを送る  

もし最も近いリージョンのインスタンスが障害を起こした場合、Traffic Manager は次に利用可能な健全なリージョンへトラフィックをルーティングします。

## 5. 設計判断（なぜこの構成になるか）

このシナリオでは、ユーザーの遅延を最小化することが重要です。そのため、ユーザーの地理的位置を考慮したルーティングが必要になります。

地理ルーティングを使用することで、ユーザーを最も近いリージョンのアプリケーションに誘導できます。これにより、ネットワーク遅延を最小化し、アプリケーションの応答速度を向上させることができます。

また、Traffic Manager のヘルスプローブ機能を利用することで、各リージョンのアプリケーションインスタンスの状態を監視できます。もしインスタンスが不健全になった場合、そのインスタンスは自動的にトラフィック対象から除外されます。

この組み合わせにより、低遅延と高可用性の両方を実現できます。

## 6. 他の選択肢が誤りな理由

パフォーマンスベースのルーティングは応答時間を基準にルーティングを行いますが、ユーザーの地理的位置を直接考慮するわけではありません。そのため、地域ベースの最適ルーティングには適していません。

重み付きラウンドロビンはトラフィックを均等に分散する方法ですが、ユーザーの近接性を考慮しないため遅延が増加する可能性があります。

マルチバリュールーティングは複数のエンドポイントを返す方式であり、DNS レベルでの単純な分散に使用されますが、地理的最適化や高度なルーティングには適していません。

## 7. 最終回答

A. 地理ルーティングを健康プローブと組み合わせて使用する

## 8. まとめ

グローバルに分散された Web アプリケーションでは、ユーザー遅延の最小化と高可用性の確保が重要になります。

Azure Traffic Manager の地理ルーティングを使用することで、ユーザーを最も近いリージョンのアプリケーションへ誘導できます。また、ヘルスプローブを利用することで、健全なインスタンスのみにトラフィックを送ることができます。

この構成により、低遅延・高可用性・自動フェイルオーバーを実現できるため、複数リージョンで動作する Web アプリケーションに最適なトラフィック管理方式となります。
