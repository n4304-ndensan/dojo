# Azure VNet ピアリング（Global Reach / Global VNet Peering）

## 1. 背景（シナリオ）

あるグローバル組織では、複数の地理的リージョンに Azure 仮想ネットワーク（VNet）を展開しており、各リージョンに配置されたシステム同士が安全に通信できるネットワークアーキテクチャを設計する必要があります。

この組織の環境では、以下のような課題が存在します。

- 複数リージョンの VNet 間でアプリケーション通信が必要  
- 機密データを扱うため、ネットワーク間の厳格な分離を維持する必要がある  
- 高可用性と災害復旧を考慮した構成が求められる  
- 地理的に離れた拠点間でも低遅延で通信できる必要がある  

このような要件を満たすためには、Azure 内の仮想ネットワーク同士を安全かつ高速に接続するネットワーク機能を選択する必要があります。

---

## 2. 要件整理

この問題の要件を整理すると、以下のポイントが重要になります。

まず、システム設計の観点から必要となる条件を明確にします。

- 異なるリージョンの **Azure VNet 同士を接続できること**
- **安全な通信（Azureバックボーン上の通信）**
- **高帯域・低遅延**
- **ネットワーク隔離を維持できること**
- **高可用性と災害復旧に対応可能**
- **スケーラブルなネットワーク構成**

このような条件を満たす Azure ネットワーク機能を選択する必要があります。

---

## 3. 技術の基本概念

Azure では、VNet 同士を接続するために **VNet Peering（VNet ピアリング）** という機能が提供されています。

VNet ピアリングとは、2つの仮想ネットワークを Azure のバックボーンネットワークを通じて直接接続する機能です。これにより、異なる VNet に存在するリソース同士が **同一ネットワーク内のように通信**できるようになります。

VNet ピアリングには次の特徴があります。

VNet ピアリングの特徴を理解するために、重要なポイントを整理します。

- Azure バックボーンネットワークを使用
- インターネットを経由しない
- 低遅延・高帯域
- VPN ゲートウェイ不要
- ルーティングがシンプル
- セキュリティグループ（NSG）による制御が可能

さらに、**Global VNet Peering** を利用することで、異なる Azure リージョン間でも同様の接続が可能になります。

---

## 4. アーキテクチャまたは設計のポイント

グローバル組織のネットワーク設計では、複数リージョンの VNet を効率よく接続しながら、セキュリティと可用性を確保することが重要になります。

VNet Peering（特に Global VNet Peering）を利用すると、次のようなアーキテクチャを構築できます。

まず、リージョンごとに VNet を配置します。  
その後、必要な VNet 同士のみをピアリング接続します。

この設計のポイントは以下の通りです。

- **リージョン間通信を Azure バックボーンで実現**
- **必要なネットワークのみ接続することで隔離を維持**
- **VPN ゲートウェイ不要でシンプルな構成**
- **高帯域・低遅延通信**
- **リージョン障害時の DR 構成を構築可能**

この仕組みにより、グローバル規模のクラウドネットワークでも高いパフォーマンスとセキュリティを両立できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題では、以下の理由から **Azure VNet Peering（Global VNet Peering）** が最適な選択になります。

まず、VNet ピアリングは Azure 内部ネットワークを利用するため、VPN と比較して次の利点があります。

- 低遅延
- 高帯域
- ネットワーク構成がシンプル
- 管理負荷が低い

さらに、Global VNet Peering を使用すると、異なるリージョンの VNet も接続できます。

また、ピアリングは **特定の VNet 間のみ通信可能**であるため、必要なネットワークだけを接続し、それ以外は分離することが可能です。

これにより

- セキュリティ
- スケーラビリティ
- 高可用性

のバランスが取れたネットワークアーキテクチャを実現できます。

---

## 6. 他の選択肢が誤りな理由

この問題では、他の選択肢も Azure ネットワーク機能ですが、要件に対して最適とは言えません。

まず **Azure Virtual WAN** について考えます。  
Virtual WAN は拠点接続や大規模ネットワークを集中管理するためのサービスであり、以下の用途に適しています。

- 多数の支社接続
- SD-WAN
- VPN / ExpressRoute 集約

しかし、この問題は **VNet 間接続が主目的**であるため、Virtual WAN はやや過剰なソリューションになります。

次に **Site-to-Site VPN** です。

Site-to-Site VPN は主に

- オンプレミス ↔ Azure
- 拠点間 VPN

で使用されます。

しかし VPN は

- インターネット経由
- 遅延が増える
- ゲートウェイ管理が必要

という点で、VNet Peering より効率が劣ります。

最後に **Azure Private Link** です。

Private Link は

- Azure PaaS サービスへのプライベートアクセス

を実現する機能であり、

- VNet → Azureサービス

の接続には適していますが、

- **VNet → VNet 接続には使用できません**

したがって、このシナリオには適合しません。

---

## 7. 最終回答

**A. Azure VNet ピアリング（Global VNet Peering / Global Reach）**

---

## 8. まとめ

この問題は、Azure ネットワーク設計において **VNet 接続方式の選択**を理解しているかを問う典型的な問題です。

特に重要なポイントは次の通りです。

Azure ネットワークサービスの役割を整理すると理解しやすくなります。

- **VNet Peering**  
  → VNet 同士を高速接続（最も基本）

- **Global VNet Peering**  
  → リージョン間 VNet 接続

- **Virtual WAN**  
  → 大規模拠点ネットワーク管理

- **Site-to-Site VPN**  
  → オンプレミス接続

- **Private Link**  
  → Azure PaaS へのプライベート接続

したがって、**複数リージョンの VNet を安全かつ高速に接続する場合は VNet Peering が最適解**となります。
