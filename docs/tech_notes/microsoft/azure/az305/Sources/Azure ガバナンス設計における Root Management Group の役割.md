# Azure ガバナンス設計における Root Management Group の役割

## 1. 背景（シナリオ）

ある多国籍企業では、Azure を利用して複数の地域にわたる IT インフラを運用しています。この組織は非常に大規模な Azure 環境を持っており、Azure の管理階層は複雑な構造になっています。

具体的には、Azure 環境は複数の管理グループ階層によって構成されています。組織には 10 個のルート管理グループがあり、その配下には 15 個の子管理グループが存在しています。さらに、それぞれの管理グループには複数のサブスクリプションが含まれており、その中で多くのリソースグループが管理されています。

このような大規模環境では、企業全体のコンプライアンスやガバナンスを統一的に管理する必要があります。そのため、ポリシーが階層全体に適用され、新しいサブスクリプションやリソースが追加された場合でも自動的にポリシーが継承される仕組みが必要になります。

## 2. 要件整理

この問題では、Azure 環境全体に対して効果的なポリシー管理を行うための最適な階層レベルを選択する必要があります。シナリオから読み取れる要件を整理することで、適切な回答を導くことができます。

まず、組織は複数の地域で事業を展開しているため、地域ごとの規制に対応する必要があります。しかし同時に、組織全体として共通のコンプライアンスルールを適用する必要があります。

さらに、新しいサブスクリプションやリソースグループが作成された場合でも、自動的にポリシーが適用される必要があります。

このシナリオの要件を整理すると次のようになります。

組織全体に共通するポリシーの適用  
新しいリソースへの自動ポリシー継承  
地域ごとの例外設定の柔軟性  
階層的なガバナンス管理  

これらの要件を満たすためには、Azure の階層構造の最上位でポリシーを管理する必要があります。

## 3. 技術の基本概念

Azure には、リソースを階層的に管理するための **Management Group（管理グループ）** という仕組みがあります。この機能により、大規模な Azure 環境でも統一的なガバナンスを実現できます。

Azure の階層構造は一般的に次の順序で構成されます。

まず最上位には **Root Management Group** が存在します。その下に複数の Management Group が配置されます。次に、その配下に Azure サブスクリプションが配置されます。そしてサブスクリプションの中にリソースグループとリソースが配置されます。

この階層構造の特徴として、上位レベルで設定したポリシーは下位レベルに自動的に継承されます。

## 4. アーキテクチャまたは設計のポイント

Azure ガバナンスを設計する際には、どの階層レベルでポリシーを適用するかが重要なポイントになります。

Root Management Group でポリシーを定義すると、そのポリシーはすべての子管理グループ、サブスクリプション、リソースグループに自動的に継承されます。

この仕組みによって、組織全体に対して統一されたガバナンスを実装できます。

一方で、地域ごとに異なる規制がある場合には、子管理グループのレベルで追加のポリシーを設定したり、ポリシーの例外を定義したりすることも可能です。

このように、Root Management Group を基盤として利用することで、中央集権型のガバナンスと柔軟な地域対応の両方を実現できます。

## 5. 設計判断（なぜこの構成になるか）

Root Management Group を使用する最大のメリットは、ポリシー管理を中央集権化できる点です。

組織全体のセキュリティやコンプライアンスルールを Root レベルで定義することで、すべての Azure リソースに対して自動的に適用することができます。

また、新しいサブスクリプションが追加された場合でも、Root レベルのポリシーが自動的に継承されるため、追加設定を行う必要がありません。

さらに、管理グループ階層を利用することで、地域ごとの要件や事業部門ごとの要件に応じて柔軟なポリシー管理を行うことができます。

## 6. 他の選択肢が誤りな理由

### B 子管理グループ（Child Management Group）

子管理グループでもポリシーを適用することは可能ですが、そのポリシーはその管理グループ配下にのみ適用されます。そのため、組織全体のガバナンスを統一するには適していません。

### C サブスクリプション

サブスクリプションレベルでポリシーを設定すると、そのサブスクリプションのみに適用されます。複数のサブスクリプションを持つ大規模環境では管理が非常に複雑になります。

### D リソースグループ

リソースグループレベルは Azure 階層の中でも下位レベルです。そのため、このレベルでポリシーを管理すると、組織全体にポリシーを適用することができません。

## 7. 最終回答

A. **Root Management Group**

## 8. まとめ

Azure の大規模環境では、Management Group を利用した階層型ガバナンスが重要になります。

Root Management Group でポリシーを定義することで、そのポリシーはすべての管理グループ、サブスクリプション、リソースグループに自動的に継承されます。

この仕組みにより、組織全体に統一されたコンプライアンスルールを適用しながら、地域ごとの要件に応じた柔軟なポリシー管理を実現できます。
