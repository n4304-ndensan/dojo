# Azure マルチテナントデータベース設計

（テナントごとの物理分離 + 自動スケーリング）

---

# 1 背景

クラウドアプリケーションでは、複数の顧客（テナント）が同一アプリケーションを利用する **マルチテナントアーキテクチャ** が広く採用されている。SaaS（Software as a Service）モデルでは、アプリケーションロジックは共有しながらも、顧客ごとのデータを安全に管理する必要がある。

今回のシステムでは、複数の企業が同一アプリケーションを利用するため、各企業（テナント）のデータを安全に分離する必要がある。さらに、テナントごとに利用量が異なるため、データベースのリソースは利用状況に応じて柔軟にスケーリングできることが求められている。

具体的には次の要件がある。

- アプリケーションロジックは共有する
    
- テナントごとにデータベースを物理的に分離する
    
- 利用パターンに応じてデータベースリソースを自動的にスケーリングする
    
- 管理のオーバーヘッドを最小限にする
    

このような要件を満たす最適な Azure データベースアプローチは

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure SQL Database]]
**Azure SQL Database Elastic Pool（弾性プール）**

である。

---

# 2 マルチテナントアーキテクチャ

マルチテナントシステムでは、データの分離方法としていくつかのパターンが存在する。

|パターン|説明|
|---|---|
|共有データベース|すべてのテナントが同一DBを使用|
|スキーマ分離|テナントごとにスキーマを分ける|
|データベース分離|テナントごとにDBを作成|

今回の要件では **データベースレベルでの物理分離** が必要であるため、テナントごとに独立したデータベースを持つ方式が求められる。

この構成の概念図は次のようになる。

```text
Application Layer
        │
        ▼
Shared Application Logic
        │
        ▼
Elastic Pool
 ├─ Tenant DB 1
 ├─ Tenant DB 2
 ├─ Tenant DB 3
 └─ Tenant DB n
```

この構成では、アプリケーションは共通のコードベースを利用するが、各テナントのデータは独立したデータベースに保存される。

---

# 3 Azure SQL Database Elastic Pool

Azure SQL Database Elastic Pool は、複数のデータベースが **共有されたコンピュートリソースを利用する仕組み**を提供するサービスである。

通常、Azure SQL Database はデータベースごとにコンピューティングリソース（DTU または vCore）を割り当てる必要がある。しかし、マルチテナントシステムでは各テナントの利用状況が異なるため、データベースごとに固定リソースを割り当てると効率が悪くなる。

Elastic Pool を利用すると、複数のデータベースが共通のリソースプールを共有するため、利用量の少ないテナントと利用量の多いテナントのバランスを自動的に調整できる。

主な特徴は次の通りである。

|機能|内容|
|---|---|
|共有リソース|複数DBでCPU・メモリ共有|
|自動スケーリング|利用状況に応じてリソース配分|
|物理分離|テナントごとに独立DB|
|管理簡素化|単一プール管理|

---

# 4 アーキテクチャ

Elastic Pool を利用したマルチテナント構成は次のようになる。

```text
Users
   │
   ▼
Application Service
   │
   ▼
Tenant Routing Logic
   │
   ▼
Azure SQL Elastic Pool
   │
   ├─ Database (Tenant A)
   ├─ Database (Tenant B)
   ├─ Database (Tenant C)
   └─ Database (Tenant N)
```

処理の流れは次の通りである。

1. ユーザーがアプリケーションにアクセス
    
2. アプリケーションがテナントIDを識別
    
3. 対応するテナントデータベースへ接続
    
4. データ処理を実行
    

この仕組みにより、テナントごとに完全に分離されたデータ管理が可能になる。

---

# 5 自動スケーリングの仕組み

Elastic Pool では、複数のデータベースが同じリソースプールを共有する。例えば、あるテナントの利用量が急増した場合でも、プール内の余剰リソースを自動的に利用できる。

```text
Elastic Pool Resources
        │
        ▼
 ┌───────────────┐
 │ Tenant DB 1   │
 │ Tenant DB 2   │
 │ Tenant DB 3   │
 │ Tenant DB N   │
 └───────────────┘
```

この仕組みにより次のメリットが得られる。

- 突発的な負荷増加への対応
    
- リソース利用効率の向上
    
- コスト削減

# 6 他の選択肢が適切でない理由

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Cosmos DB]]
Cosmos DB を単一データベースでパーティション分割する方法では、テナントごとの論理分離は可能であるが、物理的に異なるデータベースに分離する要件を満たさない。パーティションキーによる分離では、同一データベース内にデータが保存されるため、強い分離が求められるシナリオには適さない。

テナントごとに Azure SQL Database を個別に作成する方法では物理分離は実現できるが、各データベースのリソース管理やスケーリングを個別に行う必要があり、管理負荷が高くなる。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Virtual Machines]]
Azure VM 上の SQL Server を使用する場合、仮想マシンの管理、パッチ適用、バックアップ管理などの運用負担が増えるため、クラウドの PaaS のメリットが失われる。

PostgreSQL のスキーマ分離では、同一データベース内でスキーマを分けるため論理分離にとどまり、物理分離の要件を満たさない。

---

# 7 メリット

Elastic Pool を利用することで、マルチテナント SaaS システムにおいて重要なメリットを得ることができる。

まず、テナントごとに独立したデータベースを持つため、データ分離が強固であり、セキュリティやコンプライアンス要件を満たしやすい。さらに、リソースを共有することでコスト効率が向上し、データベースごとのリソース割り当てを個別に管理する必要がなくなる。

また、Azure が提供する PaaS データベースサービスであるため、バックアップ、パッチ適用、可用性管理などの運用作業を Azure に任せることができ、管理者の負担を軽減できる。

---

# 8 まとめ

今回の要件は次の通りである。

- テナントごとにデータを物理的に分離
    
- アプリケーションロジックは共有
    
- 利用パターンに応じた自動スケーリング
    
- 管理オーバーヘッドの最小化
    

これらの要件を満たす最適なアプローチは

**Azure SQL Database Elastic Pool**

である。

Elastic Pool を利用することで、テナントごとのデータベース分離を維持しながら、リソースの自動スケーリングと効率的な管理を実現できる。これは多くの SaaS アプリケーションで採用されている標準的なマルチテナントデータベース設計パターンである。
