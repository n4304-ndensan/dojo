# Azure 仮想マシンのディザスタリカバリ設計（Azure Site Recovery）

## 1. 背景（シナリオ）

ある企業では、Azure 仮想マシン上でアプリケーションを運用しています。このアプリケーションはビジネスに重要な役割を持っており、障害が発生した場合でもできるだけ早くサービスを復旧できる仕組みが必要です。

企業の IT チームは、ディザスタリカバリ（DR）計画を設計する必要があります。この計画では、仮想マシンがホストされている Azure リージョンが停止した場合でも、別のリージョンでサービスを復旧できるようにする必要があります。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#RTO]]
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#RPO]]
また、企業は復旧目標を明確に定義しています。復旧時間目標（RTO）は1時間以内、復旧ポイント目標（RPO）は15分以内です。つまり、障害が発生した場合でも、1時間以内にサービスを再開し、最大でも15分以内のデータ損失に抑える必要があります。

さらに、このシステムは非本番環境で使用されるため、通常運用時のコストはできるだけ低く抑える必要があります。つまり、常にセカンダリ環境をフル稼働させるような高コストな構成は避ける必要があります。

このような条件を満たすディザスタリカバリソリューションを Azure 上で設計する必要があります。

---

## 2. 要件整理

この問題では、ディザスタリカバリ設計に関する複数の重要な要件があります。まずはシステム設計に影響する条件を整理する必要があります。

最初に、復旧時間目標があります。これは障害発生後にサービスを復旧するまでの許容時間です。

このシナリオでは以下の目標が設定されています。

- RTO（Recovery Time Objective）  
  1時間以内

次に、データ損失の許容範囲を示す復旧ポイント目標があります。

- RPO（Recovery Point Objective）  
  15分以内

さらに、コストに関する要件があります。この環境は非本番用途であるため、通常時の運用コストをできるだけ低く抑える必要があります。

この条件を満たすためには、次のような特徴を持つソリューションが必要になります。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Virtual Machines]]
- VM データの継続的レプリケーション  
- セカンダリリージョンへのフェイルオーバー  
- 自動化された復旧プロセス  
- 通常時のコスト最適化  

これらの要件を満たす Azure のサービスを選択する必要があります。

## 3. 技術の基本概念

### Disaster Recovery（DR）

ディザスタリカバリとは、システム障害やデータセンター障害が発生した際に、サービスを迅速に復旧するための仕組みです。クラウド環境では、別のリージョンへシステムを複製することで DR を実現することが一般的です。

DR 設計では主に次の2つの指標が使用されます。

まず、RTO（Recovery Time Objective）があります。これはシステムを復旧するまでに許容される時間を表します。

次に、RPO（Recovery Point Objective）があります。これは復旧時に失われてもよいデータ量を時間で表したものです。

これらの目標を満たすためには、適切なレプリケーションとフェイルオーバーの仕組みが必要になります。

---

### Azure Site Recovery

Azure Site Recovery（ASR）は、Azure やオンプレミスのワークロードを別のリージョンへレプリケーションし、災害時にフェイルオーバーできるようにするサービスです。

このサービスはディザスタリカバリ専用に設計されており、仮想マシンやアプリケーションをセカンダリリージョンにレプリケートすることができます。

ASR の重要な特徴を理解するためには、その機能を整理する必要があります。

Azure Site Recovery は次のような機能を提供します。

- 仮想マシンの継続的レプリケーション  
- 自動フェイルオーバー  
- フェイルオーバーテスト  
- リカバリプランのオーケストレーション  

これにより、障害発生時に迅速な復旧を実現できます。

---

## 4. アーキテクチャまたは設計のポイント

Azure Site Recovery を使用すると、仮想マシンをセカンダリリージョンに非同期レプリケーションすることができます。このレプリケーションは数分単位で行われるため、RPO を小さく保つことが可能です。

通常運用時には、セカンダリリージョンでは仮想マシンは起動していません。レプリケーションデータのみが保存されています。そのため、アクティブアクティブ構成と比較してコストを大幅に抑えることができます。

障害が発生した場合には、Azure Site Recovery が自動的にセカンダリリージョンで仮想マシンを起動します。このプロセスは自動化されているため、迅速な復旧が可能です。

また、リカバリプランを使用すると、複数の仮想マシンの起動順序や依存関係を定義できます。これにより、複雑なアプリケーション構成でもスムーズな復旧が可能になります。

---

## 5. 設計判断（なぜこの構成になるか）

この問題では **RTO 1時間、RPO 15分、低コスト**という条件を同時に満たす必要があります。

Azure Site Recovery は VM レプリケーションとフェイルオーバーを自動化するため、短い RTO を実現できます。また、継続的レプリケーションにより RPO を 15 分以内に抑えることも可能です。

さらに、通常時はセカンダリリージョンで仮想マシンを実行しないため、コスト効率が高いというメリットがあります。

このような特徴により、Azure Site Recovery は Azure 仮想マシンのディザスタリカバリに最適なサービスとなります。

## 6. 他の選択肢が誤りな理由

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Backup]]
### Azure Backup + GRS

Azure Backup はデータ保護には適していますが、ディザスタリカバリ用の自動フェイルオーバー機能は提供しません。そのため、RTO 1時間という要件を満たすことが困難です。

### VM Snapshot

定期的なスナップショットはバックアップとしては有効ですが、セカンダリリージョンで仮想マシンを復旧するためには多くの手動作業が必要になります。迅速な復旧を保証することはできません。

### Azure Traffic Manager

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure DNS]]
Traffic Manager は DNS ベースのトラフィックルーティングサービスです。トラフィックのフェイルオーバーはできますが、仮想マシンのレプリケーションや復旧は行えません。
