# Azure 仮想マシンのバックアップ戦略（RPO と不変バックアップ）

## 1. 背景（シナリオ）

ある企業では、ビジネスクリティカルなアプリケーションを Azure 仮想マシン上で運用しています。このアプリケーションは企業の重要業務を支えているため、障害発生時には迅速にデータを復旧できるバックアップ戦略が必要です。

このシナリオでは特に厳しいデータ保護要件が設定されています。まず、リカバリーポイント目標（RPO）が 4 時間と定義されています。これは、システム障害が発生した場合でも最大 4 時間分のデータしか失われてはならないという意味です。

さらに、バックアップデータはセキュリティの観点から **90 日間の不変性（immutability）** を持つ必要があります。つまり、バックアップデータは管理者や攻撃者によって誤って削除されたり改ざんされたりしないように保護されていなければなりません。この要件は、ランサムウェア攻撃や内部不正などのリスクを防ぐために重要です。

したがって、今回のバックアップ戦略では、単にバックアップを作成するだけでなく、RPO 要件とバックアップデータの不変性を同時に満たす必要があります。

---

## 2. 要件整理

この問題を正しく理解するためには、まずバックアップ戦略に関する要件を整理する必要があります。問題文から読み取れる主な要件は、データ保護とセキュリティの2つの観点に分けることができます。

まず、バックアップ頻度に関する要件があります。アプリケーションのデータ損失を最小限に抑えるため、バックアップは比較的高い頻度で取得する必要があります。

- リカバリーポイント目標（RPO）が **4時間**

これは、バックアップまたは復旧ポイントが少なくとも 4 時間ごとに作成される必要があることを意味します。

次に、バックアップデータの保護に関する要件があります。バックアップデータは長期間保持されるだけでなく、削除や変更から保護される必要があります。

- バックアップデータは **90日間不変である必要がある**

この不変性は、管理者が誤って削除した場合だけでなく、悪意のある攻撃による削除からも保護するための重要なセキュリティ要件です。

これらの条件を満たすためには、Azure の標準的なバックアップサービスを利用しつつ、不変ストレージ機能を利用する必要があります。

---

## 3. 技術の基本概念

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Backup]]
### Azure Backup

Azure Backup は、Azure 上の仮想マシンやデータベースなどのリソースを保護するためのマネージドバックアップサービスです。Azure Backup を使用すると、バックアップのスケジュール、保持期間、復元ポイントなどを一元的に管理できます。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Recovery Services Vault]]
Azure Backup は Recovery Services Vault を使用してバックアップデータを保存します。この Vault はバックアップデータの管理、復元、監査ログなどを提供する重要なコンポーネントです。

Azure Backup の主な特徴には次のようなものがあります。

- 自動バックアップスケジュール  
- 中央管理されたバックアップポリシー  
- 長期保持ポリシー  
- セキュリティ機能（ソフト削除など）

これにより、企業はバックアップの運用を大幅に簡素化できます。

---

### Immutable Vault

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#WORM]]
Azure Backup では、Recovery Services Vault に対して **Immutable Vault（不変ボールト）** を構成することができます。この機能はバックアップデータを WORM（Write Once Read Many）形式で保存することを可能にします。

WORM モデルでは、バックアップデータが保存された後は一定期間変更や削除ができなくなります。これにより、管理者の誤操作や悪意のある攻撃によるバックアップ削除を防ぐことができます。

Immutable Vault の主な特徴を理解するためには、バックアップデータ保護の観点から重要なポイントを整理する必要があります。

- 指定された保持期間中はバックアップデータを削除できない  
- バックアップデータの改ざんを防止できる  
- ランサムウェア攻撃からバックアップを保護できる  

このような機能により、企業はバックアップデータを長期間安全に保持できます。

---

## 4. アーキテクチャまたは設計のポイント

今回のバックアップ戦略では、Azure Backup を使用して仮想マシンのバックアップを取得します。バックアップデータは Recovery Services Vault に保存され、その Vault に対して Immutable Vault 機能を有効化します。

バックアップポリシーでは、バックアップ頻度を RPO 要件に合わせて設定します。例えば 4 時間以内に復旧ポイントを取得できるようにスケジュールを構成します。

さらに、バックアップデータの保持期間を 90 日以上に設定し、Immutable Vault によってその期間中はバックアップが削除されないようにします。

この構成により、バックアップデータは Azure Backup によって定期的に取得され、Vault に安全に保存されます。もし誤ってバックアップを削除しようとしても、Vault の不変性によって削除は拒否されます。

この設計は、セキュリティとデータ保護の両方の要件を満たすバックアップアーキテクチャになります。

---

## 5. 設計判断（なぜこの構成になるか）

この問題の重要なポイントは **バックアップとディザスタリカバリの違い**を理解することです。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Virtual Machines]]
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Site Recovery]]
Azure Site Recovery はディザスタリカバリのためのサービスであり、VM のレプリケーションやフェイルオーバーを提供します。しかし Site Recovery はバックアップデータの長期保存や不変性を提供するサービスではありません。

一方、Azure Backup はデータ保護を目的としたサービスであり、長期保持ポリシーや不変バックアップなどの機能を提供します。Immutable Vault を使用することで、バックアップデータを削除や改ざんから保護できます。

今回のシナリオでは、バックアップデータの不変性が非常に重要な要件であるため、Azure Backup と Immutable Vault を組み合わせる構成が最適になります。

---

## 6. 他の選択肢が誤りな理由

### A Azure Backup とインスタントスナップショット

この構成では Azure Backup を使用しますが、不変ストレージの設定が含まれていません。そのためバックアップデータが削除される可能性があり、90 日間の不変性要件を満たせません。

### B Azure Site Recovery

Azure Site Recovery はディザスタリカバリのためのサービスです。VM のレプリケーションやフェイルオーバーは可能ですが、バックアップデータの不変性を保証する機能はありません。

### D 手動スナップショット

手動スナップショットはバックアップ管理としては適切ではありません。また週次スナップショットでは RPO が 4 時間という要件を満たすことができません。

## 7. 最終回答

C. リカバリーサービス用のバスト付き Azure Backup。不変の格納庫で構成されています。
