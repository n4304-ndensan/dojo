# Azure 大容量データ転送設計

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Blob Storage]]
（オンプレミス → Azure Blob Storage 50TB 移行）

---

# 1 背景

ある組織では、オンプレミス環境で運用しているファイルサーバーのデータを **Azure Blob Storage** に移行する必要がある。クラウド移行プロジェクトでは、既存の大量データを安全かつ迅速にクラウドへ転送することが重要な課題となる。

今回のシナリオでは次の条件がある。

- 移行対象データ量は **50TB**
    
- 移行期限は **1週間以内**
    
- インターネット回線の帯域幅が **制限されている**
    
- データ転送は **安全に実行する必要がある**
    

このような条件では、通常のインターネット経由のデータ転送では時間がかかりすぎる可能性がある。そのため、ネットワーク転送ではなく **物理デバイスを利用したオフラインデータ転送**を検討する必要がある。

これらの要件を満たす最適な Azure サービスは

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Data Box]]
**Azure Data Box**

である。

---

# 2 Azure Data Box

Azure Data Box は、大容量データを Azure に転送するための **物理データ転送サービス**である。Microsoft が提供する専用ハードウェアをオンプレミス環境に配送し、そのデバイスにデータをコピーした後、デバイスを Azure データセンターへ返送することでクラウドへデータをアップロードする仕組みである。

この方法は、ネットワーク帯域が制限されている環境や、大容量データの短期間移行が必要な場合に特に有効である。

Azure Data Box の主な特徴は次の通りである。

|機能|内容|
|---|---|
|大容量転送|最大数百TBの転送|
|高速コピー|ローカルネットワーク利用|
|オフライン転送|インターネット帯域不要|
|高セキュリティ|データ暗号化|

---

# 3 Azure Data Box の仕組み

Azure Data Box を利用したデータ転送の流れは次のようになる。

```text
On-Premises File Server
        │
        ▼
Azure Data Box Device
        │
        ▼
Physical Shipping
        │
        ▼
Azure Data Center
        │
        ▼
Azure Blob Storage
```

処理の流れは以下の通りである。

1. Azure Portal で Data Box を注文
    
2. Microsoft が Data Box デバイスをオンプレミスに配送
    
3. ローカルネットワーク経由でデータをコピー
    
4. デバイスを Microsoft に返送
    
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Storage Account]]
5. Microsoft がデータを Azure Storage にアップロード
    

この方法では、データ転送のボトルネックとなるインターネット帯域を回避できる。

---

# 4 Data Box を選択すべき理由

今回の要件では、ネットワーク帯域が制限されているため、インターネット経由で 50TB のデータを転送することは現実的ではない可能性が高い。

例えば、100Mbps 回線でデータ転送を行った場合の理論値は次のようになる。

```text
100 Mbps ≈ 12.5 MB/s

50 TB ≈ 50,000 GB

転送時間 ≈ 約46日
```

このため、1週間以内に転送することはほぼ不可能である。

一方、Azure Data Box を使用すると、データはローカルネットワーク速度でコピーできるため、数時間から数日で転送が完了する可能性がある。

---

# 5 セキュリティ

Azure Data Box は高いセキュリティを提供するよう設計されている。データはデバイス上で暗号化され、輸送中も安全に保護される。

主なセキュリティ機能は次の通りである。

- AES 256 ビット暗号化
    
- デバイスロック機能
    
- 改ざん検知
    
- データ消去機能
    

これにより、物理輸送中のデータ漏洩リスクを最小化できる。

---

# 6 他の選択肢が適切でない理由

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Data Factory]]
Azure Data Factory を使用したデータ転送は、クラウドとオンプレミスのデータ統合に適したサービスであるが、ネットワーク経由でデータを転送するため帯域幅の制約を受ける。50TB のデータを1週間以内に転送するには不十分な可能性が高い。

AzCopy は Azure Storage への高速データ転送ツールであり、マルチスレッド転送をサポートしているが、やはりインターネット帯域に依存する。そのため帯域が制限されている環境では十分な速度が得られない。

VPN 経由で Robocopy を使用する方法も同様にネットワーク帯域の制限を受けるため、大容量データ移行には適さない。

Azure Import/Export サービスもオフライン転送を提供するが、ディスク準備や配送などの手作業が多く、スケジュールの確実性の点で Data Box より劣る場合がある。

---

# 7 アーキテクチャ

Azure Data Box を利用したデータ移行アーキテクチャは次のようになる。

```text
On-Premises Data Center
       │
       ▼
File Server
       │
       ▼
Azure Data Box Device
       │
       ▼
Secure Shipping
       │
       ▼
Azure Data Center
       │
       ▼
Azure Blob Storage
```

この構成では、オンプレミスからクラウドへのデータ転送を物理デバイスを利用して高速に実現できる。

---

# 8 メリット

Azure Data Box を利用することで次のメリットが得られる。

まず、ネットワーク帯域に依存しないため、大容量データを短期間で Azure に転送できる。また、データは暗号化されているため、安全に輸送することが可能である。

さらに、Azure Storage と直接統合されているため、データが Azure に到着すると自動的に Blob Storage に取り込まれる。

---

# 9 まとめ

今回の要件は次の通りである。

- オンプレミスから Azure Blob Storage へ **50TB のデータ転送**
    
- **1週間以内に完了**
    
- **インターネット帯域が制限されている**
    

これらの条件を満たす最適な Azure ソリューションは

**Azure Data Box**

である。

Azure Data Box を利用することで、大容量データを安全かつ高速に Azure へ移行でき、ネットワーク帯域の制約を回避することが可能になる。これは大規模クラウド移行プロジェクトで広く利用されている標準的なデータ転送方法である。
