# Azureで複数サブスクリプションを管理するためのガバナンス設計（Tags と Management Groups）

## 1. 背景（シナリオ）

ある組織では、複数のAzureサブスクリプションにまたがって多数のリソースを運用しています。これらのリソースは、組織のさまざまな部門やプロジェクト、さらには開発環境や本番環境といった用途ごとに分類されています。

このような環境では、単にリソースを作成するだけではなく、組織全体のガバナンスを維持することが重要になります。特に、以下のような要求が存在します。

まず、リソースを部門・プロジェクト・環境などの観点で整理する必要があります。さらに、監査やコンプライアンスの観点から、リソースの状況を容易にレポートできることも求められています。

このため、Azure環境全体でリソースを効率よく整理し、複数サブスクリプションにわたるコンプライアンス管理を実現するための仕組みを導入する必要があります。

## 2. 要件整理

この問題のポイントは、Azureリソースを単に作成することではなく、**組織全体で管理しやすい構造を作ること**です。

特に重要な要件は以下の通りです。

- 複数のAzureサブスクリプションを横断して管理できる
- リソースを部門やプロジェクトなどの基準で分類できる
- 組織のコンプライアンス状況を報告できる
- 管理構造がスケーラブルである
- リソースを簡単に検索・分析できる

これらの要件を満たすには、Azureの**ガバナンス機能**を利用する必要があります。

## 3. 技術の基本概念

Azureでは、リソース管理を効率化するための複数のガバナンス機能が提供されています。その中でも、この問題で特に重要になるのが **Azure Tags** と **Management Groups** です。

まず、**Azure Tags** について説明します。

タグとは、Azureリソースに付与できる **キーと値のペア（Key-Value）** です。タグを利用することで、リソースに意味的な分類を付けることができます。

例えば、次のようなタグを設定することができます。

- Department: Finance
- Project: CRM
- Environment: Production

このようにタグを付けることで、Azure PortalやAzure CLI、レポート機能を利用してリソースを簡単に分類・検索できるようになります。

次に **Azure Management Groups** です。

管理グループは、複数のサブスクリプションを階層構造で管理するための機能です。

Azureの管理階層は次のような構造になっています。

Tenant  
Management Group  
Subscription  
Resource Group  
Resource

この構造により、組織単位でポリシーやアクセス制御を適用できます。

例えば、次のような管理構造を作ることが可能です。

- Management Group: Production
- Management Group: Development
- Management Group: Finance Department

この階層構造により、複数サブスクリプションを一元管理できます。

## 4. アーキテクチャまたは設計のポイント

Azure環境で複数サブスクリプションを管理する場合、リソース整理とガバナンスを両立させることが重要です。

まず、リソースの分類について考えます。

タグを使用することで、リソースにメタデータを追加できます。これにより、組織は次のような視点でリソースを分析できます。

- 部門ごとのリソース
- プロジェクトごとのリソース
- 環境（Dev / Test / Prod）
- コスト配分

タグはレポート作成にも非常に有効です。

次に、複数サブスクリプションの管理について考えます。

管理グループを使用すると、サブスクリプションを階層的に整理できます。この階層構造を利用することで、以下のような管理が可能になります。

- RBACアクセス制御
- Azure Policy適用
- コンプライアンス管理
- 組織単位のガバナンス

つまり、タグは「分類」、管理グループは「構造管理」を担当します。

## 5. 設計判断（なぜこの構成になるか）

このシナリオでは、**複数のサブスクリプションを横断したリソース管理とコンプライアンスレポート**が求められています。

この要件を満たすには、次の2つの機能の組み合わせが最適です。

Azure Tags  
Azure Management Groups

まず、タグを使用することでリソースを分類できます。これにより、部門・プロジェクト・環境といった観点でレポートを作成できます。

次に、管理グループを使用することで、複数のサブスクリプションを組織構造に合わせて整理できます。

さらに、管理グループレベルでポリシーを適用することで、組織全体のコンプライアンスを一貫して維持できます。

このように、タグと管理グループを組み合わせることで、**リソース整理とガバナンス管理の両方を実現できます。**

## 6. 他の選択肢が誤りな理由

### A. ARMテンプレートとAzure Blueprints

ARMテンプレートとBlueprintは主に **リソースのデプロイメントの標準化**に使用されます。

つまり、

- リソース作成
- 環境テンプレート

に焦点を当てた機能です。

しかし、この問題の要件である **既存リソースの整理やレポート作成**には直接対応していません。

### B. Azure Policy と Azure Resource Graph

Azure Policyはコンプライアンスの適用には非常に有効です。しかし、リソースの分類や組織構造の整理は行いません。

Azure Resource Graphはリソース検索のためのツールであり、リソース管理の構造を作る機能ではありません。

### D. Azure Cost Management と Security Center

これらはそれぞれ次の用途です。

- Cost Management: コスト分析
- Security Center: セキュリティ管理

いずれもリソース構造の整理やコンプライアンスレポートを目的とした機能ではありません。

## 7. 最終回答

複数のサブスクリプション間でリソースを整理し、コンプライアンス報告を容易にする最適なソリューションは次です。

**Azure Tags と Azure Management Groups**

したがって **正解は C** です。

## 8. まとめ

Azureで大規模な環境を管理する場合、ガバナンス設計が非常に重要になります。

特に次の2つの機能が重要です。

まず、**Azure Tags** です。  
タグはリソースにメタデータを付与し、部門・プロジェクト・環境などの分類を可能にします。

次に **Azure Management Groups** です。  
管理グループは複数のサブスクリプションを階層構造で管理でき、ポリシーやアクセス制御を一括適用できます。

この2つを組み合わせることで、組織全体のAzureリソースを整理し、コンプライアンス管理やレポート作成を効率化できます。
