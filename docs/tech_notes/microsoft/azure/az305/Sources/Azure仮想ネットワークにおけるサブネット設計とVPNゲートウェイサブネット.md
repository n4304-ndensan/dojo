# Azure仮想ネットワークにおけるサブネット設計とVPNゲートウェイサブネット

## 1. 背景（シナリオ）

ある企業では、新しいAzure環境を構築することになり、そのネットワークアーキテクチャの設計を担当することになりました。  
この環境では **100台のAzure仮想マシン（VM）** をホストする予定があります。

一方、企業にはすでに **オンプレミス環境が存在しており、IPアドレス空間として 10.1.0.0/16 を使用しています。**

Azure環境では、オンプレミスのサーバーと仮想マシンが **シームレスに通信できるようにサイト間VPN（Site-to-Site VPN）を構成**する必要があります。

そのため、Azure側では次の2つのサブネットを作成する計画です。

まず、Azure仮想マシンを配置するための **VM専用サブネット（SubnetA）** を作成します。  
次に、VPN接続を提供するための **GatewaySubnet（VPNゲートウェイ専用サブネット）** を作成します。

さらに、この設計では以下の要件を満たす必要があります。

- 100台以上のVMを配置できる十分なIPアドレス
- 将来的なスケーリングに対応できる余裕
- VPNゲートウェイが利用できる適切なサブネットサイズ
- 高可用性と拡張性を確保したネットワーク設計

## 2. 要件整理

この問題では、ネットワーク設計に必要な条件を整理することが重要です。  
まず、設計上の主要要件を明確にします。

このシナリオにおける要件は以下の通りです。

- Azure上に **100台の仮想マシンを配置する**
- Azure VM用の専用サブネット **SubnetA** を作成する
- **Site-to-Site VPN** 用の **GatewaySubnet** を作成する
- 既存のオンプレミスネットワークは **10.1.0.0/16**
- 将来的なスケールを考慮したアドレス設計
- Azureネットワークのベストプラクティスに従う

この要件を満たすには、VMサブネットとゲートウェイサブネットの両方を適切なサイズで設計する必要があります。

## 3. 技術の基本概念

Azureの仮想ネットワーク設計では、IPアドレスとサブネットサイズの理解が重要です。

まず、CIDR表記（Classless Inter-Domain Routing）について理解する必要があります。  
CIDRでは「/」以降の数値によってネットワークサイズが決まります。

主なサブネットサイズの例は次の通りです。

- /22 → 1024アドレス
- /23 → 512アドレス
- /24 → 256アドレス
- /25 → 128アドレス
- /26 → 64アドレス
- /27 → 32アドレス

また、Azureではサブネット内の **最初の4つと最後の1つのIPアドレスが予約される** ため、実際に利用できるIPアドレスは少なくなります。

例えば /24 の場合：

- 総アドレス数: 256
- Azure予約: 5
- 使用可能: 251

この仕組みを理解しておくことで、VM数に対して十分なサブネットサイズを選択できます。

次に **GatewaySubnet** の概念です。

Azure VPN Gateway は **専用サブネット GatewaySubnet に配置される必要があります。**

Microsoftのベストプラクティスでは、将来的なスケーリングや可用性を考慮して **/27 より大きいサイズを推奨**しています。

## 4. アーキテクチャまたは設計のポイント

この問題では、VM用サブネットとVPNゲートウェイサブネットのサイズを適切に選ぶことが重要です。

まず、VMサブネットについて考えます。

VMサブネットは次の条件を満たす必要があります。

- 100台以上のVMを配置できる
- 将来の増加に対応できる
- IPアドレス枯渇を防ぐ

これらを満たすためには、以下のような設計が望ましいです。

- /23 サブネット（約512アドレス）
- 約500以上の使用可能IP
- 将来的なVM追加に対応可能

次に、VPNゲートウェイサブネットです。

VPN Gatewayは専用サブネットに配置されるため、次の点が重要になります。

- GatewaySubnet という名前が必要
- 他のリソースは配置できない
- 将来のVPN構成変更にも対応するサイズ

そのため、以下のような設計が望ましいです。

- /26 サブネット
- 約64アドレス
- 高可用性構成に対応

## 5. 設計判断（なぜこの構成になるか）

最適な設計は次の構成です。

SubnetA  
10.1.0.0/23

GatewaySubnet  
10.1.2.0/26

この構成が優れている理由は以下の通りです。

まず、VMサブネットについて説明します。

- /23 は 512アドレス
- 約507利用可能
- 100VM以上を安全に配置可能
- 将来のスケールに対応

次にVPNゲートウェイサブネットです。

- /26 は 64アドレス
- Azure VPN Gateway の拡張に対応
- 将来的な構成変更に柔軟

さらに重要なのは **アドレス範囲の衝突がないこと**です。

10.1.0.0/23 は

10.1.0.0 – 10.1.1.255

10.1.2.0/26 は

10.1.2.0 – 10.1.2.63

つまり、互いに重複していません。

## 6. 他の選択肢が誤りな理由

### A

SubnetA  
10.1.0.0/25

GatewaySubnet  
10.1.0.128/27

この構成の問題点を説明します。

まず /25 サブネットは次の通りです。

- 128アドレス
- 約123使用可能

100VMを配置すると **ほぼ余裕がありません。**

将来的な拡張ができないため不適切です。

### B

SubnetA  
10.1.0.0/24

GatewaySubnet  
1.10.0/27

この選択肢には **IPアドレスの誤り**があります。

- GatewaySubnet が VNet範囲外
- アドレス設計として不正

したがって無効です。

### D

SubnetA  
10.1.0.0/22

GatewaySubnet  
10.1.4.0/27

VMサブネットは十分ですが、問題は **GatewaySubnet /27** です。

/27 は最小サイズであり、将来的な拡張を考えると十分とは言えません。

そのためベストプラクティスではありません。

## 7. 最終回答

正しい設計は次の通りです。

SubnetA  
10.1.0.0/23

GatewaySubnet  
10.1.2.0/26

よって **正解は C** です。

## 8. まとめ

この問題では、Azureネットワーク設計における **サブネットサイズとVPNゲートウェイサブネットのベストプラクティス**が問われています。

重要なポイントを整理すると次の通りです。

まず、VMサブネット設計では以下が重要です。

- VM数より十分に大きいIPアドレス空間
- 将来のスケーラビリティ
- IP枯渇を防ぐ設計

次にVPNゲートウェイサブネットです。

- GatewaySubnet という専用サブネット
- 将来を考慮したサイズ
- /26 以上が推奨される場合が多い

Azureネットワーク設計では **「現在の要件＋将来の拡張」** を考慮することが非常に重要です。
