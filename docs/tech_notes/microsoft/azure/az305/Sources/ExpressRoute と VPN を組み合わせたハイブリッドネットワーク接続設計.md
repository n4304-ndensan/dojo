# ExpressRoute と VPN を組み合わせたハイブリッドネットワーク接続設計

## 1. 背景（シナリオ）

ある企業では、オンプレミスデータセンターと Azure クラウド環境を接続するハイブリッドネットワークアーキテクチャを設計しています。企業のワークロードは Azure 上の仮想ネットワークで稼働しており、オンプレミスのシステムと頻繁に通信する必要があります。

この接続設計では、通常のインターネット接続ではなく、より高いセキュリティとパフォーマンスを提供する接続方式が求められています。具体的には、オンプレミスネットワークと Azure の間で **公共インターネットを経由しないプライベート接続**を実現する必要があります。

さらに、この接続は企業の重要な業務システムを支えるため、安定した通信品質と保証された可用性が必要です。そのため、接続には **高帯域幅と SLA（Service Level Agreement）** が求められています。

また、この企業では Microsoft 365 などの Microsoft クラウドサービスも利用しています。そのため、Azure リソースへのプライベート接続だけでなく、Microsoft 365 のような Microsoft サービスへも適切にアクセスできる必要があります。

このような条件を満たすためには、Azure のハイブリッド接続サービスを組み合わせたネットワーク設計が必要になります。

---

## 2. 要件整理

問題文の条件を整理すると、ネットワーク設計に関する重要な要件がいくつか存在します。それぞれの条件を理解することで、どの接続方式を選択するべきかが明確になります。

まず、接続経路に関する要件があります。企業はオンプレミス環境と Azure の間で安全な通信を行う必要があります。そのため、インターネットを経由しない専用接続が必要です。

- Azure とオンプレミス間の通信は **公共インターネットを経由しない**

次に、パフォーマンスと可用性に関する要件があります。ビジネスクリティカルなシステムではネットワークの停止が許されないため、高帯域幅と SLA を持つ接続が必要です。

- **高帯域接続**
- **保証された稼働時間 SLA**

さらに、Microsoft サービスへのアクセス要件があります。企業は Microsoft 365 などのサービスを利用しているため、それらのサービスにも適切にアクセスできる必要があります。

- Microsoft 365 などの **Microsoft サービスへの接続**

これらの条件を満たすネットワークソリューションを設計する必要があります。

---

## 3. 技術の基本概念

### ExpressRoute

ExpressRoute は、オンプレミスネットワークと Microsoft クラウド（Azure や Microsoft 365）を接続するための専用ネットワーク接続サービスです。ExpressRoute を使用すると、企業のネットワークは Microsoft のバックボーンネットワークに直接接続されます。

この接続はインターネットを経由しないため、通常の VPN 接続よりも高いセキュリティと安定性を提供します。また、高帯域幅と SLA が保証されるため、大規模な企業環境で広く利用されています。

ExpressRoute の接続方式には複数のピアリングタイプがあります。これらのピアリングは、接続するサービスの種類によって使い分けられます。

ExpressRoute の主なピアリングタイプを理解するためには、それぞれの用途を整理することが重要です。

- **Private Peering**
- **Microsoft Peering**

Private Peering はオンプレミスネットワークと Azure 仮想ネットワークを接続するために使用されます。これにより、企業は Azure 内の仮想マシンやサービスへプライベートネットワーク経由でアクセスできます。

Microsoft Peering は Microsoft 365 や Azure DevOps などの Microsoft サービスに接続するために使用されます。これにより、Microsoft サービスへの通信も Microsoft のバックボーンネットワークを経由します。

---

### Site-to-Site VPN

Site-to-Site VPN は、オンプレミスネットワークと Azure 仮想ネットワークをインターネット経由で接続する仕組みです。この接続は IPsec トンネルを使用して暗号化された通信を行います。

VPN 接続は ExpressRoute よりも低コストで導入できますが、インターネットを経由するため通信品質はインターネットの状況に依存します。そのため、通常は ExpressRoute の代替ではなく **バックアップ接続**として使用されることが多いです。

---

## 4. アーキテクチャまたは設計のポイント

今回のネットワーク設計では、ExpressRoute を使用してオンプレミスネットワークと Azure を接続します。ExpressRoute を使用することで、通信は Microsoft のバックボーンネットワークを経由し、公共インターネットを通過することはありません。

この接続では Private Peering を使用して Azure 仮想ネットワークに接続します。これにより、オンプレミスのシステムは Azure 内の仮想マシンやサービスへ安全にアクセスできます。

さらに、ネットワークの可用性を高めるために Site-to-Site VPN をフェイルオーバー用の接続として構成します。ExpressRoute に障害が発生した場合でも、VPN 接続を通じて通信を継続できます。

この構成により、企業は高帯域の専用接続と冗長性のあるネットワーク構成を同時に実現できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題では **プライベート接続、高帯域、SLA、可用性**のすべてを満たす必要があります。

ExpressRoute は Microsoft のバックボーンネットワークを使用する専用接続であり、公共インターネットを経由しません。そのためセキュリティとパフォーマンスの両方の要件を満たすことができます。

しかし、どのような専用回線でも障害の可能性はゼロではありません。そのため、ビジネスクリティカルなシステムではバックアップ接続を用意することが重要です。

Site-to-Site VPN をフェイルオーバー接続として構成することで、ExpressRoute に障害が発生した場合でもオンプレミスと Azure の通信を維持できます。

このように、ExpressRoute と VPN を組み合わせた構成が最も信頼性の高いハイブリッド接続設計になります。

---

## 6. 他の選択肢が誤りな理由

### A Site-to-Site VPN のみ

VPN 接続はインターネットを経由するため、専用回線のような高帯域と SLA を提供できません。今回の要件である「公共インターネットを経由しない接続」を満たさないため適切ではありません。

### B Microsoft Peering のみ

Microsoft Peering は Microsoft 365 などの Microsoft サービスへの接続に使用されますが、Azure 仮想ネットワークへの接続には Private Peering が必要です。そのため、この構成だけでは Azure ワークロードへの接続を実現できません。

### C Private Peering のみ

Private Peering は Azure 仮想ネットワークへの接続には適していますが、冗長性が不足しています。フェイルオーバー接続がないため、可用性の要件を満たすことができません。
