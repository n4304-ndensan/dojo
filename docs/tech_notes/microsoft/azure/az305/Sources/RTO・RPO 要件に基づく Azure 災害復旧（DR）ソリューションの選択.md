# RTO・RPO 要件に基づく Azure 災害復旧（DR）ソリューションの選択

## 1. 背景（シナリオ）

企業がオンプレミスで運用しているシステムをクラウドと連携させる際には、災害発生時にどのようにシステムを復旧するかという **ディザスタリカバリー（Disaster Recovery: DR）戦略** を設計する必要がある。DR の設計では、どれだけ早くサービスを復旧できるか、どれだけデータ損失を許容できるかといった指標が重要になる。これらの指標は一般的に **RTO（Recovery Time Objective）** と **RPO（Recovery Point Objective）** と呼ばれる。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Backup]]
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Site Recovery]]
RTO はサービス停止後にどれだけ早くシステムを復旧する必要があるかを示す時間目標であり、RPO は災害時にどれだけのデータ損失を許容できるかを示す指標である。Azure では、このような DR 要件を満たすために **Azure Site Recovery** や **Azure Backup** といったサービスが提供されている。これらのサービスは目的が異なり、Site Recovery は主に迅速なフェールオーバーを実現するためのレプリケーションベースの DR ソリューションであり、Backup はデータの保護と長期保存を目的としたバックアップソリューションである。

今回のシナリオでは、オンプレミス環境で稼働している三つのアプリケーションが存在しており、それぞれ異なる RTO・RPO 要件を持っている。そのため、各アプリケーションの要件に最適な DR ソリューションを選択する必要がある。

---

## 2. 要件整理

問題に登場する三つのアプリケーションは、それぞれ異なる運用特性と復旧要件を持っている。まず営業システムは、セカンダリーデータセンターへのフェールオーバーを想定しており、できるだけ短い停止時間で復旧できることが求められている。この場合、バックアップから復元する方式では復旧に時間がかかるため、リアルタイムまたは準リアルタイムでレプリケーションを行う仕組みが必要になる。

次に財務システムは、RTO が 10 分と非常に短く設定されており、さらに Azure 上で稼働できるようにする必要がある。また、財務データはコンプライアンスや監査要件により **7 年間のデータ保持** が必要とされている。この場合、迅速な復旧を可能にする DR 機能と、長期保存を可能にするバックアップの両方が必要になる。

最後にレポート作成システムは、日次復旧でよく、RTO も 8 時間と比較的長い。これは即時のフェールオーバーが必要なシステムではなく、バックアップからの復元でも業務要件を満たすことができるタイプのワークロードである。

---

## 3. Azure Site Recovery の役割

Azure Site Recovery（ASR）は、オンプレミスのサーバーや仮想マシンを Azure にレプリケーションし、障害が発生した際に自動的に Azure 側へフェールオーバーすることができる DR サービスである。このサービスの特徴は、継続的にデータをレプリケーションすることで、非常に短い RTO と RPO を実現できる点にある。

Site Recovery は主に次のようなシナリオに適している。まず、重要な業務システムであり、停止時間を最小限に抑える必要がある場合である。リアルタイムまたは数分単位のレプリケーションによって、災害発生時でも短時間でシステムを再稼働させることができる。また、オンプレミスから Azure への DR を構築する際にもよく使用される。

営業システムのようにセカンダリーデータセンターへのフェールオーバーを前提としたシステムでは、このようなレプリケーション型の DR が適している。

---

## 4. Azure Backup の役割

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Recovery Services Vault]]
Azure Backup は、データや仮想マシンのバックアップを取得し、Azure Recovery Services Vault に保存することで、データ保護と復旧を可能にするサービスである。このサービスは主にデータ保護と長期保存を目的としており、バックアップの保存期間を数年単位で設定することができる。

Azure Backup は次のようなシナリオで利用されることが多い。まず、長期間のデータ保持が必要な場合である。例えば財務データや監査ログなどは数年間保存する必要があることが多く、バックアップによってこれを実現できる。また、RTO 要件が比較的緩やかなシステムでは、バックアップからの復元で十分対応できる。

レポート作成システムのように、即時復旧の必要がなく、数時間以内に復旧できればよいシステムでは、バックアップのみで DR 要件を満たすことができる。

---

## 5. 技術的な構成

今回の三つのアプリケーションを Azure の DR ソリューションにマッピングすると、次のような構成になる。

```id="dr-architecture"
Sales Application
      │
      │ Continuous replication
      ▼
Azure Site Recovery

Finance Application
      │
      ├─ Azure Site Recovery (fast failover)
      │
      └─ Azure Backup (7-year retention)

Reporting Application
      │
      ▼
Azure Backup
```

営業システムは短い復旧時間を求めるため Site Recovery のみを使用する。財務システムは迅速な復旧と長期保存の両方が必要なため Site Recovery と Backup の両方を使用する。レポート作成システムは復旧時間が長くても許容されるため、Backup のみで対応できる。

---

## 6. 他の選択肢が誤りな理由

営業システムに Backup のみを使用する構成は適切ではない。バックアップからの復元では復旧に時間がかかるため、セカンダリーデータセンターへの迅速なフェールオーバーという要件を満たすことができない。

また、財務システムで Site Recovery のみを使用する構成も不十分である。確かに Site Recovery は迅速な復旧を実現できるが、7 年間のデータ保持という要件を満たすことができない。長期保存にはバックアップが必要になる。

さらに、レポート作成システムに Site Recovery を導入する構成は過剰設計になる可能性がある。RTO が 8 時間であるため、バックアップからの復元でも要件を満たすことができる。この場合、Site Recovery を使用するとコストや運用の複雑さが増えるだけになる。

---

## 7. 最終回答

**B.  
Sales：Site Recovery のみ  
Finance：Site Recovery と Backup  
Reporting：Backup のみ**

---

## 8. まとめ

Azure の DR 設計では、すべてのワークロードに同じソリューションを適用するのではなく、それぞれのアプリケーションの RTO と RPO 要件に応じて最適なサービスを選択することが重要である。短時間で復旧する必要があるシステムには Azure Site Recovery が適しており、長期保存や比較的緩やかな復旧要件を持つシステムには Azure Backup が適している。

今回のシナリオでは、営業システムは迅速なフェールオーバーが必要なため Site Recovery を使用し、財務システムは迅速な復旧と長期保存の両方が必要なため Site Recovery と Backup を組み合わせる。そしてレポート作成システムは復旧時間に余裕があるため Backup のみで対応するという構成が最も合理的な DR 設計となる。
