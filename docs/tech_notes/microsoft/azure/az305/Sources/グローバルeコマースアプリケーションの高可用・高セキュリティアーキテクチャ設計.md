# グローバルeコマースアプリケーションの高可用・高セキュリティアーキテクチャ設計

## 1. 背景（シナリオ）

ある組織は、複数のAzureリージョンに展開されるグローバルなeコマースアプリケーションのアーキテクチャ設計を任されています。このアプリケーションは世界中のユーザーが利用するため、高い可用性と回復力を備えた構成である必要があります。

特に、ユーザーの地理的位置やリアルタイムのパフォーマンスメトリクスに基づいてトラフィックを最適なリージョンへルーティングできる仕組みが必要です。また、eコマースサイトであるため、SQLインジェクションやクロスサイトスクリプティングなどの一般的なWeb攻撃からアプリケーションを保護する必要があります。

さらに、リージョン障害が発生した場合でもサービス停止を最小限に抑えるフェールオーバー機能や、トラフィック量の変動に対応できる自動スケーリング機能も求められています。

---

## 2. 要件整理

このシナリオでは、グローバルアプリケーションの設計に必要な要件を明確に理解することが重要です。

まず、ユーザーの場所やパフォーマンスに応じたトラフィックルーティングが必要です。これにより、ユーザーは最も近く、かつ最も応答性の高いリージョンに接続できます。

次に、高可用性とフェールオーバー機能が必要です。特定のリージョンで障害が発生した場合でも、別リージョンへトラフィックを切り替えることでサービス継続が可能になります。

さらに、アプリケーション層のセキュリティも重要です。特にWebアプリケーションでは、一般的な攻撃から保護するためにWAF（Web Application Firewall）の導入が必要になります。

最後に、eコマースアプリケーションはセールやイベントなどによりトラフィックが大きく変動するため、自動スケーリング機能が必要になります。

この問題の要件は次のように整理できます。

- グローバルなトラフィックルーティング  
- 高可用性とフェールオーバー  
- Webアプリケーションセキュリティ（WAF）  
- 自動スケーリング  
- レイヤー7のトラフィック制御  

---

## 3. 技術の基本概念

Azureでグローバルアプリケーションを設計する場合、複数のロードバランシングサービスが存在します。それぞれの役割を理解することが重要です。

まずAzure Front Doorについて説明します。Azure Front Doorはグローバルレベルで動作するHTTP/HTTPSロードバランサーです。ユーザーの地理的位置やバックエンドのパフォーマンスに基づいてトラフィックを最適なリージョンへルーティングできます。

主な機能は次の通りです。

- グローバルロードバランシング  
- パフォーマンスベースルーティング  
- 地理ベースルーティング  
- 自動フェールオーバー  
- WAF統合  

次にAzure Application Gatewayについて説明します。Application Gatewayはレイヤー7ロードバランサーとして動作し、HTTPリクエストの内容に基づいたルーティングを行うことができます。

代表的な機能は次の通りです。

- URLパスベースルーティング  
- セッションアフィニティ  
- SSL終端  
- Web Application Firewall  
- オートスケール  

この2つのサービスを組み合わせることで、グローバルレベルとリージョンレベルの両方で高度なトラフィック管理が可能になります。

---

## 4. アーキテクチャまたは設計のポイント

グローバルアプリケーションの設計では、トラフィック管理を階層化することが重要です。

まず、世界中のユーザーからのトラフィックはAzure Front Doorで受け取ります。Front Doorはユーザーの場所やバックエンドの応答速度を考慮し、最適なリージョンへトラフィックを送信します。

その後、各リージョンではApplication Gatewayがトラフィックを処理します。

このアーキテクチャの流れは次のようになります。

1. ユーザーがグローバルエンドポイントへアクセス  
2. Azure Front Doorが最適なリージョンを選択  
3. トラフィックがApplication Gatewayへ転送  
4. Application Gatewayがバックエンドサービスへルーティング  

この構成により、次のメリットが得られます。

- グローバル負荷分散  
- 高可用性  
- レイヤー7ルーティング  
- WAFによるセキュリティ  
- 自動スケーリング  

特にApplication GatewayのAutoscale機能により、トラフィック量の増減に応じてインスタンス数を自動調整できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題では、複数の要求を同時に満たす必要があります。

まず、ユーザーの位置とパフォーマンスに基づいたトラフィックルーティングにはAzure Front Doorが最適です。Front Doorはグローバルエッジネットワークを利用して、最適なバックエンドにトラフィックを送ることができます。

次に、アプリケーションレベルのトラフィック制御にはApplication Gatewayが適しています。URLパスベースルーティングやセッション管理など、Webアプリケーションに必要な高度な機能を提供します。

また、両サービスともWAF機能を提供しており、SQLインジェクションやXSSなどの攻撃からアプリケーションを保護できます。

さらにApplication GatewayのAutoscale機能により、トラフィック増加時にもパフォーマンスを維持できます。

このため、グローバルアプリケーションでは次のような構成がベストプラクティスとなります。

Azure Front Door  
＋  
Azure Application Gateway（Autoscale）

---

## 6. 他の選択肢が誤りな理由

まずTraffic Managerの選択肢について考えます。Traffic ManagerはDNSベースのロードバランサーであり、グローバルルーティングは可能ですが、アプリケーション層のセキュリティやWAF機能を直接提供するものではありません。また、リアルタイムHTTPレベルのルーティング制御もFront Doorほど高度ではありません。

次にLoad Balancerの選択肢です。Azure Load Balancerはレイヤー4のロードバランサーであり、HTTPリクエスト内容に基づくルーティングやアプリケーションレベルのセキュリティを提供できません。

最後にAzure Policyを含む選択肢です。Azure Policyはリソースのガバナンスやコンプライアンス管理のためのサービスであり、トラフィック管理やアプリケーションセキュリティには直接関係しません。

そのため、これらの選択肢は問題の要件を完全には満たしません。

---

## 7. 最終回答

A.  
Azure Front Door と Azure Application Gateway（Autoscale）

---

## 8. まとめ

グローバルなWebアプリケーションをAzureで設計する場合、複数のレベルでトラフィック管理を行うことが重要です。

Azure Front Doorは次の役割を担います。

- グローバルトラフィック管理  
- パフォーマンスベースルーティング  
- 自動フェールオーバー  
- WAFセキュリティ  

一方、Azure Application Gatewayはリージョンレベルでのアプリケーション制御を提供します。

- レイヤー7ロードバランシング  
- URLルーティング  
- セッション管理  
- WAF  
- オートスケール  

この2つを組み合わせることで、**高可用性・高セキュリティ・高パフォーマンスのグローバルアーキテクチャ**を実現できます。
