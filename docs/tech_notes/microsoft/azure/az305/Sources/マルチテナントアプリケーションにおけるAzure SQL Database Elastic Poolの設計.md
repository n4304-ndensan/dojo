# マルチテナントアプリケーションにおけるAzure SQL Database Elastic Poolの設計

## 1. 背景（シナリオ）

ある組織では、Azure上で稼働するマルチテナントアプリケーションを運用しています。このアプリケーションは合計50個の独立したSQLデータベースで構成されており、それぞれのデータベースのサイズは10GBから50GB程度です。

マルチテナント環境では、各テナントごとにデータベースを分離することでセキュリティとデータ分離を確保できます。しかし、各データベースの使用量やワークロードは常に同じではなく、時間帯やユーザー活動によって大きく変動します。

このシステムには以下の重要な要件があります。

まず、ビジネス上の理由から高可用性が非常に重要であり、99.99%のSLAを満たす必要があります。次に、ワークロードの変動に応じてリソースを自動的に調整できる仕組みが必要です。また、50個のデータベースを個別に最大リソースでプロビジョニングするとコストが非常に高くなるため、コスト効率も重要な要件となります。

---

## 2. 要件整理

この問題では、マルチテナントデータベース環境における3つの重要な要件を満たす必要があります。

まず高可用性です。クラウドサービスとして提供されるデータベースは、システム障害が発生してもサービスを継続できる設計が求められます。この問題では特に99.99%のSLAが必要です。

次に自動スケーリングです。テナントごとに使用パターンが異なるため、特定のデータベースだけ負荷が高くなる場合があります。そのような場合でもリソースを柔軟に利用できる仕組みが必要になります。

最後にコスト効率です。50個のデータベースをすべて最大容量で個別に構成すると、多くのリソースが未使用のままになる可能性があります。そのため、複数のデータベース間でリソースを共有できる仕組みが望まれます。

これらの要件を整理すると次のようになります。

- 高可用性（SLA 99.99%）
- 複数データベースの効率的管理
- ワークロード変動への対応
- コスト最適化

---

## 3. 技術の基本概念

Azure SQL Databaseには、複数のデータベースを管理するためのいくつかのサービスモデルがあります。その中でもマルチテナント環境で特に重要なのがElastic Poolです。

Elastic Poolとは、複数のデータベースでコンピューティングリソースを共有する仕組みです。通常のAzure SQL Databaseでは、各データベースごとに専用のリソースが割り当てられます。しかしElastic Poolでは、複数のデータベースが共通のリソースプールを利用します。

Elastic Poolの主な特徴として、次のようなものがあります。

- 複数データベースでリソース共有
- 動的なリソース割り当て
- 変動するワークロードへの対応
- コスト効率の向上

例えば、50個のデータベースがある場合でも、すべてのデータベースが同時に最大負荷になる可能性は低いです。そのため、共通のリソースプールを使用することで、必要なときだけリソースを利用できます。

---

## 4. アーキテクチャまたは設計のポイント

マルチテナントアプリケーションでは、テナントごとにデータベースを分離する設計がよく採用されます。このアプローチは「Database-per-tenant」と呼ばれます。

この構成では、次のようなメリットがあります。

- テナント間のデータ分離
- セキュリティ向上
- バックアップやリストアの独立性

しかし、すべてのデータベースを個別にスケールさせるとコストが増加する問題があります。

そこでElastic Poolを利用することで、次のような構成を実現できます。

まず、すべてのテナントデータベースを1つのElastic Poolに配置します。その後、プール全体に対してDTUまたはvCoreリソースを割り当てます。

この構成により、以下のメリットが得られます。

- データベース間でリソース共有
- ピーク負荷の吸収
- コスト削減
- 管理の簡素化

またAzure SQL Databaseは標準で高可用性を提供しており、自動フェールオーバー機能によってSLA 99.99%を実現できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題では、複数のデータベースが存在し、ワークロードが予測不可能である点が重要なポイントです。

Elastic Poolは、まさにこのような環境を想定して設計されています。

Elastic Poolを採用する理由は次の通りです。

まず、複数データベースのリソース共有が可能になります。これにより、一部のデータベースの負荷が高い場合でも、プール内の余剰リソースを利用できます。

次に、コスト最適化が可能になります。すべてのデータベースを個別に最大容量で構成する必要がなくなるため、無駄なリソースコストを削減できます。

さらに、Azure SQL DatabaseはPaaSサービスであるため、バックアップ、パッチ適用、可用性管理などが自動化されています。これにより、運用管理の負担も大幅に軽減されます。

そのため、マルチテナントデータベース環境ではElastic Poolが最適な選択となります。

---

## 6. 他の選択肢が誤りな理由

まずAzure SQL Database Hyperscaleについて説明します。このサービスは非常に大規模な単一データベース向けに設計されています。最大数TB以上のストレージや多数の読み取りレプリカを必要とするケースに適しています。しかし、この問題では多数の小〜中規模データベースが存在するため適していません。

次にAzure VM上のSQL Serverです。この構成では仮想マシンの管理、パッチ適用、バックアップなどをユーザー自身が管理する必要があります。PaaSであるAzure SQL Databaseと比べると管理負担が大きく、高可用性の構成も複雑になります。

最後にAzure Cosmos DBです。Cosmos DBはNoSQLデータベースサービスであり、リレーショナルSQL Serverデータベースとはデータモデルが異なります。そのため既存のSQLベースのアプリケーションには適していません。

## 8. まとめ

マルチテナントアプリケーションでは、複数のデータベースを効率的に管理することが重要になります。

Azure SQL Database Elastic Poolは、次のようなメリットを提供します。

- 複数データベース間でのリソース共有
- ワークロード変動への柔軟な対応
- 高可用性（SLA 99.99%）
- コスト最適化
- 管理負担の削減

そのため、50個以上のデータベースを持つマルチテナントアプリケーションでは、Elastic Poolが最適なアーキテクチャとなります。
