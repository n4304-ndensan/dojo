# マルチテナントアプリケーション向け Azure Data Lake Storage Gen2 設計

## 1. 背景（シナリオ）

ある組織では、Azure上でマルチテナントアプリケーションを構築しています。このアプリケーションは複数のクライアント（テナント）にサービスを提供し、それぞれのクライアントは独自のデータを安全に保存する必要があります。

この環境では、各テナントのデータは完全に分離され、他のテナントからアクセスできないようにする必要があります。また、各テナントのデータは個別の暗号化キーで保護される必要があります。

さらに、このアプリケーションでは大量のデータを分析する必要があり、データ分析ワークロードに適したストレージ基盤が求められています。加えて、データの整理を容易にするために階層的なディレクトリ構造をサポートするストレージが必要になります。

このような要件を満たすストレージソリューションをAzureのサービスから選択する必要があります。

---

## 2. 要件整理

この問題では、マルチテナントアプリケーションのストレージ基盤として適切なサービスを選択する必要があります。

シナリオから読み取れる主な要件を整理すると次の通りです。

まず、テナントごとのデータ分離が必要です。各テナントのデータは他のテナントから完全に分離される必要があります。

次に、暗号化要件があります。すべてのデータは保存時に暗号化され、暗号化キーは顧客自身が管理する必要があります。

さらに、大量データの分析が可能なストレージである必要があります。つまりビッグデータ分析と連携できる必要があります。

最後に、データ整理のために階層的な名前空間が必要です。

これらの要件をまとめると次の通りです。

- テナントごとのデータ分離  
- 顧客管理キー（CMK）による暗号化  
- 保存時の暗号化  
- 大規模データ分析への対応  
- 階層的な名前空間  

---

## 3. 技術の基本概念

Azure Data Lake Storage Gen2 は、大規模データ分析向けに設計されたストレージサービスです。

このサービスはAzure Blob Storageを基盤としており、そこに「Hierarchical Namespace（階層的名前空間）」機能を追加したものです。

階層的名前空間を使用することで、ストレージ内のデータをディレクトリ構造として管理できます。

例えば、次のような構造を作成できます。

/tenantA  
/tenantA/logs  
/tenantA/data  

/tenantB  
/tenantB/logs  
/tenantB/data  

このような構造により、テナントごとのデータ分離が容易になります。

また、Data Lake Storage Gen2はビッグデータ分析サービスとネイティブ統合されています。

代表的な統合サービスには次のものがあります。

- Azure Synapse Analytics  
- Azure Databricks  
- Azure HDInsight  

これにより、大量データの分析処理を効率的に実行できます。

---

## 4. アーキテクチャまたは設計のポイント

マルチテナント環境では、ストレージ設計が非常に重要になります。

Azure Data Lake Storage Gen2では、ディレクトリベースの構造を利用してテナントごとにデータを分離できます。

この構成では次のような設計を採用します。

まず、ストレージアカウント内にテナントごとのディレクトリを作成します。

次に、Azure RBACまたはACLを利用してディレクトリ単位のアクセス制御を設定します。

さらに、Azure Key Vaultと統合して顧客管理キー（Customer Managed Key）を使用します。

この構成により次のメリットがあります。

- テナントごとのデータ分離  
- セキュアな暗号化管理  
- データアクセス制御  
- ビッグデータ分析対応  

また、Storage Service Encryption（SSE）により、保存時のデータは自動的に暗号化されます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題のポイントは「階層的名前空間」と「分析機能」です。

Azure Data Lake Storage Gen2は、唯一これらの要件を満たすAzureストレージサービスです。

Data Lake Storage Gen2を使用することで次の機能を利用できます。

- Hierarchical Namespace  
- 顧客管理キー（CMK）  
- 保存時暗号化  
- 大規模データ分析統合  
- 高いスケーラビリティ  

また、Blob Storageベースであるため、高い耐久性とスケーラビリティも提供されます。

そのため、マルチテナントアプリケーションのデータレイクとして最適です。

---

## 6. 他の選択肢が誤りな理由

Azure Blob StorageはData Lake Storage Gen2の基盤ですが、階層的名前空間や高度な分析機能を標準で提供していません。

Azure Cosmos DBはNoSQLデータベースであり、分析向けデータレイクとして使用するサービスではありません。

Azure Table Storageはキー・バリュー型ストレージであり、大規模データ分析や階層構造を必要とするワークロードには適していません。

B.

## 8. まとめ

マルチテナントアプリケーションのストレージ設計では、データ分離、セキュリティ、分析機能が重要になります。

Azure Data Lake Storage Gen2は次の機能を提供します。

- 階層的名前空間  
- 顧客管理キーによる暗号化  
- 保存時暗号化  
- ビッグデータ分析との統合  

これにより、マルチテナント環境でも安全でスケーラブルなデータ管理が可能になります。

そのため、このシナリオの最適なストレージソリューションは **Azure Data Lake Storage Gen2** です。
