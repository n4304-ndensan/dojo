# マルチリージョン Web アプリケーションのグローバル負荷分散（Azure Front Door）

## 1. 背景（シナリオ）

ある企業では、Web アプリケーションを Azure 上の複数リージョンにデプロイしています。このアプリケーションは世界中のユーザーが利用するため、ユーザーの地理的位置に応じて最適なリージョンへトラフィックを誘導する必要があります。

例えば、アジアのユーザーはアジアリージョンのアプリケーションに接続し、ヨーロッパのユーザーはヨーロッパリージョンのアプリケーションに接続するようにすることで、ネットワーク遅延を最小限に抑えることができます。

さらに、このシステムには高い可用性が求められています。もし1つのリージョンが停止した場合でも、ユーザーのリクエストは自動的に別の正常なリージョンへ転送される必要があります。

つまり、このアーキテクチャでは次の2つの重要な要件を満たす必要があります。

まず、ユーザーの位置に基づいて **最も低いネットワーク遅延を持つリージョンへルーティングすること**です。

次に、リージョン障害が発生した場合に **自動的に別のリージョンへフェイルオーバーすること**です。

このようなグローバルレベルのトラフィックルーティングを実現する Azure サービスを選択する必要があります。

---

## 2. 要件整理

この問題では、グローバル負荷分散ソリューションに関するいくつかの重要な要件があります。これらの条件を整理することで、最適なサービスを選択することができます。

まず、トラフィックルーティングの範囲に関する要件があります。アプリケーションは複数の Azure リージョンに配置されているため、リージョン間でトラフィックを制御する必要があります。

次に、ユーザー体験に関する要件があります。ユーザーは自分の位置から最も低いネットワーク遅延を持つリージョンに接続される必要があります。

さらに、可用性に関する要件があります。もし1つのリージョンが利用できなくなった場合には、すべてのトラフィックを自動的に別の正常なリージョンへ送る必要があります。

これらの条件をまとめると、次のような要件になります。

- グローバルレベルの負荷分散  
- レイテンシーベースのルーティング  
- 自動フェイルオーバー  
- Web アプリケーション向けのトラフィック管理  

このような条件を満たす Azure のサービスを選択する必要があります。

---

## 3. 技術の基本概念

### グローバルロードバランシング

グローバルロードバランシングとは、複数のリージョンに配置されたサービス間でトラフィックを分散する仕組みです。この仕組みにより、ユーザーの地理的位置に応じて最適なサービスへリクエストをルーティングすることができます。

グローバルロードバランシングでは、主に次の2つの目的があります。

まず、ユーザーに最も近いリージョンへトラフィックをルーティングすることで、ネットワーク遅延を減らすことです。

次に、障害発生時に別のリージョンへトラフィックを自動的に切り替えることで、高い可用性を確保することです。

このような機能は、世界中のユーザーがアクセスする Web アプリケーションにとって非常に重要です。

---

### Azure Front Door

Azure Front Door は、Microsoft のグローバルエッジネットワークを利用したアプリケーションレベル（Layer 7）のグローバルロードバランシングサービスです。

Azure Front Door は世界中のエッジロケーションに存在しており、ユーザーのリクエストは最寄りのエッジポイントに到達します。その後、Front Door が最適なバックエンドリージョンへトラフィックをルーティングします。

この仕組みにより、ユーザーは最も低いレイテンシーを持つリージョンへ接続されます。

Azure Front Door は次のような機能を提供します。

- レイテンシーベースのルーティング  
- グローバル HTTP/HTTPS ロードバランシング  
- バックエンドヘルスチェック  
- 自動フェイルオーバー  

これらの機能により、高可用性のマルチリージョン Web アプリケーションを構築できます。

---

## 4. アーキテクチャまたは設計のポイント

Azure Front Door を使用すると、複数リージョンの Web アプリケーションを単一のエントリーポイントで公開できます。ユーザーは Front Door のエンドポイントにアクセスし、Front Door がバックエンドリージョンを選択します。

Front Door は各リージョンのバックエンドサービスの状態を継続的に監視しています。もし特定のリージョンが利用できなくなった場合、Front Door は自動的にそのリージョンをトラフィックルーティングから除外します。

その結果、ユーザーのリクエストは自動的に残りの正常なリージョンへ送られます。

また、Front Door はアプリケーションレイヤーで動作するため、HTTP/HTTPS トラフィックの最適化やセキュリティ機能を提供することも可能です。

---

## 5. 設計判断（なぜこの構成になるか）

この問題の重要なポイントは **グローバルルーティングと低遅延ルーティングの両方を実現すること**です。

Azure Front Door は Microsoft のグローバルエッジネットワークを利用して、ユーザーを最適なリージョンへルーティングします。また、バックエンドのヘルスチェック機能を使用することで、障害が発生したリージョンを自動的に除外することができます。

このように、Azure Front Door はグローバルロードバランシング、レイテンシーベースのルーティング、自動フェイルオーバーという要件をすべて満たすことができます。

そのため、このシナリオでは Azure Front Door が最適な選択になります。

## 6. 他の選択肢が誤りな理由

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Application Gateway]]
### Azure Application Gateway

Application Gateway はリージョンレベルの Layer 7 ロードバランサーです。単一リージョン内のトラフィック管理には適していますが、グローバルルーティング機能は提供していません。

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Load Balancer]]
### Azure Load Balancer

Azure Load Balancer は Layer 4 のロードバランサーであり、リージョン内のトラフィック分散に使用されます。複数リージョン間のルーティングはサポートしていません。

### Azure Traffic Manager

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure DNS]]
Traffic Manager は DNS ベースのトラフィックルーティングサービスです。レイテンシーベースのルーティングは可能ですが、DNS キャッシュの影響を受けるため、リアルタイムなフェイルオーバーには Azure Front Door の方が適しています。

---

## 7. 最終回答

B. Azure Front Door
