# レガシーアプリケーション向けKerberos認証のAzure移行

## 1. 背景（シナリオ）
組織は、Kerberos認証を使用するレガシーアプリケーションをAzureに移行する必要があります。ユーザーはオンプレミスのActive Directory (AD) に対して認証する必要がありますが、厳格なセキュリティポリシーにより、オンプレミスリソースへの直接アクセスは禁止されています。この制約を満たしつつ、アプリケーションがユーザーを安全に認証できるソリューションが求められています。

## 2. 要件整理
- Kerberos認証に対応したマネージドドメインサービスが必要
- オンプレミスADへの直接接続を避ける
- Azure ADと統合され、クラウド内で認証が完結
- レガシーアプリケーションの移行をサポート

## 3. 技術の基本概念
- **Azure AD ドメインサービス (Azure AD DS)**:
  - マネージドドメインサービスを提供
  - ドメイン参加、グループポリシー、Kerberos/NTLM認証をサポート
  - Azure AD と統合され、既存ユーザーをクラウド環境で認証可能
- **その他の選択肢**:
  - **Azure AD Application Proxy**: 主にWebアプリのリモートアクセス用で、Kerberos認証を必要とするレガシーアプリには不十分
  - **Azure Virtual Network**: ネットワーク環境を提供するが認証機能は持たない
  - **Azure AD B2B**: ゲストユーザーとのコラボレーション向けで、オンプレミスAD認証には非対応

## 4. アーキテクチャまたは設計のポイント
- Azure AD DS を利用することで、Kerberos認証を必要とするレガシーアプリケーションはクラウド上で安全に認証可能
- オンプレミスADへの接続を避け、セキュリティポリシーに準拠
- ユーザー管理はAzure ADと統合されるため、既存認証情報を再利用可能

## 5. 設計判断（なぜこの構成になるか）
- 必須のKerberos認証をサポート
- フルマネージドドメインによりオンプレミスADへの依存を排除
- セキュリティ制限を遵守しつつ、レガシーアプリケーションをクラウドに移行可能

## 6. 他の選択肢が誤りな理由
- **Azure AD Application Proxy**: Kerberosを必要とするレガシーアプリには対応不可
- **Azure Virtual Network**: 認証機能は提供せず、ネットワーク接続のみ
- **Azure AD B2B**: ゲストユーザー管理用で、オンプレミスAD認証の代替には不適

## 7. 最終回答
**A) Azure AD ドメイン サービス (Azure AD DS)**

## 8. まとめ
Azure AD DS を導入することで、レガシーアプリケーションはKerberos認証を維持しながら、オンプレミスADへの直接接続を回避できます。クラウド環境内で安全かつ効率的に認証が可能となり、セキュリティポリシーの遵守とスムーズな移行を両立できます。
