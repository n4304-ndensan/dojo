# 可用性ゾーン障害に耐える Azure VM 配置設計（Dedicated Host）

## 1. 背景（シナリオ）

ある企業では、既存のアプリケーションを Azure 仮想マシンへ移行しました。このアプリケーションはビジネス上非常に重要であり、高い可用性が求められています。そのため、インフラストラクチャの設計では単一障害点を排除し、データセンター障害やゾーン障害が発生してもアプリケーションが継続して稼働する必要があります。

今回のシナリオでは特に厳しい可用性要件があります。プライマリリージョン内の **2つの可用性ゾーンが同時に障害を起こした場合でも、アプリケーションは利用可能でなければなりません**。つまり、インフラは最低でも3つの可用性ゾーンに分散して配置される必要があります。

さらに、この企業にはセキュリティおよびコンプライアンスに関する追加要件があります。それは、仮想マシンが実行される物理ホストを他の顧客と共有してはならないというものです。Azure の標準的な VM 配置では複数テナントが同一物理ホストを共有する可能性がありますが、この企業ではそのようなマルチテナント環境を許容できません。

このような要件を満たしながら、できるだけコスト効率の高い構成を設計する必要があります。

---

## 2. 要件整理

この問題を理解するためには、まず設計要件を整理することが重要です。問題文には複数の条件が含まれており、それぞれがアーキテクチャ選択に影響します。

まず、可用性に関する要件があります。アプリケーションは単一の可用性ゾーンだけでなく、複数ゾーンにわたって稼働する必要があります。特に今回は、2つの可用性ゾーンが障害になった場合でも稼働できる構成が求められています。

- プライマリリージョン内で **2つの可用性ゾーンが停止してもアプリケーションが利用可能**

次に、インフラの分離に関する要件があります。企業は仮想マシンが実行される物理ハードウェアを他の顧客と共有することを許可していません。つまり、専用ハードウェア上で VM を実行する必要があります。

- 仮想マシンは **専用の物理ホストで実行する必要がある**

最後に、コスト効率の要件があります。企業は高可用性と専用ハードウェアを確保しながらも、可能な限り費用対効果の高い構成を選択する必要があります。

- 要件を満たしつつ **最も費用対効果の高い構成**

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Dedicated Host]]
[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Availability Zone]]
これらの条件を満たす Azure サービスとして最も適しているのが **Azure Dedicated Host と Availability Zones の組み合わせ**です。

## 3. 技術の基本概念

### Azure Availability Zones

Azure の Availability Zone は、同一リージョン内に存在する複数の物理的に独立したデータセンターの集合です。各ゾーンは独立した電源、ネットワーク、冷却設備を持っており、1つのゾーンで障害が発生しても他のゾーンには影響しないように設計されています。

Availability Zone を使用することで、アプリケーションはデータセンター障害に対して高い耐障害性を持つようになります。

可用性ゾーンの重要な特徴を理解するためには、まずゾーン構成の基本的な考え方を整理する必要があります。Azure のリージョンでは通常、次のようなゾーン構成が提供されています。

- Zone 1  
- Zone 2  
- Zone 3  

これらのゾーンは物理的に分離されたデータセンターであり、1つまたは2つのゾーンに障害が発生しても、残りのゾーンでワークロードを実行し続けることが可能です。

---

### Azure Dedicated Host

Azure Dedicated Host は、Azure が提供する物理サーバーを単一の顧客専用として割り当てるサービスです。通常の Azure 仮想マシンは共有インフラストラクチャ上で実行されますが、Dedicated Host を使用すると物理ホストを他の顧客と共有しません。

Dedicated Host を使用すると、次のようなメリットがあります。

- 物理サーバーを **単一顧客専用で利用できる**  
- 規制やコンプライアンス要件に対応できる  
- 仮想マシンの配置をホスト単位で制御できる  

この仕組みにより、企業はクラウド環境でも専用インフラに近い運用が可能になります。

---

## 4. アーキテクチャまたは設計のポイント

今回の要件では、可用性ゾーンと専用ホストの両方を考慮する必要があります。可用性ゾーンだけでは物理ホストの共有を防ぐことはできませんし、Dedicated Host だけではゾーン障害への耐性を確保できません。

そのため、この設計では Dedicated Host を使用した仮想マシンを **3つの可用性ゾーンに分散配置**します。これにより、アプリケーションはゾーンレベルの障害に対して高い耐障害性を持つようになります。

この構成では、各可用性ゾーンに専用ホストを配置し、そのホスト上で仮想マシンを実行します。これにより、物理ホストの専有という要件を満たしつつ、ゾーン障害への耐性を確保できます。

もし2つの可用性ゾーンが同時に停止した場合でも、残りのゾーンに配置された仮想マシンがアプリケーションを継続して提供できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題では、可用性要件と物理分離要件を同時に満たす必要があります。Azure の標準的な VM 配置では物理ホストが共有されるため、専用ホストを使用する必要があります。

さらに、可用性ゾーンを使用することでデータセンター障害への耐性を確保できます。3つのゾーンに分散配置することで、2つのゾーンが同時に停止してもサービスを継続できます。

また、リージョン間配置を行うとデータ転送コストやアーキテクチャの複雑性が増加するため、単一リージョン内で可用性ゾーンを利用する構成が最もコスト効率の高い方法になります。

このような理由から、Dedicated Host と Availability Zones を組み合わせた構成が最適な解決策になります。

---

## 6. 他の選択肢が誤りな理由

### A 単一可用性セット

可用性セットは VM を異なる障害ドメインと更新ドメインに分散する仕組みですが、単一データセンター内でのみ機能します。可用性ゾーンのように複数データセンターへ分散することはできません。そのためゾーン障害への耐性を提供できません。

### B 単一可用性ゾーン

単一の可用性ゾーンに VM を配置すると、そのゾーンが障害を起こした場合にアプリケーションが停止します。また、今回の要件である「2つのゾーン障害への耐性」を満たすこともできません。

### D 複数リージョン

複数リージョンに展開すると可用性は高くなりますが、今回の要件はプライマリリージョン内でのゾーン障害への耐性です。さらにリージョン間通信にはレイテンシやコストが追加されるため、最も費用対効果の高い構成とは言えません。

---

## 7. 最終回答

C. Azure Dedicated Host を使用して 3つの可用性ゾーンに仮想マシンを展開する
