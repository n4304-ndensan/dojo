# 大規模 SQL Server データベースを最小ダウンタイムで Azure に移行する設計

## 1. 背景（シナリオ）

ある企業では、オンプレミス環境で運用されている **SQL Server データベース**を Azure に移行する計画を立てています。このデータベースは企業の主要な業務システムを支えており、データ量は約 **4TB** に達しています。また、このデータベースは日々大量のトランザクションが発生するため、トランザクションログの増加速度が非常に高いという特徴があります。

このような大規模データベースの移行では、単にデータをコピーするだけではなく、システムの停止時間をできる限り短くすることが重要です。業務システムが長時間停止するとビジネスに大きな影響が出るため、移行作業は **最小限のダウンタイム**で完了する必要があります。

そのため、移行方法を選択する際には、データ転送の方法だけでなく、移行期間中のデータ同期方法や最終的な切り替え手順なども考慮する必要があります。

---

## 2. 要件整理

この問題では、データベース移行の方法を選択するためのいくつかの重要な要件があります。これらの条件を整理することで、最適な移行方法が見えてきます。

まず、データベースの規模に関する要件があります。今回のデータベースは 4TB という非常に大きなサイズであり、移行作業には相当な時間がかかる可能性があります。

- データベースサイズ **4TB**

次に、ワークロードの特性があります。このデータベースではトランザクションログの増加率が高く、頻繁にデータ変更が発生しています。

- **トランザクションログの増加率が高い**

さらに、運用上の重要な要件があります。企業はシステム停止時間を最小限に抑える必要があります。

- **ダウンタイムを最小化**

これらの条件を考慮すると、単純なバックアップ・リストア方式やオフライン移行ではなく、移行中もデータベースを稼働させながらデータを同期する方法が必要になります。

---

## 3. 技術の基本概念

[[docs/tech_notes/microsoft/azure/az305/Azure用語集.md#Azure Database Migration Service]]
### Azure Database Migration Service（DMS）

Azure Database Migration Service（DMS）は、オンプレミスデータベースを Azure に移行するためのマネージドサービスです。このサービスは SQL Server、MySQL、PostgreSQL などのデータベース移行をサポートしています。

DMS は移行方法として **オフライン移行**と**オンライン移行**の2種類を提供しています。それぞれの移行方式には異なる特徴があります。

移行方式の違いを理解するためには、それぞれの特徴を整理する必要があります。

まずオフライン移行は、データベースを停止した状態で移行を行う方法です。この方式ではデータの整合性を保ちやすいですが、移行作業中はシステムを利用することができません。

一方でオンライン移行は、ソースデータベースを稼働させたままデータをコピーし、その後トランザクションログの変更を継続的に同期する方式です。この方式では、最終的な切り替え時のみ短時間の停止が必要になります。

---

## 4. アーキテクチャまたは設計のポイント

今回のシナリオでは、Azure Database Migration Service の **オンラインモード**を使用することが最適です。オンラインモードでは、移行プロセスが複数の段階に分かれて実行されます。

まず最初に、ソースデータベースからターゲットデータベースへ **初期データロード**が行われます。この段階では、データベース全体が Azure 側にコピーされます。

次に、初期コピーが完了した後、トランザクションログを使用して **変更データの継続的な同期**が行われます。この段階では、オンプレミスのデータベースは引き続き稼働しながら、変更されたデータが Azure 側に反映されます。

最後に、最終カットオーバーを実行します。このタイミングでソースデータベースを停止し、残っている変更データを同期した後に Azure のデータベースへ切り替えます。

この手順により、ダウンタイムは最終切り替えの短い時間だけになります。

---

## 5. 設計判断（なぜこの構成になるか）

この問題の重要なポイントは **大規模データベースと高いログ増加率**です。

4TB のデータベースを完全にコピーするには長時間が必要になります。もしオフライン移行を使用した場合、移行作業中はアプリケーションが利用できなくなり、ビジネスに大きな影響が出ます。

オンライン移行では、データベースを稼働させたままデータコピーを行い、その後トランザクションログを使用して変更を同期します。そのため、実際のダウンタイムは最終的な切り替え時のみになります。

この方式は特に次のような状況で有効です。

- 大規模データベース
- 高頻度のデータ更新
- 業務停止が許されないシステム

これらの条件に該当する場合、Azure Database Migration Service のオンラインモードが最も適した移行方法になります。

## 6. 他の選択肢が誤りな理由

### A オフラインモードの DMS

オフライン移行では、データベースを停止してからデータコピーを行う必要があります。4TB のデータベースでは移行時間が長くなり、長時間のダウンタイムが発生します。

### C URL へのバックアップ

バックアップと復元による移行はシンプルな方法ですが、バックアップ取得から復元完了までの間はデータベースを停止する必要があります。トランザクションログが大量に増加する環境では同期が困難になります。

### D SSMS デプロイウィザード

SSMS のデプロイウィザードは比較的小規模なデータベース移行に適しています。大規模データベースや継続的なログ同期には適していません。

---

## 7. 最終回答

B. オンラインモードで Azure Database Migration Service（DMS）を使用する
