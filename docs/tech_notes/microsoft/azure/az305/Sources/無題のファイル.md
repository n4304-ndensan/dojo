# Azure SQL Database の高可用性アーキテクチャ（Business Critical + Auto Failover Group）

## 1. 背景（シナリオ）

ある組織は、重要なアプリケーション向けに **回復力の高い Azure SQL Database アーキテクチャ** を設計する必要があります。  
このアプリケーションには以下の要件があります。

- データ損失ゼロ（RPO ≈ 0）
- 高可用性
- リージョン障害時でもサービス継続
- 自動フェイルオーバー
- 最適なパフォーマンス
- コスト効率

つまり **HA（High Availability）+ DR（Disaster Recovery）** を同時に満たす設計が必要です。

---

# 2. 要件整理

この問題の設計要件を整理すると次の4つです。

### 可用性
- リージョン障害でも継続動作

### データ保護
- データ損失ゼロ

### フェイルオーバー
- 自動フェイルオーバー

### パフォーマンス
- 高いトランザクション性能

---

# 3. Azure SQL Database の可用性設計

Azure SQL Databaseには主に次の **可用性レベルの設計要素**があります。

### 同一リージョンの可用性

#### ゾーン冗長 (Zone Redundant)

同一リージョン内の **複数の Availability Zone** にレプリカを配置します。

メリット

- データセンター障害に耐える
- 自動フェイルオーバー
- RPO ≈ 0

---

### リージョン障害対策

#### Auto Failover Groups

別リージョンへ自動フェイルオーバーを行う機能です。

特徴

- セカンダリリージョンへレプリケーション
- 自動フェイルオーバー
- 読み取りレプリカ提供
- グローバルDR

---

# 4. Business Critical 層の特徴

Azure SQL Database の **Business Critical tier** はミッションクリティカルなワークロード向けです。

主な特徴

- 高速ローカルSSD
- Always On アーキテクチャ
- 複数レプリカ
- 低レイテンシ
- ゾーン冗長サポート

つまり

**最高レベルのパフォーマンス + 高可用性**

を提供します。

---

# 5. 推奨アーキテクチャ

この問題の理想構成は次です。

```

Primary Region  
├ Business Critical (Zone redundant)  
│  
│ Auto Failover Group  
│  
Secondary Region  
├ Business Critical

```

この構成により

### 同一リージョン障害

→ Availability Zoneでフェイルオーバー

### リージョン障害

→ Auto Failover Groupで自動切替

---

# 6. 他の選択肢が誤りな理由

## A  
Active Geo Replication + Standard tier

問題点

- Standardはパフォーマンスが低い
- 自動フェイルオーバーではない
- 手動切替

---

## C  
Hyperscale + Manual failover

問題点

- 手動フェイルオーバー
- DR自動化なし

---

## D  
Basic tier + PITR

問題点

- 単一リージョン
- 高可用性なし
- DRなし

---

# 7. 正解

**B  
Business Critical tier + Zone redundant + Auto Failover Group**

---

### 同一リージョン HA

```

Zone Redundant

```

### リージョン DR

```

Auto Failover Group

```

### 高パフォーマンス

```

Business Critical

```

---

# まとめ

重要アプリケーションのAzure SQL設計では

- Business Critical → 高性能
- Zone redundancy → リージョン内HA
- Auto Failover Group → リージョンDR

この **3つの組み合わせ** が最も強力な可用性アーキテクチャになります。
