# 管理グループ階層で仮想ネットワーク管理を委任する RBAC 設計

## 1. 背景（シナリオ）

ある企業では、Enterprise Agreement（EA）に紐づく複数の Azure サブスクリプションを運用しています。これらのサブスクリプションは Azure の **管理グループ（Management Group）階層**の下で整理されており、組織全体のガバナンスを管理グループ単位で実施しています。

企業のネットワークチームは、すべての Azure サブスクリプションに存在する仮想ネットワークを管理する必要があります。さらに、この管理は現在存在しているサブスクリプションだけでなく、今後追加されるサブスクリプションにも自動的に適用される必要があります。

ただし、このアクセス権の付与には重要な条件があります。それは **最小特権の原則（Principle of Least Privilege）** を守ることです。つまり、ネットワーク管理に必要な権限だけを付与し、それ以上の権限は与えてはいけません。

この要件を満たすためには、Azure の RBAC（Role-Based Access Control）と管理グループ階層の仕組みを理解したうえで、適切なスコープに適切なロールを割り当てる必要があります。

---

## 2. 要件整理

この問題では、アクセス管理の設計要件がいくつか存在します。まず、それぞれの条件を整理しておくことで、最適な RBAC 構成が見えてきます。

まず、対象となるリソース範囲に関する要件があります。ネットワークチームは、単一のサブスクリプションではなく、管理グループ階層内にあるすべてのサブスクリプションの仮想ネットワークを管理する必要があります。

- すべての **現在のサブスクリプション**
- すべての **将来追加されるサブスクリプション**

次に、権限の範囲に関する要件があります。ネットワークチームは仮想ネットワークを管理できれば十分であり、他の Azure リソースや管理操作を行う必要はありません。

- 仮想ネットワークの管理権限が必要
- それ以外の権限は不要

最後に、運用管理の要件があります。サブスクリプションが増えるたびに手動で権限設定を行うのは管理負荷が高く、ガバナンス上も望ましくありません。

- **将来のサブスクリプションにも自動適用される必要がある**

このような要件を満たすためには、Azure RBAC の **スコープ継承（Scope Inheritance）** を活用する必要があります。

---

## 3. 技術の基本概念

### Azure RBAC のスコープ構造

Azure RBAC では、アクセス権は特定のスコープに対して割り当てられます。このスコープは Azure のリソース階層に基づいています。

Azure の RBAC スコープは階層構造になっており、上位スコープで設定された権限は下位スコープに継承されます。この階層構造を理解することが、今回の問題の鍵になります。

Azure のリソース階層は次のような順序で構成されています。

- 管理グループ（Management Group）
- サブスクリプション
- リソースグループ
- リソース

この階層構造では、管理グループが最も上位に位置します。管理グループで割り当てられた RBAC 権限は、その配下のすべてのサブスクリプションとリソースに継承されます。

---

### Network Contributor ロール

Azure には多くの組み込みロールがありますが、その中の一つに **Network Contributor** というロールがあります。このロールは、ネットワークリソースの管理に特化した権限を提供します。

Network Contributor ロールには、次のような操作を実行する権限が含まれています。

- 仮想ネットワークの作成
- サブネットの管理
- ネットワーク設定の変更
- ネットワークリソースの削除

一方で、このロールには次のような権限は含まれていません。

- リソースへのフル管理権限
- RBAC 管理権限
- 課金管理

そのため、Network Contributor は **最小特権の原則に適したロール**といえます。

---

## 4. アーキテクチャまたは設計のポイント

今回の要件では、ネットワークチームに対して仮想ネットワーク管理の権限を付与する必要があります。しかし、この権限は単一サブスクリプションではなく、管理グループ階層全体に適用される必要があります。

そのため、RBAC のロール割り当てを **ルート管理グループのスコープ**で行います。ルート管理グループは Azure 環境の最上位に位置する管理グループであり、その配下にはすべての管理グループとサブスクリプションが含まれます。

ルート管理グループで Network Contributor ロールを割り当てると、その権限は次の対象に自動的に適用されます。

- 既存のすべてのサブスクリプション
- 管理グループ配下のすべてのリソース
- 将来追加されるサブスクリプション

このような RBAC 設計により、ネットワークチームはすべての仮想ネットワークを管理できるようになります。また、新しいサブスクリプションが追加された場合でも、自動的に同じ権限が適用されるため、運用管理の負担を大幅に削減できます。

---

## 5. 設計判断（なぜこの構成になるか）

この問題のポイントは **スコープの選択と最小特権の両立**です。

まず、個別のサブスクリプションにロールを割り当てる方法では、新しいサブスクリプションが追加されるたびに RBAC 設定を更新する必要があります。これは運用負荷が高く、管理ミスの原因にもなります。

一方で、管理グループのルートスコープでロールを割り当てれば、すべてのサブスクリプションに対して自動的に権限が継承されます。

さらに、Owner ロールのような強力な権限を付与すると最小特権の原則に違反してしまいます。そのため、ネットワーク管理に限定された **Network Contributor ロール**を使用することが重要です。

このように、ルート管理グループで Network Contributor ロールを割り当てる構成が、最もシンプルでスケーラブルなガバナンス設計になります。

---

## 6. 他の選択肢が誤りな理由

### B カスタム RBAC ロール

カスタムロールを作成して各サブスクリプションに割り当てることは可能ですが、新しいサブスクリプションが追加されるたびに設定が必要になります。この方法はスケーラビリティが低く、運用管理の負担が増加します。

### C Owner ロール

Owner ロールは Azure リソースに対してほぼすべての操作を実行できる強力な権限です。ネットワーク管理に必要な範囲を超えているため、最小特権の原則に違反します。

### D 各サブスクリプションで Network Contributor

各サブスクリプションごとにロールを割り当てると、現在のサブスクリプションには適用できますが、将来追加されるサブスクリプションには自動適用されません。管理の手間が増えるため適切ではありません。

---

## 7. 最終回答

A. ルート管理グループスコープのチームに Network Contributor ロールを割り当てる
